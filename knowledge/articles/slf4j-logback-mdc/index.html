01

FACADE & PROVIDER

SLF4J 定义入口，Logback 提供实现

应用和库通过 SLF4J API 记录日志，运行时由 Provider 实现具体输出。应用应只保留一个期望的 Provider；多个绑定、桥接循环或同时直接依赖多个日志 API，会产生启动警告、重复输出甚至递归。

SLF4J API

稳定的日志调用门面。

facade
Provider

运行期选择的日志实现。

backend
Appender

把事件写入控制台、文件或其他目标。

output
Encoder

把事件编码为文本或结构化格式。

format

公共库通常只依赖 SLF4J API，不强迫使用者采用某个实现；最终应用负责选择 Provider、配置级别和输出目的地。

02

LOG EVENT

参数化记录事实，不拼接含糊字符串

使用占位参数构造日志，让禁用级别时避免不必要字符串拼接，并让参数保持独立。每条事件包含稳定动作、结果、目标类型、耗时和关联 ID；不要把完整对象直接 toString 后输出。

  • ✦ERROR：当前操作失败，需要明确处置或告警
  • ✦WARN：出现异常条件但系统仍可继续，需要关注
  • ✦INFO：低频、关键生命周期与业务状态变化
  • ✦DEBUG/TRACE：诊断细节，生产环境按范围临时启用
日志不是敏感数据仓库

密码、Token、Cookie、银行卡、证件号、完整请求体和个人隐私默认不记录；确需识别时使用脱敏值或受控哈希。

03

MAPPED DIAGNOSTIC CONTEXT

MDC 为同一执行上下文补充关联字段

MDC 是与当前线程关联的键值上下文，适合保存 requestId、traceId、tenantId 等低体积诊断字段。请求入口写入，在 finally 中恢复或清除，防止线程池复用把上一次请求身份带到下一次。

上下文生命周期CONTEXT
接收请求
  → 校验或生成 requestId
  → 写入 MDC
  → 执行业务并记录日志
  → finally 恢复原值或 clear

MDC 通常不会自动跨线程传播。CompletableFuture、自建线程池、响应式流和消息消费需要显式捕获与恢复上下文，并确保异常路径同样清理。不要把整个用户对象或无界标签放进 MDC。

04

ASYNC & ROLLING

异步日志用队列换取调用线程时间

AsyncAppender 把日志事件放入队列，由后台线程写出。它能降低调用线程等待，但队列容量有限；磁盘或下游故障时，事件可能阻塞、丢弃或占用更多内存，具体行为必须配置并压测。

  • ✦队列有界并监控占用、丢弃和写入延迟
  • ✦关键 ERROR 是否允许丢失需要单独策略
  • ✦滚动策略同时限制单文件、总大小与保留天数
  • ✦容器优先标准输出时，避免应用和平台重复滚动
  • ✦优雅停机为后台队列保留刷新时间
05

EXCEPTIONS

异常只在负责处理的边界完整记录一次

同一异常在 DAO、Service、Controller 层层打印会制造重复堆栈。底层要么补充上下文后继续抛出，要么转换为稳定异常；最终处理边界记录一次完整堆栈、请求标识与安全上下文。

Message

stack
Context

携带 requestId 与安全业务标识。

where
Response

对外返回稳定错误码，不泄漏内部堆栈。

06

LOG GOVERNANCE

结构化日志与运行治理清单

  1. 01定义统一字段名、时间、时区、级别和服务身份
  2. 02把日志当作事件结构输出，避免依赖文本正则解析
  3. 03控制高基数字段，搜索 ID 不等于建立指标标签
  4. 04对日志量、磁盘、采集延迟、解析失败和丢弃告警
  5. 05定期扫描秘密与个人信息，验证脱敏和访问权限
  6. 06通过并发与异步测试验证 MDC 不串号、不残留
