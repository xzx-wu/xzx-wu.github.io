GC 不是越少越好：真正目标是让应用在内存预算内满足吞吐、延迟和稳定性要求。

JVM 档案 · Collector Selection
01

GOALS

先定义想改善什么

吞吐

应用线程执行时间占总时间的比例，批处理通常更关注。

throughput
停顿

单次与高分位暂停时间，在线服务通常有明确 SLO。

pause latency
内存占用

堆上限、提交内存与收集器额外结构都受容器或机器预算约束。

footprint
CPU 余量

并发收集会与应用共享 CPU，低停顿并非没有成本。

headroom

四个目标不能同时无限优化。先记录业务 P99、吞吐、错误率、分配速率和 GC 指标，才能判断变化是否真的改善用户体验。

02

GARBAGE FIRST

G1：平衡停顿与吞吐

G1 把堆划分为多个 region，增量回收价值较高的区域，并用并发标记与暂停预测尝试满足目标停顿。JDK 21 的多数常见环境默认选择 G1。

  • ✦适合希望获得较稳定停顿、同时保持较好吞吐的通用服务
  • ✦MaxGCPauseMillis 是目标而非硬实时保证
  • ✦避免固定年轻代大小，以保留 G1 的暂停预测调节能力
  • ✦大对象可能作为 humongous object 占用连续 region，需要日志验证
G1 基线SHELL
java -Xms4g -Xmx4g \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -Xlog:gc*:file=gc-g1.log:time,uptime,level,tags \
  -jar archive-app.jar
03

Z GARBAGE COLLECTOR

ZGC：以极低停顿为核心目标

ZGC 把昂贵工作尽量与应用线程并发执行，停顿目标与堆大小弱相关，适合对尾延迟敏感且能提供足够 CPU 与内存余量的应用。JDK 21 同时提供分代 ZGC 与较早的非分代模式。

JDK 21 分代 ZGCSHELL
java -Xms4g -Xmx4g \
  -XX:+UseZGC \
  -XX:+ZGenerational \
  -Xlog:gc*:file=gc-zgc.log:time,uptime,level,tags \
  -jar archive-app.jar
  • 01低停顿不等于总响应时间自动降低
  • 02并发回收需要 CPU 余量跟上对象分配速度
  • 03SoftMaxHeapSize 可表达期望软上限，但 JVM 仍可使用到 Xmx
  • 04版本升级前后需重新基准，不能沿用旧结论
04

SELECTION

按业务目标选择

优先从 G1 开始

通用 Web 服务、停顿目标为几十到数百毫秒、同时重视吞吐。

balanced
评估 ZGC

P99/P999 对短暂停顿极敏感、堆较大、机器有并发 GC 余量。

low latency
不要忽略 Parallel GC

批处理只重视总吞吐且能接受较长暂停时，也应纳入对照。

throughput
同一负载做 A/B 基准

使用相同 JDK、堆、容器资源、流量模型和预热时间，对比业务延迟、吞吐、CPU、内存与错误率，而不是只比较 GC 暂停。

05

BASELINE

先保持参数最少

  • ✦明确 -Xmx，给非堆内存与系统留出余量
  • ✦开启可轮转、可保留的统一 GC 日志
  • ✦G1 先用默认启发式，仅按需要设暂停目标
  • ✦ZGC 首要调节手段通常是提供足够最大堆
  • ✦删除来源不明的历史参数，再逐个验证必要性
06

EVIDENCE

从日志定位真正限制

  1. 01对齐业务延迟尖峰与 GC 暂停时间线
  2. 02观察分配速率、存活集、晋升和并发周期是否及时
  3. 03区分 CPU 饱和、堆不足、对象滞留和大对象问题
  4. 04先修复无界缓存、突发分配和超大对象，再调收集器
  5. 05每次只改变少量变量，保留可回滚对照结果
不要用平均暂停掩盖尾部

同时查看最大值、P99、暂停原因和发生频率。少量极端暂停往往比平均值更接近用户实际体验。
