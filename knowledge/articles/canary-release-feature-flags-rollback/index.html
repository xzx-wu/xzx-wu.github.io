02

CANARY STEPS

每一级流量都要有最小观察时间和决策条件

  1. 01部署新版本但不接收生产流量，完成健康与依赖检查
  2. 02导入内部、测试租户或极小稳定分桶
  3. 03观察至少覆盖典型请求和后台任务的窗口
  4. 04对比基线与 Canary 的技术和业务信号
  5. 05通过人工审批或受控自动分析扩大流量
  6. 06完成全量后继续观察并清理旧副本

按比例分流未必等于风险均匀：大客户、地域、长连接和缓存亲和会造成流量偏斜。需要同时检查请求量、用户群和业务价值分布。

03

FEATURE FLAGS

功能开关是运行时控制面，不是永久分支

Feature Flag 可按用户、租户、环境或稳定哈希评估，常用于发布开关、实验和运维保护。评估输入必须最小化并受信，默认结果和配置不可用时的行为需要显式设计。

  • ✦每个开关有负责人、用途、创建时间和到期日
  • ✦服务端执行关键开关，不能信任客户端声明
  • ✦评估结果进入日志、Trace 或低基数指标
  • ✦开关状态与代码版本兼容关系有测试
  • ✦紧急开关的修改权限、审批和审计更严格
  • ✦发布完成后删除开关和死亡分支
04

ANALYSIS & GUARDRAILS

技术信号和业务信号共同决定是否扩大

错误

错误率、异常类型、依赖失败和数据校验失败。

errors
延迟

关键路径分位数、超时和队列等待。

latency
容量

CPU、内存、连接池、线程池和队列饱和。

saturation
业务

成功转化、金额、任务完成与用户投诉。

outcome

阈值、分析窗口和最小样本量要在发布前定义。低流量阶段不能仅因“没有错误”判定成功，可能只是尚未覆盖关键场景。

05

ROLLBACK BOUNDARIES

应用回滚容易，数据和外部副作用未必可逆

回退旧镜像不能自动撤销 Schema、消息、缓存和第三方调用。数据库演进可采用 expand/contract：先增加兼容结构，双版本共存并完成回填，再在所有旧代码退出后收缩。

  • ✦制品不可变并能按版本快速恢复
  • ✦新旧版本同时运行时协议前后兼容
  • ✦消息 Schema 支持跨版本消费和回放
  • ✦数据写入变化准备补偿或修复脚本
  • ✦回滚动作保留授权、时间和原因
  • ✦风险可控时评估修复前进而非盲目回退
停止扩大不等于自动回滚

自动分析可以暂停发布，但回滚、补偿和数据修复应遵循预先授权的发布策略与实际影响判断。

06

RELEASE CHECKLIST

渐进式发布执行清单

  1. 01明确负责人、风险、回退边界和沟通窗口
  2. 02验证制品、配置、数据库与消息兼容性
  3. 03预先建立基线面板、分析指标和 Runbook
  4. 04每阶段记录目标、样本、观察时间和决策
  5. 05演练暂停、回退、配置恢复和流量切换
  6. 06全量后复盘告警质量并清理临时资源与开关
