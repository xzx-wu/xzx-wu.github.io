01

DIAGNOSIS LOOP

建立排查闭环

  • 01从接口监控或慢查询日志确定具体 SQL
  • 02带上真实参数、表结构和数据量复现
  • 03使用 EXPLAIN 查看优化器选择
  • 04必要时用 EXPLAIN ANALYZE 对比估算与实际
  • 05调整索引、SQL 或数据访问方式
  • 06重新测量耗时、扫描行数和写入成本
先保存现场

记录原 SQL、参数、执行计划、耗时、返回行数和表的索引信息。没有基线，就无法判断优化是真的有效，还是只在一次偶然测试中变快。

02

PROBLEM QUERY

准备一条问题 SQL

订单查询SQL
SELECT o.id, o.user_id, o.amount, o.created_at
FROM order_record o
WHERE o.user_id = 10001
  AND o.status = 2
  AND o.created_at >= '2026-07-01 00:00:00'
ORDER BY o.created_at DESC
LIMIT 20;
查看执行计划SQL
EXPLAIN FORMAT = TREE
SELECT o.id, o.user_id, o.amount, o.created_at
FROM order_record o
WHERE o.user_id = 10001
  AND o.status = 2
  AND o.created_at >= '2026-07-01 00:00:00'
ORDER BY o.created_at DESC
LIMIT 20;

排查时应同时执行 SHOW CREATE TABLE order_record 和 SHOW INDEX FROM order_record，确认实际表结构，不要根据代码中的实体类猜测线上索引。

03

TRADITIONAL OUTPUT

传统执行计划关键字段

type

访问方式。常见值包括 const、eq_ref、ref、range、index 和 ALL，但不能只按名称机械排名。

关注访问范围
possible_keys / key

前者是候选索引，后者是最终索引。候选存在但未选中，通常是成本估算的结果。

候选 ≠ 实际
key_len

提供实际使用索引键长度的线索，可辅助判断联合索引使用到哪些部分。

结合字段类型判断
rows / filtered

分别表示预计检查行数和过滤比例。两者都是估算值，需要与实际执行对比。

估算扫描规模
04

EXTRA DETAILS

正确理解 Extra

  • ✦Using index：通常表示查询由覆盖索引完成
  • ✦Using index condition：使用索引条件下推减少回表
  • ✦Using where：存储引擎返回后仍需应用条件过滤
  • ✦Using filesort：需要额外排序步骤，不代表必然写磁盘
  • ✦Using temporary：执行中使用内部临时表
  • ✦possible_keys：候选索引不代表最终选择

单个标记不能直接判定查询好坏。一个返回 10 行的小表全表扫描可能完全合理；一个“命中索引”但扫描数百万索引记录的查询仍然可能很慢。

05

ACTUAL EXECUTION

使用 EXPLAIN ANALYZE

EXPLAIN ANALYZE 会实际执行查询，并以树形格式展示每个迭代器的估算成本、估算行数、实际耗时、实际行数和循环次数。

对比估算与实际SQL
EXPLAIN ANALYZE FORMAT = TREE
SELECT o.id, o.user_id, o.amount, o.created_at
FROM order_record o
WHERE o.user_id = 10001
  AND o.status = 2
ORDER BY o.created_at DESC
LIMIT 20;
它会真正执行语句

不要在生产环境中对未知成本的查询直接使用。先评估语句类型、数据规模与业务影响，必要时在隔离的只读环境验证。

06

OPTIMIZATION ACTIONS

从执行计划落到优化动作

扫描行数过大

检查过滤条件选择性、联合索引顺序、隐式转换和函数运算。

缩小访问范围
回表次数过多

考虑覆盖索引、两阶段查询或减少不必要的返回列。

避免 SELECT *
额外排序明显

判断过滤条件和排序能否共享同一联合索引顺序。

WHERE + ORDER BY
估算偏差很大

检查统计信息和数据倾斜，必要时执行 ANALYZE TABLE 后复核。

ANALYZE TABLE
更新表统计信息SQL
ANALYZE TABLE order_record;
07

COMMON MISTAKES

常见优化误判

  • ×只看 type，不看实际行数和耗时
  • ×看到全表扫描就立即创建单列索引
  • ×在少量测试数据上判断索引收益
  • ×只优化 SQL，不检查接口是否重复调用
  • ×强制索引后不验证不同参数的数据分布
  • ×忽略新增索引对写入和存储的影响
