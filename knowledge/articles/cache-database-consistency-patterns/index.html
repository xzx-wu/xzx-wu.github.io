架构卷轴 · Cache Consistency
01

CACHE-ASIDE

数据库是事实源，应用负责旁路缓存

  1. 01读取缓存，命中则返回
  2. 02未命中时查询数据库
  3. 03把结果以有限 TTL 写入缓存
  4. 04返回数据库结果

Cache-Aside 适合读多写少且允许短暂陈旧的数据。缓存淘汰后会自动从数据库重建，但它不提供数据库与缓存之间的原子事务。

先定义一致性等级

余额、权限和库存通常比文章详情、配置展示更敏感。不能用同一个 TTL 和同一种失效策略覆盖所有数据。

02

WRITE STRATEGY

数据库提交后失效缓存是常见基线

写请求先在数据库事务中完成变更，提交成功后删除对应缓存；下一次读取从数据库重建。相比直接更新缓存，失效策略减少复杂派生值和并发写覆盖，但删除失败仍需要补偿。

写路径INVALIDATE
校验权限与业务版本
  → 数据库事务更新
  → COMMIT
  → 删除缓存 key
  → 删除失败进入重试 / 事件修复

不要在数据库事务提交前删除缓存后长时间执行事务：并发读可能回源旧数据并重新写入缓存。具体顺序仍需结合并发模型、事务隔离和业务容忍度验证。

03

FAILURE WINDOWS

不存在无代价的“完美双写”

数据库成功，删除失败

缓存继续提供旧值，需要可靠事件、重试或过期时间兜底。

repair
缓存删除，并发读回填旧值

使用版本校验、短事务或在严格场景串行化关键写。

race
更新缓存，数据库失败

缓存暴露不存在的新状态，因此不应先把新值当事实发布。

dirty
消息重复或乱序

携带业务版本，旧事件不能覆盖新状态。

version

“延迟双删”只能缩小特定竞态窗口，延迟值难以覆盖所有事务时长与网络抖动，不是普适保证。要求更强一致时，应减少缓存范围或选择能提供明确一致性语义的方案。

04

MULTI-LEVEL CACHE

每增加一级缓存，就增加一套失效协议

进程内缓存延迟低，却分散在多个实例；Redis 等共享缓存便于集中失效，但仍有网络和热点成本。多级缓存必须定义 L1、L2 的版本、TTL、广播失效和节点离线后的恢复方式。

  • ✦缓存键包含租户和数据版本边界
  • ✦本地缓存容量有界并提供淘汰统计
  • ✦失效消息允许重复且处理幂等
  • ✦节点重连后不能继续信任过期本地状态
  • ✦客户端跟踪只用于优化，不替代授权校验
  • ✦敏感字段不进入不受控共享缓存
05

HOT KEY & STAMPEDE

缓存失效时保护数据库承载能力

  • ✦TTL 增加受控抖动，避免同批键同时过期
  • ✦单飞或互斥重建限制同一键回源并发
  • ✦热点数据可提前刷新并保留短暂旧值
  • ✦不存在结果使用短 TTL 负缓存，防止穿透
  • ✦限制批量查询和单值大小，防止大 Key
  • ✦回源、降级与限流共享统一容量预算

负缓存不能掩盖刚创建的数据，也不能把权限拒绝缓存成所有用户共享结果。缓存键必须包含影响结果的身份与查询维度。

06

REPAIR & OPERATIONS

让陈旧状态可发现、可比较、可修复

  1. 01监控命中率、回源量、重建延迟、淘汰和错误
  2. 02记录删除失败、修复队列积压和版本拒绝
  3. 03提供按业务键校验数据库与缓存的受控工具
  4. 04定期抽样比对关键值，发现长期漂移
  5. 05演练 Redis 故障、热点失效和数据库降速
  6. 06变更 TTL、Key 结构和缓存范围时保留回滚方案
