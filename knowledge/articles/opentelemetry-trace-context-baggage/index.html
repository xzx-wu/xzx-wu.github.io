01

TRACE & SPAN

用父子关系还原一次分布式工作

Trace 由共享 Trace ID 的 Span 组成。Span 描述一个操作，包含名称、起止时间、父级、状态、属性、事件和链接。服务名、版本、环境等进程级信息应作为 Resource 统一附加。

属性 Attributes

描述可搜索维度，如 HTTP 方法、路由模板和数据库系统。

key=value
事件 Events

记录 Span 内有时间点的事实，不替代完整应用日志。

timestamp
状态 Status

表达操作是否错误，不能只由 HTTP 状态机械推导业务结果。

Unset / Error / Ok
链接 Links

关联无法表达为单一父子的工作，如批处理和消息扇入。

SpanLink
02

CONTEXT PROPAGATION

进程内传递 Context，进程间注入载体

Context 保存当前 Span 等跨调用状态。HTTP 客户端把追踪信息注入请求头，服务端提取后创建子 Span；线程池、异步回调和响应式流需要显式或由框架插桩传播当前 Context。

传播链路W3C TRACE CONTEXT
入口提取 traceparent
  → 创建 server span
  → Context 进入当前作用域
  → 客户端创建 child span 并注入 traceparent
  → 下游提取并继续同一 trace
作用域必须成对关闭

线程复用时若 Context 未清理，后续无关请求可能被错误关联。使用 try/finally 或语言 SDK 提供的作用域模式。

03

BAGGAGE

Baggage 跨边界传播业务元数据，但不会自动成为 Span 属性

Baggage 是随 Context 传播的键值集合，可用于把受控业务维度传到下游。它与 Span Attributes 不同：Baggage 需要显式读取和使用，也可能跨服务边界进入请求载体。

  • ✦只放低基数、体积小、确有传播价值的数据
  • ✦不要放密码、Token、个人数据和内部密钥
  • ✦在信任边界校验或删除外部传入项
  • ✦限制项数、长度和允许键名
  • ✦需要搜索的维度显式复制到 Span 属性
  • ✦不要把 Baggage 当授权依据
04

ASYNC & MESSAGING

异步边界要区分因果关系与处理关系

消息生产者把传播字段写入消息属性，消费者提取后创建消费 Span。同步请求通常使用父子关系；消息长时间排队、批量消费或多源聚合时，Span Link 往往更能表达真实关系。

  1. 01记录消息系统、目的地模板和操作类型
  2. 02避免把消息正文、用户标识作为高基数属性
  3. 03生产、投递和处理延迟分开观测
  4. 04重试保留消息业务标识，但每次尝试拥有可辨认 Span
  5. 05批处理使用链接或批次属性表达多个来源
  6. 06跨线程执行前确认 Context 传播是否由框架覆盖
05

SAMPLING

采样控制成本，也决定你能看到多少证据

头部采样在 Trace 开始时决定，成本可控但无法预知最终错误；尾部采样在收集到更多 Span 后决定，可保留错误或慢链路，但需要更高内存与等待成本。父级采样决策应在服务间一致传播。

  • ✦为错误、关键业务和低流量入口保留足够样本
  • ✦采样规则按服务与流量评估，不照搬固定比例
  • ✦监控 Collector 丢弃、队列、导出失败和限流
  • ✦关键审计不能依赖会被采样丢弃的 Trace
  • ✦变更采样前估算存储、网络与查询成本
  • ✦用指标补足 Trace 不适合承担的全量统计

OPERATIONS

链路追踪落地与排障清单

  1. 01统一服务名、环境、版本和实例资源属性
  2. 02优先使用语义约定，减少团队自造字段
  3. 03从入口、HTTP/RPC、数据库和消息边界验证传播
  4. 04将 traceId 注入结构化日志，实现日志与 Trace 互跳
  5. 05对导出端启用批处理、超时、重试和内存保护
  6. 06演练上下文丢失、Collector 故障与后端限流
