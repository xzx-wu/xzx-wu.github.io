运行时档案 · Incident Response
01

PRINCIPLES

先分级，再采证据

  • 01确认用户影响、故障时间和当前是否仍在恶化
  • 02保留 PID、实例、版本、发布记录和监控时间线
  • 03优先低开销信息，再决定 JFR、直方图或堆转储
  • 04需要重启时先记录取证缺口，恢复后复盘补齐监控
不要把重启当作根因

重启能够释放内存和线程，但也会清除现场。若业务允许，至少先保存 JVM 参数、线程转储和关键监控截图或导出数据。

02

BASELINE

建立同一时间线

低开销基线SHELL
date
ps -p <pid> -o pid,ppid,%cpu,%mem,etime,command
jcmd <pid> VM.version
jcmd <pid> VM.flags
jcmd <pid> GC.heap_info
jcmd <pid> Thread.print -l > threads-01.txt

命令输出要带时间戳，并与应用日志、GC 日志、宿主机或容器 CPU、内存、磁盘和网络指标对齐。单个瞬时快照不能证明趋势。

03

OUT OF MEMORY

先读完整异常类型

Java heap space

检查对象增长、缓存边界、分配速率与堆上限；结合直方图和堆转储分析。

heap
Metaspace

关注动态类生成、类加载器泄漏和类数量持续增长。

class metadata
Direct buffer memory

检查 NIO 直接内存、网络框架缓冲区和释放时机。

native buffer
unable to create native thread

核对线程数、线程栈大小、进程限制与容器内存。

threads
GC overhead limit exceeded

表示大量时间用于 GC 却回收很少，应查存活集增长而非只关闭保护机制。

gc pressure
按需采集SHELL
jcmd <pid> GC.class_histogram

# 堆转储可能触发停顿并消耗大量磁盘，先确认空间与业务影响
jcmd <pid> GC.heap_dump /var/log/app/heapdump.hprof
04

HIGH CPU

从进程定位到线程

  • 01确认 CPU 是持续高还是周期性尖峰
  • 02用操作系统工具找出高 CPU 线程 ID
  • 03将十进制线程 ID 转成十六进制，与线程转储 nid 对照
  • 04间隔数秒采集多份线程转储，确认调用栈是否持续
  • 05结合 JFR 查看方法采样、锁、分配与 GC 事件
Linux 示例SHELL
top -H -p <pid>
printf '%x\n' <thread-id>

jcmd <pid> Thread.print -l > threads-01.txt
# 间隔数秒后再次采集，而不是只看一份
jcmd <pid> Thread.print -l > threads-02.txt

若高 CPU 来自 GC 线程，继续进入 GC 分析；若业务线程反复停留在同一段计算、序列化、正则或自旋代码，则沿对应调用栈定位输入与代码路径。

05

GC PRESSURE

频繁 GC 看四个问题

分配速率是否突增

短命对象过多会增加 Young GC 频率，关注请求量与热点分配。

allocation rate
存活集是否增长

Old 区持续上升且回收后不下降，可能存在缓存或对象滞留。

live set
单次停顿是否过长

区分频率与停顿时长，并检查并发阶段是否跟得上分配速度。

pause
堆是否设置失衡

过小会频繁回收，过大也可能加重部分停顿和容器风险。

heap sizing

首先读 GC 日志中的触发原因、回收前后容量和停顿时间，再决定是否调堆或切换收集器。没有对象增长和流量证据时，直接调参数容易掩盖代码问题。

06

DIAGNOSTIC TOOLS

jcmd 与 JFR 常用命令

诊断工具箱SHELL
jcmd <pid> help
jcmd <pid> VM.command_line
jcmd <pid> VM.flags
jcmd <pid> GC.heap_info
jcmd <pid> GC.class_histogram
jcmd <pid> Thread.print -l

jcmd <pid> JFR.start \
  name=incident settings=profile duration=2m \
  filename=/var/log/app/incident.jfr
先评估开销与权限

不同诊断命令的影响级别不同，堆转储尤其可能产生停顿和大文件。生产执行前确认磁盘、权限、数据敏感性和应急流程。

07

PREPARATION

把取证能力放进启动参数

基础建议SHELL
java \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/log/app/heapdump.hprof \
  -Xlog:gc*:file=/var/log/app/gc.log:time,uptime,level,tags \
  -jar archive-app.jar
  • ✦预留堆转储磁盘空间并配置文件保留策略
  • ✦监控堆、Metaspace、线程、GC 停顿和分配速率
  • ✦记录发布、扩缩容和配置变更事件
  • ✦定期演练 JFR、线程转储和堆分析流程
