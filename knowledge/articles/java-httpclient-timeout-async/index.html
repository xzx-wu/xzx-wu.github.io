CLIENT LIFECYCLE

构建一次，在同一配置边界内复用

HttpClient 创建后不可变，可以在多个请求间共享。复用有助于连接管理与 HTTP/2 会话利用，也避免每次调用重复创建执行器和 TLS 状态。按服务身份、代理、TLS 或隔离需求划分少量客户端，而不是每个请求 new 一个。

02

TIME BUDGET

连接超时与请求超时不是同一个旋钮

HttpClient.Builder.connectTimeout 约束建立新连接的等待；HttpRequest.Builder.timeout 为该请求设置时限。已有连接复用时不会重新经历建连，因此两者不能互相替代。

预算分层示意TIMEOUT
上游总截止时间：2000 ms
  ├─ 获取/建立连接预算：300 ms
  ├─ 单次请求预算：1200 ms
  ├─ 可选一次重试与退避：400 ms
  └─ 本地处理与余量：100 ms

预算必须从上游截止时间向下传递。只配置很长的底层超时，会让线程和连接在下游故障时持续占用；只配置很短的超时，又可能在正常尾延迟下制造重试放大。

03

SYNC & ASYNC

send 与 sendAsync 共享同一网络事实

send 阻塞当前线程直到响应可用或失败；sendAsync 返回 CompletableFuture。异步可以减少等待线程，但后续解析、回调阻塞、执行器容量和异常传播仍需设计。

  • ✦异步链中显式处理超时、取消和异常，不丢弃 CompletionException 根因
  • ✦CPU 密集或阻塞回调使用受控执行器，避免占满公共线程池
  • ✦限制同时在途请求数，为下游设置背压或信号量
  • ✦取消 Future 不应被误解为服务端一定停止执行
  • ✦请求上下文、Trace 与 MDC 在线程切换时显式传播
04

BODY HANDLER

响应体选择决定内存与关闭责任

BodyHandlers.ofString() 适合有明确大小上限的文本响应；ofByteArray() 会把正文完整放入内存；ofFile() 可直接写入文件；ofInputStream() 把流交给调用者并要求及时关闭。

状态码成功不代表正文可以无限读取

必须限制响应大小、读取时长和并发数。流式响应要在所有成功、异常和取消路径关闭，否则连接与文件描述符会持续占用。

先检查状态码和 Content-Type，再进入反序列化。错误响应也应限制体积，只提取可安全记录的摘要和 requestId。

05

SECURITY

TLS、重定向、代理与认证都需显式边界

  • ✦使用受信任证书与主机名校验，不通过“忽略证书”修复生产问题
  • ✦重定向策略考虑跨主机时 Authorization 等敏感头是否泄漏
  • ✦代理配置与 NO_PROXY 范围可审计，避免内部地址误走外部代理
  • ✦凭据从专门提供者获取，不写入 URI、日志或指标标签
  • ✦只允许必要协议、目标主机与端口，降低 SSRF 风险

当请求目标来自外部输入时，应在解析后重新校验 scheme、主机、端口和最终解析地址，并谨慎处理重定向。

06

RELIABILITY

可靠远程调用检查清单

  1. 01统一复用客户端，并记录连接与请求超时配置
  2. 02请求携带 requestId、幂等键和可传播的截止时间
  3. 03只对可恢复且幂等的失败进行有限重试
  4. 04指标区分 DNS、连接、TLS、首字节与总耗时
  5. 05记录状态码、目标服务与尝试次数，敏感信息脱敏
  6. 06通过故障注入验证超时、取消、半开连接与大响应
