01

CORE TYPES

六类常用结构

String

缓存对象、计数器、分布式令牌和位图底层。

GET / SET / INCR
Hash

同一对象的字段集合，适合局部字段读写。

HGET / HSET
List

按插入顺序保存元素，适合简单队列和时间线。

LPUSH / BRPOP
Set

无序去重集合，适合成员关系、共同关注。

SADD / SISMEMBER
Sorted Set

成员唯一且带 score 排序，适合排行榜和延迟任务索引。

ZADD / ZRANGE
Stream

追加日志、消费组和待确认消息，适合事件流。

XADD / XREADGROUP
02

USE CASES

场景与结构对应

  • ✦限流计数使用 String + INCR + EXPIRE，并处理首次设置竞态
  • ✦用户标签与在线成员使用 Set，不用逗号拼接字符串
  • ✦排行榜使用 Sorted Set，score 相同需明确二级排序语义
  • ✦可靠消费需要 Stream 消费组、ACK、pending 与重试策略
  • ✦复杂文档若频繁整体读取，String 序列化可能比大量 Hash 字段更简单
03

EXPIRATION

TTL 是数据生命周期，不只是省内存

原子写入并过期REDIS
SET session:9f31 "user-1001" EX 1800 NX
TTL session:9f31

覆盖写、重命名和部分命令会影响 TTL，应按实际命令语义验证。永久 key 必须有明确理由；缓存 key 则要考虑随机抖动和批量过期压力。

04

ATOMICITY

单条命令原子，但多步业务未必原子

  • 01优先使用原生命令完成集合更新、计数和条件写
  • 02多步读改写可使用 Lua/Functions 在服务端原子执行
  • 03MULTI/EXEC 保证命令批次执行，不提供数据库式回滚
  • 04脚本保持短小，避免长时间阻塞 Redis 执行线程
05

MEMORY

避免大 Key、热 Key 与阻塞扫描

  • ✦限制单 value 大小和集合成员数量，必要时分桶
  • ✦使用 SCAN 渐进遍历，不在生产执行 KEYS 全量扫描
  • ✦监控 hot key、big key、内存碎片和淘汰数量
  • ✦删除超大集合评估 UNLINK 等异步释放能力
  • ✦Key 命名包含业务、实体、标识和版本，但避免过长
复杂度要乘真实规模

标记为 O(N) 的命令在 N 很小时可能安全，在百万成员集合上可能成为事故。使用前看命令复杂度和返回量。
