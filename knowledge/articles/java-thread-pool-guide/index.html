01

SUBMISSION FLOW

任务提交后的决策顺序

  • 01运行线程少于 corePoolSize：创建核心线程
  • 02核心线程已满：尝试把任务放入队列
  • 03队列已满且未达 maximumPoolSize：创建额外线程
  • 04线程和队列都已饱和：执行拒绝策略
maximumPoolSize 可能不生效

使用无界队列时，任务通常会一直入队，线程数不会因为队列饱和而增长到最大值。线程参数必须与队列类型一起理解。

02

SEVEN KNOBS

核心参数分别控制什么

corePoolSize

常驻工作线程的目标数量，默认按任务到达逐步创建。

基础并发度
maximumPoolSize

队列无法接收任务时允许增长到的最大线程数量。

峰值并发上限
keepAliveTime

超过核心数量的空闲线程等待新任务的最长时间。

回收额外线程
workQueue

核心线程繁忙时保存待执行任务，容量决定可承受的排队量。

等待区
threadFactory

控制线程名称、优先级、守护属性及未捕获异常处理等。

可诊断性
handler

线程与队列均饱和或线程池关闭时，决定如何处理新任务。

03

QUEUE STRATEGY

三类队列策略

直接移交

SynchronousQueue 不存储任务，必须直接交给可用线程，容易快速扩容或拒绝。

SynchronousQueue
无界队列

可以吸收短时突发，但持续过载时队列可能不断增长并放大延迟与内存风险。

LinkedBlockingQueue()
有界队列

把可接受的等待任务数量显式化，便于实施背压和容量告警。

ArrayBlockingQueue
优先队列

任务必须具备稳定的优先级规则，同时避免低优先级任务长期饥饿。

PriorityBlockingQueue

后端服务通常更需要有界队列，因为“拒绝并快速暴露过载”往往比把请求积压到超时或内存不足更可控。

04

REJECTION

拒绝策略是业务决策

  • ✦AbortPolicy：抛出异常，显式暴露过载
  • ✦CallerRunsPolicy：由提交线程执行，形成一定反压
  • ✦DiscardPolicy：静默丢弃，必须明确业务允许
  • ✦DiscardOldestPolicy：丢弃队首后重试，可能破坏顺序
不要默认静默丢任务

任务是否允许丢弃、能否重试、是否幂等、如何记录失败，需要由业务语义决定。自定义策略至少要记录指标，并避免在拒绝回调中执行不可控的阻塞操作。

05

BOUNDED EXAMPLE

一个可诊断的有界线程池

显式配置线程池JAVA
/**
 * 创建用于发送通知的有界线程池。
 */
public final class NotificationExecutors {

    private NotificationExecutors() {
    }

    /**
     * 使用可识别的线程名称、有界队列和显式拒绝策略。
     */
    public static ThreadPoolExecutor create() {
        AtomicInteger sequence = new AtomicInteger();
        ThreadFactory factory = task -> {
            Thread thread = new Thread(task);
            thread.setName("notification-worker-" + sequence.incrementAndGet());
            thread.setUncaughtExceptionHandler((current, error) ->
                    System.err.println(current.getName() + ": " + error.getMessage()));
            return thread;
        };

        return new ThreadPoolExecutor(
                4,
                8,
                60,
                TimeUnit.SECONDS,
                new ArrayBlockingQueue<>(200),
                factory,
                new ThreadPoolExecutor.AbortPolicy());
    }
}

示例参数只是结构演示，不是生产默认答案。真实数值必须根据请求速率、任务耗时、下游容量、内存预算和可接受等待时间测量。

06

CAPACITY PLANNING

线程数和队列如何估算

CPU 密集任务通常受核心数限制，I/O 密集任务会在等待期间释放 CPU 使用机会，但线程数也不能无限增加。可先用吞吐目标和任务服务时间估算在途任务，再通过压测观察 CPU、队列、上下文切换和下游负载。

  • 01记录任务平均值、P95、P99 执行时间
  • 02明确峰值提交速率与可接受排队时间
  • 03确认数据库、HTTP 下游的并发容量
  • 04压测时同时观察吞吐、延迟与资源曲线
队列不是免费的缓冲区

队列越大，拒绝发生得越晚，但过载时请求等待时间更长。容量设计应优先满足服务等级，而不是追求“永不拒绝”。

07

OPERATIONS

监控和优雅关闭

  • ✦当前池大小、活跃线程数、最大历史线程数
  • ✦队列大小、剩余容量、任务等待时间
  • ✦已完成任务数、拒绝次数和任务异常数
  • ✦shutdown 后等待既有任务，超时再 shutdownNow
关闭流程JAVA
executor.shutdown();
try {
    // 给正在执行和排队的任务留出收尾时间
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        executor.shutdownNow();
    }
} catch (InterruptedException exception) {
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}
