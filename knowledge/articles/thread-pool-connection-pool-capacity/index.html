工程档案 · Capacity
01

REQUEST CHAIN

一次数据库请求经过哪些许可

  1. 01入口服务器接受请求并分配处理线程
  2. 02业务可能提交到独立线程池执行
  3. 03任务从连接池借用数据库连接
  4. 04数据库调度会话并执行 SQL
  5. 05事务结束、归还连接、释放线程

线程池上限控制应用并发，连接池上限控制数据库并发。如果 200 个工作线程同时争夺 20 个连接，剩余线程会在连接获取处阻塞，并继续占用内存和请求预算。

02

NESTED QUEUES

多层队列如何放大延迟

入口排队

HTTP 工作线程或事件循环后方已有第一层等待。

request queue
业务线程池排队

无界队列暂时避免拒绝，却让任务等待时间不可控。

executor queue
连接池排队

活跃连接达到上限后，工作线程再次阻塞。

connection wait
数据库内部等待

锁、I/O、CPU 调度形成最后一层等待。

database wait
队列只吸收短峰值

持续到达率超过处理能力时，队列只能推迟失败。有限队列和明确拒绝可以更快保护系统。

03

CAPACITY BUDGET

从数据库预算反推实例配置

全局关系FORMULA
数据库应用连接预算
> 最大实例数 × 单实例连接池上限
  + 定时任务、迁移、管理等额外连接

单实例数据库并发
≤ 单实例连接池上限
  • 01压测找到数据库 CPU、I/O 或锁竞争的吞吐拐点
  • 02为故障切换、扩容和管理连接保留余量
  • 03将数据库型任务的并发限制在可用连接附近
  • 04CPU 计算与数据库阻塞任务使用不同隔离池

连接池略小于相关工作线程并非必然错误，但差距越大，等待连接的线程越多。应按任务中数据库占比、事务时间和目标吞吐测量，而不是套用固定倍数。

04

TIMEOUT BUDGET

让内部失败早于外部请求

预算示例TEXT
HTTP 请求总预算：2000 ms
  线程池排队：最多 100 ms
  获取连接：最多 200 ms
  SQL 执行：最多 1200 ms
  序列化与异常处理：保留 500 ms

示例仅表达顺序，不是通用配置。若连接获取超时或 SQL 超时大于 HTTP 总超时，客户端离开后内部工作仍可能继续占用线程与连接。

05

BULKHEAD

用隔离与拒绝保护关键链路

  • ✦报表、批量任务与在线交易使用不同执行器和连接预算
  • ✦队列设置明确上限，拒绝时返回可识别的过载错误
  • ✦调用方只对幂等请求做有限、带退避的重试
  • ✦限流发生在昂贵工作之前，不把压力全部推到数据库
  • ✦自动扩容要同时核算扩容后的数据库总连接上限
06

OBSERVABILITY

把等待分层记录

  • 01线程池 active、queue、completed、rejected 与任务等待时长
  • 02连接池 active、idle、pending、timeout 与持有时长
  • 03SQL 延迟、事务时长、锁等待、数据库 CPU 与 I/O
  • 04按接口区分资源消耗，避免平均值掩盖热点
  • 05压测同时观察吞吐、P99、错误率和资源拐点
只看利用率不够

90% 活跃可能是高效运行，也可能是即将饱和。等待时长、拒绝率和数据库端指标才能说明是否需要调整容量。
