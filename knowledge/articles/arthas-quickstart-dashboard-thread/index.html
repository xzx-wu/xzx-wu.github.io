01

SAFETY

先确认权限与影响

  • ✦使用与目标 JVM 相同的操作系统用户连接，避免扩大权限
  • ✦记录目标 PID、实例、时间窗口、问题现象和操作人
  • ✦优先执行 dashboard、thread、jvm 等只读命令
  • ✦生产环境限定类名、方法、次数和采样时间
  • ✦不要把 Arthas 的远程端口直接暴露到公网
工具也会消耗资源

线程采样、对象展开、堆转储和字节码增强都可能增加 CPU、内存或停顿。每条命令执行前先看帮助和影响范围。

02

ATTACH

下载启动器并选择目标 JVM

本机连接SHELL
curl -O https://arthas.aliyun.com/arthas-boot.jar
java -jar arthas-boot.jar

# 出现 Java 进程列表后，输入目标进程前的序号

Attach 失败时依次检查：目标进程是否存在、用户是否一致、容器 PID 命名空间是否一致、临时目录权限以及 JDK Attach 能力。不要仅凭进程名选择，在多实例机器上应再次核对启动参数。

03

DASHBOARD

获得 JVM 全局快照

观察三次，每次间隔五秒ARTHAS
dashboard -i 5000 -n 3
线程区

查看高 CPU 线程、线程状态和累计时间。

CPU / STATE
内存区

关注堆、非堆、直接内存和已使用比例。

查看收集器、次数、耗时和最近变化。

count / time
运行区

核对启动时间、运行时长、处理器和 Java 版本。

runtime

Dashboard 是采样视图，不应只凭一次快照下结论。把它和业务延迟、流量、容器 CPU 限制及系统指标放到同一时间线。

04

THREAD

从高 CPU、阻塞和死锁切入

常用线程命令ARTHAS
thread -n 5 -i 1000
thread -b
thread --state BLOCKED
thread 42
  • 01thread -n 通过采样找出最繁忙线程并打印栈
  • 02thread -b 尝试定位阻塞其他线程的持锁线程
  • 03按状态过滤可发现 BLOCKED、WAITING 或 TIMED_WAITING 聚集
  • 04指定线程 ID 查看完整栈，并与日志 traceId、请求时间对齐

高 CPU 栈要连续采样：同一业务栈持续出现才具有更强证据。偶尔采到 GC、JIT 或 Arthas 自身线程不等于根因。

05

RUNTIME FACTS

核对真实运行环境

JVM 与环境信息ARTHAS
jvm
sysprop
sysenv
vmoption
logger
  • ✦jvm 查看 JVM、类加载、编译器、GC 与线程概况
  • ✦sysprop 核对 profile、编码、临时目录等系统属性
  • ✦sysenv 只读取必要环境变量，输出中可能包含敏感信息
  • ✦vmoption 先查看而不是直接修改诊断选项
  • ✦logger 可确认实际日志实现、级别和配置来源
06

WORKFLOW

第一现场排查顺序

  1. 01记录告警时间、接口、实例、流量与最近变更
  2. 02用 dashboard 判断主要矛盾在 CPU、线程、内存还是 GC
  3. 03用 thread 对热点或阻塞栈进行多次采样
  4. 04通过 jvm、sysprop、logger 核对运行事实
  5. 05形成可验证假设后，再进入 watch、trace 或 JFR
  6. 06保存关键输出并执行 stop 退出
