BUILT-IN LOADERS

JDK 内置类加载器层次

  • ✦Bootstrap 加载器由虚拟机实现，加载关键 Java 运行时类
  • ✦Platform 加载器负责平台类与相关模块
  • ✦System/Application 加载器通常加载应用类路径和模块路径内容
  • ✦自定义 ClassLoader 可以从网络、插件目录或加密资源定义类
查看加载器JAVA
ClassLoader loader = ArchiveService.class.getClassLoader();
while (loader != null) {
    System.out.println(loader);
    loader = loader.getParent();
}
// Bootstrap 加载器通常以 null 表示
03

DELEGATION

先交给父加载器查找

ClassLoader.loadClass 的常见委派模型是：先检查是否已加载，再委派父加载器，父加载器找不到时由当前加载器查找。这样可以避免应用随意替换核心类，并提高共享类的一致性。

“双亲”是惯用中文名称

模型本质是父级委派链，并不要求每个自定义加载器都严格父优先。容器、插件和模块系统可能根据隔离目标调整查找顺序。

两个插件各自定义 com.example.Api，即使字节码相同，只要定义加载器不同，JVM 仍把它们视为不同类型。常见表现是看似“同类”的 ClassCastException、ServiceLoader 找不到实现或静态单例重复。

05

SPECIAL CASES

为什么会打破父优先

  • 01插件需要使用不同版本依赖，采用 child-first 隔离
  • 02Java SPI 由上层接口发现下层应用实现，使用线程上下文类加载器
  • 03应用服务器隔离不同部署单元，避免依赖互相污染
  • 04热部署使用新加载器加载新类，旧加载器需能被完整回收

线程、ThreadLocal、静态集合和后台任务若仍引用旧加载器及其类，会导致 Metaspace 与应用对象共同泄漏。
