诊断档案 · Class Loading
01

SYMPTOMS

哪些问题指向类加载

  • ✦NoSuchMethodError、NoClassDefFoundError 或签名不匹配
  • ✦本地修复已发布，线上行为仍像旧代码
  • ✦同名类由不同 ClassLoader 加载导致类型无法转换
  • ✦资源文件存在于 JAR，却出现 ResourceNotFound
  • ✦热部署后类数量或 Metaspace 持续增长
02

CLASSLOADER

查看层级、实例与资源来源

定位加载器ARTHAS
classloader
classloader -t
classloader --url-classes --jar order-service -n 30
classloader -c 2a139a55 -r application.yml

classloader 可以统计各加载器的实例与已加载类数量，查看层级、URL 和资源。输出全部类可能非常庞大，应使用 JAR、类名和数量参数限制范围。

03

SC / SM / JAD

确认类来源、方法和实际字节码

运行时类证据ARTHAS
sc -d com.example.order.OrderService
sm -d com.example.order.OrderService submit
jad --source-only com.example.order.OrderService
sc -d

查看 code-source、ClassLoader、接口、父类和类加载信息。

where
sm -d

确认 JVM 中真实存在的方法签名和描述符。

signature
jad

反编译已加载字节码，核对分支、常量和调用是否符合发布版本。

actual code

反编译源码不一定恢复原变量名或语法结构，重点比较控制流、调用目标、常量和异常处理，而不是逐行文本。

04

RETRANSFORM

热更新是应急操作，不是发布流程

加载与清理条目ARTHAS
retransform /tmp/OrderService.class
retransform -l
retransform -d 1
retransform --classPattern com.example.order.OrderService

Retransform 将外部 class 字节码应用到已加载类。执行前必须核对类名、包名、目标 JDK、依赖、ClassLoader hash 和编译产物。常见 JVM 限制下不能随意增加或删除字段、方法，也不能把结构变更当作普通方法体替换。

stop 不一定恢复热更新

官方文档说明，若未删除 retransform 条目并再次触发转换，执行 Arthas stop 后变更仍可能继续生效。必须按记录 ID 清理并验证。

05

SAFE WORKFLOW

生产热修复最小流程

  1. 01先使用 sc、classloader、jad 证明当前类确实错误
  2. 02在同 JDK、同依赖和同源码基线下编译目标 class
  3. 03在隔离环境验证方法行为、异常和兼容性
  4. 04准备回滚 class、变更窗口、负责人和业务验证项
  5. 05使用精确 ClassLoader 执行，立即验证关键接口
  6. 06尽快通过正式构建发布，避免运行态与仓库长期漂移
06

BOUNDARIES

哪些情况不要热改

  • ✦需要新增字段、方法、父类或接口的结构性变更
  • ✦涉及数据库结构、消息协议或多个服务共同升级
  • ✦无法准确识别目标 ClassLoader 的多应用容器
  • ✦没有回滚字节码、业务验证和操作审计
  • ✦根因尚未确认，只想“试一下”不同代码

这类场景应走正式发布、灰度和回滚流程。Arthas 更适合证明运行事实，而不是替代版本管理。
