DEPENDENCY GRAPH

直接依赖与传递依赖共同形成类路径

项目声明的依赖是直接依赖，这些组件继续引入传递依赖。相同坐标出现多个版本时，Maven 依据依赖调解选择版本；通常使用依赖树中路径更近的声明，路径相同时较早声明者生效。显式 dependencyManagement 可以统一版本选择。

“较近版本获胜”不意味着它一定兼容。最终选中的 API 可能与编译某个库时使用的版本不同，导致 NoSuchMethodError、ClassNotFoundException 或行为变化。

02

SCOPE & OPTIONAL

作用域决定依赖在哪些类路径出现

  • ✦compile：默认作用域，参与编译、测试和运行并可传递
  • ✦runtime：编译主代码不需要，运行和测试需要
  • ✦provided：编译测试可见，运行时预期由容器或 JDK 提供
  • ✦test：只用于测试编译与执行，不传递给使用者
  • ✦optional：当前项目使用，但不自动传递给下游消费者

exclusion 应用于排除确实不需要或冲突的特定传递依赖，并在项目中显式声明替代版本。不要用大量排除隐藏依赖设计问题，也不要把必要运行库错误标为 provided。

03

DEPENDENCY TREE

先证明版本从哪里来，再修改 POM

依赖冲突诊断MAVEN
mvn dependency:tree
mvn dependency:tree -Dverbose
mvn dependency:tree -Dincludes=com.fasterxml.jackson.core:*
mvn help:effective-pom
  1. 01从异常中的类与方法定位组件坐标
  2. 02在依赖树中查看选中版本、遗漏版本与引入路径
  3. 03检查父 POM、BOM、Profile 和 effective POM 的版本来源
  4. 04确认打包产物、容器或插件没有额外覆盖类路径
  5. 05使用最小版本对齐或精确 exclusion 修复并回归
04

DEPENDENCY MANAGEMENT

dependencyManagement 管版本，不自动引入依赖

父 POM 可在 dependencyManagement 中定义组织统一版本，子模块仍需在 dependencies 中声明实际使用的组件。BOM 通过 import scope 导入一组经过验证的版本组合，适合 Spring、Jackson 或云 SDK 等多模块生态。

不要为 BOM 管理的模块逐个覆盖版本

局部覆盖可能破坏模块之间的兼容组合。确需升级时优先升级整个 BOM，并通过集成测试验证。

应用项目可以严格统一版本；可复用库应尽量减少传递依赖和版本强制，避免把内部实现选择扩散给所有使用者。

05

ENFORCER

让依赖收敛问题在 CI 阶段失败

Maven Enforcer 的 dependencyConvergence 规则可以要求依赖图中的同一坐标收敛到一致版本，帮助在运行前发现潜在不一致。它比“最终选出一个版本即可”更严格，也可能暴露需要 BOM、排除或升级才能解决的历史依赖。

  • ✦在父工程统一 Enforcer 与规则版本
  • ✦对已知例外给出精确范围、原因和到期计划
  • ✦同时检查 Java 与 Maven 最低版本、重复类和禁止依赖
  • ✦规则升级先在全量模块验证，避免突然阻塞发布
06

BUILD GOVERNANCE

构建工具链也需要版本与产物边界

  • ✦使用 Maven Wrapper 固定团队构建入口和 Maven 版本
  • ✦锁定插件版本，避免默认版本随环境改变
  • ✦区分 dependencies 与 build plugins，分别治理
  • ✦CI 从干净环境构建并保存依赖树、测试与产物摘要
  • ✦使用可信仓库、校验和与依赖来源控制供应链风险
  • ✦Shade 只用于明确的可执行隔离场景，并处理服务资源合并

可重复构建要求同一源码与配置得到可解释的依赖和产物。若构建依赖本机缓存、动态版本或未锁定插件，线上问题就难以复现。
