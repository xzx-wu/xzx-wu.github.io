文档模型不是把关系表换成 JSON，而是让一次业务读写拥有清晰边界。

文档档案 · Modeling
01

WORKLOAD FIRST

先列出读写模式

  • ✦哪些字段总是一起读取，哪些只在详情页读取
  • ✦写入是整体替换、局部更新，还是持续追加数组
  • ✦关系是一对一、一对少量，还是无界一对多
  • ✦查询需要哪些过滤、排序、分页和聚合
  • ✦数据保留多久，是否需要 TTL、归档或分片

只有把高频请求、结果大小和增长上限写出来，才能判断哪些信息适合放在同一文档中。

02

EMBED OR REFERENCE

嵌入优化局部性，引用控制增长

适合嵌入

生命周期一致、经常一起读取、数量有界，并希望单文档原子更新。

order + snapshot items
适合引用

实体独立复用、独立更新、关系规模大或数组可能无界增长。

authorId / productId
适度冗余

复制稳定展示字段换取少一次查询，同时建立明确更新机制。

name snapshot
无界数组是危险信号

日志、消息、评论等持续增长集合通常应拆成独立 collection，通过父实体标识查询，而不是无限追加到一个文档。

03

SCHEMA EVOLUTION

灵活结构也要有契约

集合验证示例MONGOSH
db.createCollection("orders", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["userId", "status", "createdAt"],
      properties: {
        status: { enum: ["CREATED", "PAID", "CLOSED"] },
        schemaVersion: { bsonType: "int" }
      }
    }
  }
})

应用读取旧文档时应兼容明确版本，并通过可重试脚本渐进迁移。不要依赖每次读取都隐式修复，否则写放大和失败重试会变得难以追踪。

04

INDEX TYPES

索引服务于查询，不服务于字段数量

  • 01单字段索引处理单一过滤或排序
  • 02复合索引覆盖固定组合的过滤与排序
  • 03数组字段索引会成为 multikey，需评估条目放大
  • 04部分索引只收录满足条件的文档，减少体积
  • 05TTL 索引按时间自动清理适合会话、临时事件等数据

每个索引都会占用内存和磁盘，并增加写入维护成本。低价值重复索引应通过查询证据和使用统计审查后移除。

05

COMPOUND INDEX

字段顺序决定可利用的前缀

订单列表索引MONGOSH
db.orders.createIndex({
  userId: 1,
  status: 1,
  createdAt: -1,
  _id: -1
})

db.orders.find({ userId: 1001, status: "PAID" })
  .sort({ createdAt: -1, _id: -1 })
  .limit(20)

常用经验是先等值字段，再考虑排序和范围字段，但最终要用实际过滤组合、选择性和 explain 验证。复合索引通常支持其前缀，跳过前导字段会削弱利用效果。

06

VERIFICATION

用真实数据规模验证

  • ✦准备接近生产分布的数据，而不是几十条样例
  • ✦检查 explain 中扫描文档、索引键和返回数量
  • ✦验证排序是否由索引完成，分页是否保持稳定顺序
  • ✦评估文档平均大小、最大增长、索引总量与工作集
  • ✦在读写混合压测中观察 P95/P99 延迟和复制压力
