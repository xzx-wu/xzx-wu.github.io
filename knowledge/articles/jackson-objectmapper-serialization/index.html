02

OBJECT MAPPER

配置完成后复用，不在请求中临时改规则

ObjectMapper 是配置与缓存的中心。应在首次读写前注册模块、命名策略和特性，之后作为共享对象复用；不同协议需要不同规则时，创建边界清晰的 Mapper 或不可变 Reader/Writer，而不是并发修改全局 Mapper。

  • ✦应用启动阶段完成模块与 Feature 配置
  • ✦API、消息、数据库 JSON 按契约隔离配置
  • ✦使用 BOM 对齐 core、annotations、databind 等模块版本
  • ✦Jackson 2.x 与 3.x 的包名和部分 API 不同，升级时不能混用示例
不要每次请求 new ObjectMapper

重复配置增加成本，也容易让不同调用出现细微规则差异。统一构建、明确命名并通过依赖注入复用更可控。

03

GENERIC TYPES

运行时必须保留完整泛型信息

Java 类型擦除后，只有 List.class 无法表达元素类型。读取集合或嵌套结果时，需要 TypeReference 或 TypeFactory 构造完整 JavaType，否则元素可能变成 Map，错误会延迟到业务强制转换阶段。

类型表达示意TYPE
目标：List<Order>

不完整：List.class
完整：TypeReference<List<Order>>
或：TypeFactory 构造 CollectionType

公共 SDK 不应只返回 Map<String, Object> 逃避建模。稳定字段使用明确类型，真正动态的扩展字段才保留树节点或受控 Map。

04

DATA CONTRACT

时间、数字、null 与未知字段必须有约定

  • ✦时间使用带时区语义的 ISO-8601 或明确约定格式
  • ✦金额与高精度数值使用 BigDecimal 或字符串契约，避免浮点误差
  • ✦区分字段缺失、显式 null、空字符串与空集合
  • ✦未知字段是拒绝、忽略还是保留，需要按协议演进策略决定
  • ✦枚举新增值应有前向兼容策略，不能默认映射到错误业务含义

对外接口应通过契约测试固定序列化结果，包括字段名、时区、精度、默认值和敏感字段排除，而不只测试“反序列化后对象不为空”。

05

MODEL DESIGN

注解用于表达契约，不用于掩盖失控模型

属性、构造器、Record 与命名策略都可以参与数据绑定。优先使用不可变模型和明确构造路径；第三方类型可通过 Mix-in 或自定义模块适配，避免为了 JSON 侵入业务核心模型。

View

06

SECURITY

反序列化不可信 JSON 必须限制能力与资源

  • ✦不要对不可信输入启用宽泛的默认多态类型
  • ✦需要多态时使用明确的类型集合与受控标识
  • ✦限制请求体大小、嵌套深度、字符串与集合规模
  • ✦异常响应不回显完整输入、密钥或内部类名
  • ✦持续更新 databind 与相关模块，并使用统一版本
  • ✦模型中显式排除密码、Token、内部权限和审计字段

多态反序列化会把输入中的类型信息转为类实例化能力，配置范围越宽，攻击面越大。外部协议更适合显式 type 字段加固定映射，而非允许任意 Java 类型。
