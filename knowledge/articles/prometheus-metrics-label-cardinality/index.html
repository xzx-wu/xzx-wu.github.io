01

METRIC TYPES

选择与被测量现象一致的类型

Counter

只增加或重置的累计量，如请求数、错误数和处理字节。

rate(counter[5m])
Gauge

可增可减的瞬时状态，如队列长度、连接数和温度。

gauge
Histogram

在可配置桶中累计观测值，可跨实例聚合分位数。

_bucket / _sum / _count
Summary

客户端计算滑动窗口分位数，分位结果通常不能直接跨实例聚合。

quantile

状态切换次数使用 Counter，当前状态数量使用 Gauge；不要对可能下降的值使用 Counter，也不要把累计请求数当 Gauge。

02

NAMING & UNITS

名称描述实体、含义与基础单位

  • ✦统一应用或库前缀，避免全局命名冲突
  • ✦单位采用秒、字节等基础单位并写入后缀
  • ✦累计量使用 _total 约定
  • ✦同一指标所有标签维度含义一致
  • ✦帮助文本说明计数对象与边界
  • ✦指标名与标签组合应形成可读句子
命名示例METRIC
http_server_request_duration_seconds
job_processed_total{result="success"}
db_pool_connections{state="active"}
03

LABEL CARDINALITY

标签用于聚合维度，不用于保存事件身份

标签组合的笛卡尔积决定时间序列数量。路径模板、方法、状态族通常可控；用户 ID、订单号、完整 URL、traceId 和异常消息接近无界，会制造高基数。

  • ✦使用 /users/{id} 路由模板，不用真实路径
  • ✦状态码按业务需要选择精确码或状态族
  • ✦异常按稳定类型归类，不用完整消息
  • ✦上线前估算标签组合和实例数量
  • ✦监控活跃序列、样本摄入和存储增长
  • ✦高维事件进入日志或 Trace，而不是 Metric
标签值来自外部输入时必须归一化

未受控的 URI、Header 或业务字段会把攻击流量转化为监控系统的资源消耗。

04

OBSERVATION MODEL

围绕请求、资源和业务结果建立指标

RED

面向请求服务：Rate、Errors、Duration。

service
USE

面向资源：Utilization、Saturation、Errors。

resource
Online serving

统计请求、错误、延迟、并发和依赖结果。

request
Batch jobs

记录最近成功时间、处理量、失败与持续时间。

job

客户端和服务端都应区分“尝试”和“完成”；若失败没有增加总尝试计数，错误率分母会被低估。业务指标还要明确事件定义和重复处理规则。

05

DISTRIBUTIONS

Histogram 的桶边界应围绕业务阈值设计

Histogram 在每个桶中累计观测数量，可用 histogram_quantile 估算聚合分位数。桶过密增加序列，桶过稀降低阈值附近精度；边界应覆盖实际分布和 SLO 阈值。

  • ✦先用数据观察分布，再调整桶
  • ✦把目标延迟阈值设置为明确桶边界
  • ✦保留 _sum 和 _count 计算平均值
  • ✦跨实例分位数优先选择可聚合 Histogram
  • ✦Summary 仅在客户端分位有明确必要时使用
  • ✦记录缺失、重置和抓取间隔对查询的影响

OPERATIONS

指标治理与发布检查

  1. 01为新指标记录负责人、用途、类型、单位和标签

  5. 05清理废弃指标前核对规则、面板和消费者
  6. 06把指标基数增长纳入容量与变更评审
