client

message
Transport

负责连接、复用、帧与流控。

network

在服务端定位方法并恢复上下文。

server
Executor

执行业务并受线程、队列与截止时间限制。

work
Status

任何一层都可能失败。定位时要区分客户端排队、DNS、建连、TLS、发送、服务端排队、业务执行和响应读取，而不是把所有异常统一记为“RPC 调用失败”。

02

SCHEMA CONTRACT

序列化格式是跨版本公共 API

Schema 定义字段编号、类型和服务方法。兼容演进应保留旧字段编号，不改变既有字段的语义，并对未知字段和新增枚举值制定前向兼容策略。

  • ✦请求与响应使用明确领域类型，避免通用 Map 承载稳定字段
  • ✦字段缺失与默认值的含义必须可区分
  • ✦删除字段后保留编号，防止未来被错误复用
  • ✦错误使用稳定状态码与业务详情，不要求解析文本
  • ✦生成代码与运行库版本通过 BOM 或工具链对齐
兼容不等于业务语义不变

消息仍能解析，只说明结构兼容。默认值、枚举解释和校验规则变化仍可能破坏调用方。

03

DEADLINE

传播剩余时间，而不是每层重新计时

Deadline 表示调用允许完成的最晚时间。客户端应显式设置，服务端在继续调用下游时传播剩余预算；若每层都设置独立固定超时，深层调用可能在上游早已放弃后继续消耗资源。

截止时间传播BUDGET
入口剩余 1200 ms
  → 服务 A 排队与处理后剩 900 ms
  → 调用服务 B 使用剩余预算
  → B 超过截止时间后停止非必要工作

收到 DEADLINE_EXCEEDED 只表示客户端没有按时得到完成结果，不能证明服务端没有执行。写操作仍需要幂等键、状态查询或补偿机制。

04

RETRY & CANCELLATION

恢复策略必须尊重幂等与总预算

  • ✦仅重试明确暂时性状态，并限制次数、退避和总时间
  • ✦写操作由业务声明幂等语义，不根据方法名猜测
  • ✦取消信号向下游传播，服务端协作停止可中断工作
  • ✦客户端取消不能保证已经发出的副作用被回滚
  • ✦SDK、代理和服务端避免多层重试叠加

重试、Hedging 和请求复制都会增加真实流量。只有在可证明降低尾延迟且下游容量允许时使用，并为每次尝试保留同一逻辑 requestId 与独立 attempt 编号。

05

STREAMING

Unary 与 Streaming 的资源语义不同

RPC 可以是单请求单响应、服务端流、客户端流或双向流。流式调用需要处理半关闭、背压、消息顺序、单消息与总流量上限，以及连接中断后的恢复位置。

Unary

边界清晰，适合普通命令与查询。

1 → 1
Server Stream

客户端发送一次，服务端持续返回。

1 → N
Client Stream

客户端分段发送，服务端汇总响应。

N → 1
Bidi Stream

双方独立读写，状态机最复杂。

N ↔ N

流式接口不是无限传输通道。必须限制在途消息、单流时长和缓冲区，并在断线重连时使用业务游标而非假设自动无损续传。

06

SECURITY & OBSERVABILITY

让远程边界可认证、可限流、可追踪

  1. 01使用 TLS 或 mTLS 校验服务身份并管理证书轮换
  2. 02服务端重新校验调用身份、权限与数据范围
  3. 03限制消息大小、并发流、队列与每方法速率
  4. 04Tracing 记录服务、方法、状态和耗时，凭据不进入标签
  5. 05指标区分逻辑调用、实际尝试、Deadline 与取消
  6. 06日志关联 requestId、peer 与状态，正文默认不记录
