01

TYPE SAFETY

把转换风险交给编译器

泛型容器JAVA
List<String> names = new ArrayList<>();
names.add("Arcane");

String first = names.get(0);

参数化类型让集合的写入类型与读取类型保持一致，减少显式强制转换。泛型类、接口和方法都可以声明类型变量，并通过上界限制可使用的能力。

02

INVARIANCE

List<Integer> 不是 List<Number>

Java 泛型默认不变。若允许把 List<Integer> 赋给 List<Number>，调用方就能向其中加入 Double，从而破坏原集合的元素约束。

不变性JAVA
List<Integer> integers = List.of(1, 2, 3);

// 编译错误：参数化类型之间不存在这种继承关系
// List<Number> numbers = integers;

List<? extends Number> readable = integers;
03

PECS

Producer Extends，Consumer Super

复制方法JAVA
/**
 * 把来源中的元素复制到目标集合。
 */
public static <T> void copy(
        List<? extends T> source,
        List<? super T> target) {
    for (T element : source) {
        target.add(element);
    }
}
  • ✦? extends T 适合读取 T，但不能安全写入具体 T
  • ✦? super T 适合写入 T，读取时只能当作 Object
  • ✦既读又写同一种精确类型时直接使用 List<T>
04

TYPE ERASURE

编译后发生了什么

参数化类型和类型变量在编译后会被擦除到原始类型或左侧上界，编译器在必要位置插入转换，并可能生成桥接方法维持重写后的多态语义。

擦除概念JAVA
class Box<T extends Number> {
    private T value;
}

// 类型变量 T 的擦除是其最左上界 Number
// 运行时不会为 Box<Integer> 生成独立的类
擦除不等于所有签名信息消失

类文件可以保留泛型签名元数据供反射读取，但普通对象在运行时仍不会携带完整的参数化类型实参。

05

RUNTIME LIMITS

由擦除带来的限制

  • 01不能直接 new T()，构造方式需要显式传入
  • 02不能创建 new List<String>[10] 这类泛型数组
  • 03不能用 instanceof List<String> 检查类型实参
  • 04两个只在类型参数上不同的方法可能擦除为相同签名
  • 05原始类型会绕开部分检查并产生 unchecked 警告
显式构造器JAVA
public static <T> T create(Supplier<T> factory) {
    return factory.get();
}

Order order = create(Order::new);
06

API DESIGN

让类型参数表达关系

  • ✦类型参数只出现一次且不建立关系时，考虑直接使用通配符
  • ✦方法级关系优先声明在方法上，不必扩大到整个类
  • ✦公共 API 避免返回原始类型和未经检查的强制转换
  • ✦无法证明安全的 unchecked cast 要缩小范围并写明不变量
