01

QUERY VS FILTER

是否需要计算相关性

Query context

判断匹配程度并计算 _score，适合全文检索与排名。

relevance
Filter context

只判断匹配与否，不贡献评分，适合状态、范围、权限和租户条件。

yes / no

filter 条件更容易被缓存，但不应为了缓存强行把需要评分的全文查询改成 filter。先保证语义，再关注执行代价。

02

TERM & MATCH

精确词项与分析文本

两类查询HTTP
GET /articles/_search
{
  "query": {
    "bool": {
      "must": [
        {"match": {"title": "Java concurrency"}}
      ],
      "filter": [
        {"term": {"status": "PUBLISHED"}}
      ]
    }
  }
}

match 会分析输入，适合 text；term 直接查精确词项，适合 keyword、数值、布尔和日期。对 text 字段盲用 term，常因词项与原始输入不同而查不到。

03

BOOLEAN QUERY

must、should、filter、must_not

  • ✦must 必须匹配并参与评分
  • ✦should 用于可选匹配或提升评分，是否必需受 minimum_should_match 影响
  • ✦filter 必须匹配但不贡献评分
  • ✦must_not 排除文档，运行在 filter context
权限条件必须进入查询

租户、数据范围和可见状态应作为服务端 filter，不能先搜索全部结果再由前端隐藏。

04

PAGINATION

搜索结果也要避免深 from

from + size 越深，协调节点与分片需要保留和合并的候选越多。连续翻页应使用稳定排序与 search_after；需要一致视图时结合 point in time。

search_afterHTTP
GET /articles/_search
{
  "size": 20,
  "sort": [
    {"createdAt": "desc"},
    {"_id": "desc"}
  ],
  "search_after": ["2026-07-17T10:00:00Z", "900001"]
}
