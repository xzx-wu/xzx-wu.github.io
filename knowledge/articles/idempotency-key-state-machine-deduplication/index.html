分布式档案 · Idempotency
01

SEMANTICS

幂等描述重复执行后的有效状态

一个操作执行一次与执行多次，对系统产生的预期有效状态相同，才具备幂等语义。HTTP 规范将方法幂等性定义为多次相同请求的预期效果与一次相同，但响应、日志和计费等附带行为仍可能不同。

读取

通常无业务副作用，但访问日志和限流计数会变化。

safe
覆盖更新

把资源设为确定值通常比“增加一次”更容易幂等。

set
创建

需要业务唯一键或幂等键识别同一意图。

create
扣减

必须绑定订单或流水，不能直接重复执行减法。

effect

幂等是业务契约而非 HTTP 方法名称。即使使用 PUT，若内部每次都新增流水或重复发送消息，整体仍不幂等。

02

IDEMPOTENCY KEY

键必须唯一标识一次业务意图

  • ✦由可信客户端生成高熵键，重试同一意图时保持不变
  • ✦作用域至少包含租户、调用者与操作类型，避免跨主体碰撞
  • ✦保存请求参数摘要，防止同一个键被用于不同内容
  • ✦记录处理状态、资源 ID、响应摘要和到期时间
  • ✦设置足以覆盖客户端重试与消息延迟的保留窗口
唯一约束示意DATA MODEL
UNIQUE(tenant_id, operation, idempotency_key)

request_hash  → 校验重复请求参数一致
status        → PROCESSING / SUCCEEDED / FAILED
resource_id   → 返回首次创建的业务资源
03

STATE MACHINE

唯一约束负责抢占，状态机负责合法推进

首次请求在数据库事务中插入幂等记录或业务唯一流水；并发重复请求由唯一约束竞争，而不是先查再插。业务状态使用条件更新，只允许从当前合法状态迁移到下一状态。

条件状态迁移SQL IDEA
UPDATE payment
SET status = 'SUCCEEDED', version = version + 1
WHERE id = ?
  AND status = 'PROCESSING'
  AND version = ?

受影响行数为零表示状态已变化或版本不匹配，需要读取当前结果再决策。不能无条件覆盖，也不能把分布式锁作为唯一持久正确性保障；锁租约到期后仍需数据库约束兜底。

04

RESULT REPLAY

重复请求要返回可解释的同一结果

SUCCEEDED

返回首次成功的资源标识与稳定响应。

replay
PROCESSING

返回处理中状态或状态查询地址，不再执行副作用。

pending
FAILED

区分可安全重试与确定业务失败。

decision
Mismatch

相同键但参数摘要不同，拒绝并记录审计。

conflict

若服务在副作用完成后、写入成功结果前崩溃，需要通过业务唯一约束、下游查询或对账恢复。只缓存 HTTP 响应而不保护业务事务，会留下重复执行窗口。

05

MESSAGE DEDUPLICATION

至少一次投递要求消费者幂等

消费者收到消息后，先基于 eventId 与 consumerName 建立唯一处理记录，再在同一数据库事务中更新业务状态。只有本地事务提交成功后才确认消息；重复消息读取已完成结果并安全确认。

  • ✦去重键包含消费组或处理器身份，避免不同业务互相覆盖
  • ✦消息 ID 由生产端稳定生成，重发时不更换
  • ✦乱序消息还需版本号或状态机，去重不能解决顺序
  • ✦去重记录清理晚于消息最大重投与归档周期
  • ✦外部副作用使用下游幂等键或可查询业务流水
06

TEST CHECKLIST

幂等验证清单

  1. 01同一键并发提交十次，只产生一次有效副作用
  2. 02相同键不同参数被明确拒绝并留下审计
  3. 03在事务提交前后分别注入超时与进程崩溃
  4. 04消息重复、乱序和延迟后仍满足业务状态机
  5. 05幂等记录过期、清理和归档不会过早开放重复窗口
  6. 06监控键冲突、处理中超时、重复命中和恢复失败
