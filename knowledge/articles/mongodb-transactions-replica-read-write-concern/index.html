一致性档案 · Transactions
01

SINGLE DOCUMENT

单文档写入天然原子

MongoDB 对单个文档的写操作是原子的。把需要共同变化且规模有界的数据放在一个文档中，通常比为每次请求开启多文档事务更简单、吞吐更高。

  • ✦更新条件同时包含当前状态，避免无条件覆盖并发变化
  • ✦使用 $inc、$set、$push 等更新操作表达服务端原子变更
  • ✦通过唯一索引保护业务唯一性，不只依赖应用先查询
  • ✦无界集合不要为追求原子性强行嵌入一个文档
02

MULTI-DOCUMENT

事务只包住必要的不变量

Java 驱动事务轮廓JAVA
try (ClientSession session = mongoClient.startSession()) {
    session.withTransaction(() -> {
        // 所有参与事务的读写都必须显式传入同一个 session
        orders.updateOne(session, orderFilter, orderUpdate);
        accounts.updateOne(session, accountFilter, accountUpdate);
        return "ok";
    });
}
  • 01事务保持短小，避免在事务内调用外部 HTTP 或等待人工操作
  • 02驱动回调可处理部分可重试错误，业务操作仍需设计幂等
  • 03限制事务读写量，关注锁、快照和复制压力
  • 04提交结果不确定时按错误标签和驱动规范处理，不盲目重复业务

REPLICA SET

Primary 接收写入，Secondary 复制 oplog

Primary

接受写操作并记录 oplog；事务中的读取也必须路由到 Primary。

write authority
Secondary

异步复制 oplog，可按读取偏好服务允许陈旧的数据。

replica
Election

Primary 不可用时，满足条件的成员选举新 Primary。

failover

副本集通常部署奇数投票成员并跨故障域分布。复制延迟、时钟、磁盘和网络质量都会影响选举与可见性。

04

READ & WRITE CONCERN

分别定义读到什么、写到什么程度

  • ✦read concern local：读取节点当前可见数据，延迟低但故障后可能回滚
  • ✦read concern majority：读取已被多数成员确认的数据
  • ✦read concern snapshot：事务或会话内读取一致时间点视图
  • ✦write concern w:1：主节点确认后返回，复制保障较弱
  • ✦write concern majority：等待多数有数据成员确认，提升持久性并增加延迟

确认级别应按数据价值设置，而不是全局追求最大或最小。支付状态与可重算缓存可能需要不同配置。

05

READ PREFERENCE

读取来源决定延迟与新鲜度

primary

默认选择，优先保证读到主节点当前状态。

freshness
secondaryPreferred

偏向副本读取，适合允许复制延迟的报表或非关键查询。

read scale
nearest

按网络延迟选择成员，不代表数据最新。

latency
读副本不是免费扩容

业务要能接受陈旧读；索引构建、缓存命中与长查询仍会消耗副本资源并影响复制。

06

OPERATIONS

一致性配置要配合观测与演练

  • 01监控复制延迟、oplog 窗口、选举次数与事务中止率
  • 02记录应用实际使用的 read/write concern 与 read preference
  • 03演练 Primary 故障、网络分区和提交结果不确定
  • 04对跨服务流程使用幂等键、状态机或补偿，不扩大数据库事务
  • 05把副本与可恢复备份分开建设并定期验证恢复
