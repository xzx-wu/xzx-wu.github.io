查询档案 · Aggregation
01

PIPELINE

每个阶段接收文档并输出文档

按用户汇总已支付订单MONGOSH
db.orders.aggregate([
  { $match: { userId: 1001, status: "PAID" } },
  { $project: { amount: 1, createdAt: 1 } },
  { $group: {
      _id: { $dateTrunc: { date: "$createdAt", unit: "day" } },
      totalAmount: { $sum: "$amount" },
      orderCount: { $sum: 1 }
  } },
  { $sort: { _id: -1 } },
  { $limit: 30 }
])

管道适合过滤、变形、分组、关联与排序。每增加一个阶段，都应明确输入规模、输出规模和是否能利用索引。

02

STAGE ORDER

尽早缩小进入后续阶段的数据

  • ✦尽可能让开头的 $match 使用索引过滤
  • ✦只传递后续需要的字段，减少大字段和网络负载
  • ✦$sort + $limit 尽量利用与过滤匹配的复合索引
  • ✦先限制候选集，再执行 $lookup、$unwind 和 $group
  • ✦优化器可能重排部分阶段，但设计时仍要表达清楚数据缩减意图
03

CARDINALITY

关注会放大数据量的阶段

$lookup

关联字段需有合适索引，并控制外表候选数和返回字段。

join fan-out
$unwind

一个数组元素变一条文档，数组长度直接放大后续工作量。

row expansion
$group / $sort

可能持有大量状态或排序数据，需关注内存、磁盘与结果基数。

blocking stage

allowDiskUse 可以让部分阶段使用临时磁盘，但它是容量保护手段，不是索引和模型问题的性能修复。

04

EXPLAIN

查看计划与实际执行统计

解释聚合执行MONGOSH
db.orders.explain("executionStats").aggregate([
  { $match: { userId: 1001, status: "PAID" } },
  { $sort: { createdAt: -1, _id: -1 } },
  { $limit: 20 }
])
  • 01nReturned：最终返回文档数
  • 02totalKeysExamined：扫描的索引键数量
  • 03totalDocsExamined：实际读取的文档数量
  • 04executionTimeMillis：本次执行时间，只能结合其他证据判断
  • 05winningPlan：获胜计划中的 IXSCAN、FETCH、COLLSCAN 等阶段
05

DIAGNOSIS

从扫描比定位问题

  1. 01确认查询条件、排序、投影、limit 与生产请求一致
  2. 02比较扫描键、扫描文档和返回文档的数量级
  3. 03COLLSCAN 先检查是否缺索引；IXSCAN 也要检查扫描范围是否过大
  4. 04FETCH 不一定是问题，但覆盖查询可以减少文档读取
  5. 05查看每个聚合阶段的输入输出，寻找基数突然膨胀的位置
  6. 06在接近生产的数据分布和并发下再次验证
一次快不代表稳定

缓存、并发、磁盘和副本负载都会影响耗时。保留计划形态和扫描量，才能在数据增长后继续比较。

06

CHECKLIST

聚合优化检查清单

  • ✦过滤与排序是否拥有顺序匹配的复合索引
  • ✦分页是否使用稳定排序和游标，而不是无限深 skip
  • ✦$lookup 外键和关联结果是否有明确上限
  • ✦数组展开、分组和排序是否可能产生巨大中间集
  • ✦慢查询、扫描量、临时磁盘和查询形状是否持续监控
