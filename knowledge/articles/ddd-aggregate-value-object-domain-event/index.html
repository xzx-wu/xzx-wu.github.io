架构卷轴 · Domain Model
01

LANGUAGE & BOUNDARY

先划清语义边界，再设计对象

同一个词在不同业务上下文里可能含义不同：销售上下文的“商品”关注价格与上架状态，仓储上下文的“商品”关注库存单位和库位。限界上下文用明确边界隔离模型、术语与规则，团队在边界内维护统一语言。

不要从数据库表直接推导领域模型

表结构是持久化视角，领域模型表达业务行为与不变量。两者可以映射，但不必一一对应。

02

ENTITY & VALUE OBJECT

实体看身份，值对象看属性

实体 Entity

跨时间保持同一身份，即使属性变化仍是同一个业务对象。

Order(orderId)
值对象 Value Object

由全部属性定义相等性，通常不可变，可整体替换。

Money(amount, currency)
领域服务

承载不自然属于单一实体或值对象、但仍是领域规则的行为。

PricingPolicy
仓储 Repository

以聚合为单位提供集合式访问，隐藏持久化细节。

OrderRepository

值对象适合金额、地址、时间范围等概念。构造时完成格式与范围校验，能让非法状态更早暴露，也能减少散落在各层的重复判断。

03

AGGREGATE

聚合是业务一致性的最小事务边界

聚合由一个根实体和边界内对象组成，外部只能通过聚合根修改内部状态。根方法负责校验业务不变量，例如“已支付订单不能再次支付”“库存不得小于零”。

  • ✦一次事务尽量只修改一个聚合
  • ✦聚合间用标识引用，避免加载巨大对象图
  • ✦边界由必须强一致的规则决定
  • ✦并发冲突通过版本或条件更新显式处理
  • ✦聚合保持小而完整，不按页面字段拼装
  • ✦查询模型可独立优化，不强迫聚合承担报表
04

RESPONSIBILITIES

应用层编排，领域层决策，基础设施层实现

  1. 01接口层完成协议解析、身份认证与输入格式校验
  2. 02应用层校验授权与数据范围，开启用例和事务
  3. 03领域对象执行规则并产生状态变化或领域事件
  4. 04仓储保存聚合，基础设施实现数据库和消息访问
  5. 05应用层返回稳定 DTO，不直接暴露领域内部结构
  6. 06审计记录操作者、请求标识和关键业务结果

身份和权限属于用例边界，不能只依赖前端隐藏按钮；领域对象仍应拒绝不符合业务状态的操作。两类检查互补，而不是彼此替代。

05

DOMAIN EVENT

领域事件记录已经发生的业务事实

事件用过去时命名，例如 OrderPaid。聚合在状态变化时产生事件，事务提交后再可靠发布；跨聚合或跨服务的后续动作由订阅者处理。若数据库与消息系统不能共享事务，可结合 Outbox 保存事件，再异步投递。

事件产生与发布边界DOMAIN EVENT
加载聚合 → order.pay(payment)
          → 产生 OrderPaid(eventId, orderId, version)
同一事务 → 保存聚合 + 保存 Outbox
事务提交 → 发布事件 → 消费者幂等处理
提交前不要触发不可撤销副作用

若邮件、远程调用或消息先成功而数据库事务随后回滚，外部世界会看到一个并未成立的事实。

06

PRACTICE CHECKLIST

从业务复杂度决定是否使用 DDD

  1. 01先与业务人员整理术语、场景和不变量
  2. 02识别上下文边界及上下文间契约
  3. 03为聚合根编写成功、拒绝与并发测试
  4. 04事件包含稳定标识、发生时间和 Schema 版本
  5. 05监控事件发布失败、重复与积压
  6. 06简单 CRUD 保持简单，不为套模式制造抽象
