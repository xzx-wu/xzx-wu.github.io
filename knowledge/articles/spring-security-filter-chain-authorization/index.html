框架档案 · Spring Security
01

FILTER CHAIN

请求先进入安全过滤器体系

Servlet 容器通过 DelegatingFilterProxy 把请求交给 Spring Bean；FilterChainProxy 再选择一个 SecurityFilterChain，并按顺序执行其中的安全过滤器。链路通常依次建立安全上下文、处理认证、保存上下文、执行授权和异常转换。

容器入口

DelegatingFilterProxy 连接 Servlet Filter 与 Spring 容器。

bridge
安全代理

FilterChainProxy 统一管理匹配、顺序和防火墙。

proxy
规则链

SecurityFilterChain 描述匹配器和具体过滤器。

policy
业务入口

控制器只会在安全链允许后收到请求。

resource

调试时应先确认请求进入哪条链、执行了哪些过滤器，再判断某条规则为什么没有生效。

02

MULTIPLE CHAINS

多条 SecurityFilterChain 只选择首个匹配项

可以分别为 API、管理端和静态资源声明不同的 SecurityFilterChain。FilterChainProxy 按顺序检查请求匹配器，只执行第一条匹配链，因此更具体的链必须排在通用链之前，并保留一条明确的兜底链。

匹配顺序示意POLICY
Order 1  /api/**    → 无状态 Token 认证
Order 2  /admin/**  → 管理端登录与强授权
Order 3  /**         → 默认拒绝或公共页面策略
没有命中安全链不等于公开访问

它意味着请求可能绕开预期的 Spring Security 保护。应让路由范围与兜底策略可审计，并对未声明资源默认拒绝。

03

AUTHENTICATION

把凭据转换为可信身份

  1. 01认证过滤器从请求中提取凭据并构造未认证 Authentication
  2. 02AuthenticationManager 把认证委托给匹配的 AuthenticationProvider
  3. 03Provider 校验凭据，并返回包含主体与权限的已认证对象
  4. 04成功结果进入 SecurityContext，供后续授权读取
  5. 05失败由 AuthenticationEntryPoint 等组件转为受控响应

密码、Token、证书等认证方式可以采用不同 Provider，但不应把原始密钥、完整 Token 或密码写入日志。无状态 API 仍需在每个请求上可靠地重建并校验身份。

04

AUTHORIZATION

请求授权与方法授权共同守住资源

请求授权适合按 URL、HTTP 方法和角色控制入口；方法授权适合在服务层表达业务动作与参数相关规则。两层可以组合，但业务资源归属、租户隔离和数据范围必须在可信后端校验，不能只依赖页面隐藏按钮。

  • ✦公共端点显式 permitAll，其余路径明确 authenticated 或 denyAll
  • ✦写操作同时校验动作权限与目标资源归属
  • ✦角色用于粗粒度分组，权限用于稳定的业务动作
  • ✦管理接口、监控端点和错误页也进入安全设计

未认证访问通常进入 401 处理；身份已建立但权限不足通常返回 403。两者混淆会让前端交互和线上排查都变得困难。

05

CUSTOM FILTER

自定义过滤器必须放在语义正确的位置

新增过滤器前先确认职责：建立身份的 Token 过滤器要早于授权过滤器；只依赖已认证身份的审计过滤器应放在认证之后。避免既把过滤器注册为容器 Filter，又加入 SecurityFilterChain，造成执行两次。

过滤器不应吞掉异常和上下文

失败应交给安全异常处理体系或输出稳定错误契约；请求结束后必须避免线程复用导致安全上下文泄漏。

CORS 决定浏览器是否允许跨域请求，CSRF 防御针对浏览器自动携带凭据的跨站伪造；是否关闭 CSRF 必须根据认证凭据的携带方式判断，而不是因为“这是 REST API”就直接关闭。

06

DIAGNOSTICS

安全链排查与测试清单

  • ✦打印或断点确认请求命中的 SecurityFilterChain 与过滤器顺序
  • ✦分别覆盖匿名、已登录无权限、已授权和过期凭据场景
  • ✦验证 401、403、登录跳转和 JSON 错误体符合接口契约
  • ✦测试路径变体、HTTP 方法、URL 编码和管理端点
  • ✦验证 CSRF、CORS、会话固定保护和退出登录行为
  • ✦日志只保留 requestId、规则与结果，不记录敏感凭据
