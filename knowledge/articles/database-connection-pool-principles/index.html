01

WHY POOLING

数据库连接为什么昂贵

创建数据库连接通常涉及网络握手、认证、会话初始化以及数据库端资源分配。如果每次 SQL 都创建并销毁连接，延迟和资源抖动会直接进入请求链路。

  • ✦复用已经建立的物理连接，降低建连成本
  • ✦限制应用可以同时占用的数据库会话数量
  • ✦在短时流量峰值中提供有限等待区
  • ✦检测失效连接并管理连接生命周期
连接池也是限流器

最大连接数决定一个应用实例能同时向数据库施加多少并发。池调得越大，并不代表数据库吞吐一定越高。

02

BORROW & RETURN

借用到归还的完整流程

初始化

池按最小空闲策略建立部分连接，或在第一次请求时按需创建。

create physical connection
借用

调用 DataSource.getConnection，优先取得空闲连接；没有空闲时可能创建或等待。

borrow
使用

业务通过代理 Connection 执行 SQL 和事务，物理连接仍由池管理。

execute
归还

调用 close 通常不是关闭物理连接，而是清理会话状态并放回空闲队列。

return

池必须在归还时恢复必要状态，例如自动提交、只读标记和事务隔离设置，避免一个请求的会话状态污染下一个借用者。

03

POOL STATES

空闲、活跃、等待与淘汰

  • ✦Idle：已建立但当前无人借用的连接
  • ✦Active：已经交给业务，尚未归还的连接
  • ✦Pending：没有可用连接，正在等待的线程
  • ✦Creating：正在建立的新物理连接
  • ✦Evicting：过期、失效或超过空闲目标而被关闭
  • ✦Closed：池关闭后不再接受新的借用

当 Active 达到上限时，新的请求只能等待归还，直到获取超时。此时继续扩大 Web 线程或接口并发，往往只会制造更多等待者。

04

CAPACITY

最大连接数如何规划

容量必须从全局看，而不是每个实例独立拍脑袋。假设数据库可分配给应用的连接预算为 200，应用最多运行 10 个实例，那么单实例上限不应忽略实例数量、管理连接和突发扩容。

容量关系FORMULA
应用总连接上限
= 单实例 maximumPoolSize × 最大实例数
+ 管理、任务、迁移等额外连接

应用总连接上限应小于数据库为该应用预留的连接预算

05

TIMEOUT LAYERS

不要只有一个“数据库超时”

连接获取超时

等待池中可用连接的最长时间，超时通常代表池饱和或连接泄漏。

connectionTimeout / maxWait
建立连接超时

驱动与数据库建立网络连接的时间限制。

connectTimeout
Socket 读取超时

SQL 已发送后等待网络响应的时间限制，由驱动配置。

socketTimeout
语句执行超时

针对单条 Statement 或事务操作的执行预算。

queryTimeout

上层请求超时应该大于内部必要步骤的预算，并为异常处理留出时间。反过来，如果数据库读取超时大于 HTTP 请求超时，客户端已经离开后 SQL 仍可能继续占用连接。

06

TRANSACTION SCOPE

事务时长就是连接占用时长

可靠归还连接JAVA
/**
 * 查询用户名称，try-with-resources 保证连接和语句按顺序归还。
 */
public String queryUserName(DataSource dataSource, long userId) throws SQLException {
    String sql = "SELECT name FROM user_account WHERE id = ?";
    try (Connection connection = dataSource.getConnection();
         PreparedStatement statement = connection.prepareStatement(sql)) {
        statement.setLong(1, userId);
        try (ResultSet resultSet = statement.executeQuery()) {
            return resultSet.next() ? resultSet.getString("name") : null;
        }
    }
}

Spring 声明式事务会把连接绑定到当前事务上下文，直到事务完成才归还。因此在事务中进行远程调用、文件处理或长时间循环，会直接减少池中可用连接。

07

LEAK & METRICS

连接泄漏与监控

  • ✦活跃连接长期接近上限，空闲连接持续为零
  • ✦等待线程和获取超时数量持续上升
  • ✦连接持有时间明显超过正常 SQL 或事务耗时
  • ✦数据库端看到大量长事务、锁等待或 Sleep 会话
  • ✦记录借用时堆栈只用于诊断，不应替代正确关闭
  • ✦同时观察应用池指标和数据库端连接状态
泄漏检测是告警，不是自动修复

发现连接长时间未归还后，应定位事务边界、异常路径和阻塞调用。强制回收仍在使用的连接可能破坏正在执行的业务。
