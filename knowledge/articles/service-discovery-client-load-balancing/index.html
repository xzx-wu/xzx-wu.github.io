DISCOVERY MODELS

客户端发现与代理发现有不同责任

Client-side

客户端获取实例列表并自行选择目标。

smart client
Proxy-side

客户端访问稳定入口，由代理选择后端。

load balancer
DNS

通过名称解析获得地址，受 TTL 与缓存影响。

name
Platform Service

平台提供稳定虚拟地址并维护后端集合。

service

客户端发现减少中间跳转但增加 SDK 复杂度；代理发现统一策略但引入额外组件和路径。选择应结合语言数量、策略一致性、连接协议与运维能力。

02

HEALTH & LIFECYCLE

注册成功不等于已经可以接收流量

  1. 01实例启动完成依赖加载和预热后再通过 readiness
  2. 02注册中心或平台把实例加入可发现集合
  3. 03客户端通过 Watch、长轮询或周期拉取更新本地视图
  4. 04下线先停止接收新流量，再等待在途请求完成
  5. 05租约到期、健康失败或主动注销后从候选集中移除

Liveness 表示是否需要重启，Readiness 表示是否能接收流量，两者不能使用同一套过于严格的下游检查。发现缓存需要在控制面短暂不可用时保留最后可用视图，同时设置过期与告警。

03

LOAD BALANCING

算法必须匹配请求成本与状态

  • ✦轮询：简单均匀，假设实例和请求成本接近
  • ✦随机：低状态，实例足够多时分布稳定
  • ✦最少请求：适合耗时差异明显，但需要可靠在途统计
  • ✦加权：表达实例容量差异，权重需要持续校准
  • ✦一致性哈希：减少键到实例映射变化，但需要处理热点与扩缩容
负载均衡不能替代过载保护

所有实例都接近饱和时，继续寻找“更空闲”节点只会延迟失败。队列上限、并发舱壁、限流和快速拒绝必须同时存在。

04

CONNECTION AWARENESS

长连接会让请求分布偏离实例分布

HTTP/2 和 RPC 通常在少量长连接上复用多个请求。DNS 或四层负载均衡可能只在建连时选择后端，因此连接长期不更新时，新实例难以获得流量，旧实例下线也可能仍有在途调用。

连接级与请求级决策ROUTING
连接级均衡：每条连接选择一次后端
请求级均衡：每个请求从可用子连接中选择

长连接场景需关注子连接数量、刷新与排空

客户端需要对实例变化建立或关闭子连接，并在优雅下线时停止分配新请求、等待旧请求完成。强制频繁重连会增加 TLS 和连接成本，需按流量验证。

05

TOPOLOGY

优先本地并不意味着永不跨区

区域、可用区、机房与网络成本可以进入路由策略。通常优先同区以降低延迟和跨区成本，但必须在本地容量不足或故障时有受控切换，并为跨区调用保留预算。

  • ✦实例元数据由可信平台写入，客户端不接受任意标签
  • ✦灰度与版本路由有明确匹配顺序和兜底边界
  • ✦会话粘滞不能替代服务端无状态与共享状态设计
  • ✦跨区切流前验证数据库、缓存和消息依赖是否同样可用
  • ✦故障恢复采用渐进回流，避免瞬时流量冲击
06

OPERATIONS

发现与路由运行清单

  1. 01监控注册实例、可用实例、发现延迟和缓存年龄
  2. 02按实例统计请求、在途、延迟、错误和拒绝
  3. 03验证启动预热、优雅下线、崩溃与网络分区
  4. 04控制面故障时使用最后视图，但禁止无限期陈旧
  5. 05路由变更记录版本、操作者、范围和回滚结果
  6. 06告警同时看无可用实例、流量倾斜和全部过载
