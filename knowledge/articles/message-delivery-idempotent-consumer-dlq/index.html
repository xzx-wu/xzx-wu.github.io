01

DELIVERY SEMANTICS

投递语义描述失败窗口中的结果

At-most-once

可能丢失但不重复，通常先确认后处理。

0..1
At-least-once

尽量不丢失但可能重复，要求消费者幂等。

1..N
Effectively-once

通过事务、幂等和去重让业务效果只发生一次。

effect
Exactly-once Scope

必须说明在哪个 Broker、事务和外部副作用边界成立。

scope

网络无法可靠区分“Broker 未收到”和“Broker 已持久化但确认丢失”。生产者重发会产生重复，因此消息 ID 和业务幂等从一开始就应存在。

02

PRODUCER CONFIRM

发布确认只证明 Broker 接收边界

  1. 01业务数据与 Outbox 事件在同一本地事务提交
  2. 02发布器发送带稳定 eventId、业务键和 Schema 版本的消息
  3. 03等待 Broker 的 Publisher Confirm 或对应确认机制
  4. 04确认超时后按同一 eventId 重试，不创建新业务事件
  5. 05成功后更新发布状态，失败持续重试并告警

Publisher Confirm 与消费者 Ack 是相互独立的两类确认：前者覆盖生产者到 Broker，后者覆盖 Broker 到消费者。Broker 接收成功不代表消费业务已经完成。

03

CONSUMER ACK

本地事务提交后再确认消息

消费者取得消息后校验 Schema、权限边界和业务键，在本地事务中完成去重记录与业务更新；只有事务提交成功后才发送 Ack。若进程在提交后、Ack 前崩溃，消息会再次投递，幂等记录负责返回已完成结果。

可靠消费顺序CONSUMER
接收消息
  → 校验与反序列化
  → 本地事务：去重记录 + 业务更新
  → 事务提交
  → Ack
自动确认可能把失败变成永久丢失

若消息在业务完成前已经确认，进程崩溃后 Broker 不再重投。关键消费应明确确认时机和未确认消息上限。

04

IDEMPOTENCY & ORDER

去重解决重复，版本与分区键解决顺序

  • ✦eventId + consumerName 建立唯一处理约束
  • ✦同一业务聚合使用稳定 routing key 进入同一有序分区
  • ✦消费者并发度不能破坏同一键的处理顺序
  • ✦消息携带 aggregateVersion，拒绝旧版本覆盖新状态
  • ✦分区内有序不等于跨分区全局有序

热点业务键会集中到单分区，顺序与吞吐必须权衡。需要全局顺序通常意味着更强协调和更低并行度，应先确认业务是否真的需要。

05

RETRY & DEAD LETTER

暂时失败重试，永久失败隔离

连接超时、短暂锁冲突等可恢复失败使用有限次数与退避；格式错误、缺失必填字段和确定业务拒绝不应无限重试。超过预算的消息进入死信交换或死信主题，保留原消息、错误、次数和时间。

  • ✦重试采用延迟队列或调度机制，避免立即 requeue 热循环
  • ✦按异常类型决定重试、丢弃、死信或人工介入
  • ✦死信队列有告警、查看、修复、重放和权限审计流程
  • ✦重放仍使用原 eventId，并再次通过幂等与版本检查
  • ✦敏感消息在日志与死信存储中加密或脱敏
06

OPERATIONS

消息链路运行治理清单

  1. 01监控发布确认、未确认消息、消费速率、积压年龄和死信
  2. 02按消费者跟踪处理成功、重复命中、重试与永久失败
  3. 03为消息大小、单消费者并发和预取数量设置上限
  4. 04演练 Broker 切换、确认丢失、消费者崩溃和毒消息
  5. 05Schema 变更保持向前向后兼容并提供回放测试
  6. 06定义积压清理、数据保留、灾备恢复与重放审批
