01

PRIMARY & REPLICA

索引由多个 Lucene 分片组成

Primary shard

文档按路由落到一个主分片，主分片处理并协调对应写入。

authoritative copy
Replica shard

主分片的冗余副本，可承担搜索并在故障时提升可用性。

redundancy
Routing

默认基于文档 ID 选择分片；自定义 routing 必须在读写两端保持一致。

placement

增加副本可以提高读取并发和故障容忍，却增加写入复制、磁盘与恢复成本；它不能替代快照备份。

WRITE PATH

协调节点到主分片和副本

  1. 01协调节点根据 routing 找到目标主分片

  3. 03操作复制到副本并按当前写入要求等待确认

  5. 05后台 refresh 和 merge 让数据可搜索并整理 segment
03

NEAR REAL TIME

Refresh 让新 segment 可搜索

refresh 会打开新的搜索可见 segment，因此 Elasticsearch 是近实时搜索。每次写入都请求 refresh=true 会频繁创建小 segment，增加刷新与合并压力。

按需等待可见HTTP
POST /articles/_doc/1001?refresh=wait_for
{
  "title": "Search Archive",
  "status": "PUBLISHED"
}

wait_for 等待下一次自然 refresh，而不是主动强制刷新。只有业务确实需要写后立即搜索时才承担这段等待。

04

BULK INDEXING

批量写入与反压

  • ✦使用 Bulk API 减少逐条请求的网络和协调开销
  • ✦逐步压测批次大小，不用单一“最佳条数”套所有文档
  • ✦检查 bulk 响应中每一项，整体 HTTP 200 不代表全部成功
  • ✦对 429 做有限退避重试，并保留幂等文档 ID
  • ✦大批导入可临时延长 refresh interval，结束后恢复并验证
05

SHARD SIZING

过多小分片同样昂贵

  • 01按预计数据量、增长周期和节点数规划主分片
  • 02使用数据流、ILM 或 rollover 控制时间序列索引大小
  • 03避免每个租户或每天无条件创建大量空索引
  • 04分片数变更通常需要 split、shrink 或 reindex
  • 05为节点故障恢复预留磁盘、网络和分片迁移能力
没有固定的万能分片大小

文档结构、查询类型、硬件和恢复目标都会影响选择。以真实工作负载和恢复演练为依据。

06

DIAGNOSIS

集群与写入排查

常用状态HTTP
GET /_cluster/health
GET /_cat/shards?v
GET /_cat/indices?v
GET /_nodes/stats/indices,fs,jvm,thread_pool

结合 unassigned 原因、磁盘水位、写线程池拒绝、segment 数、merge 时间、refresh 时间和 JVM 压力判断瓶颈。不要只看到 yellow/red 就盲目重启节点。
