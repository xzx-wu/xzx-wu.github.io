01

MENTAL MODEL

数组、桶与节点

可以把 HashMap 理解为一组桶。键的哈希值经过处理后映射到某个桶；桶内保存键值节点。理想情况下，键均匀分布，查找只需定位桶并比较少量节点。

  • ✦容量：当前桶数组能够容纳的桶数量
  • ✦大小：已经保存的键值映射数量
  • ✦负载因子：空间使用与冲突概率的权衡
  • ✦阈值：触发扩容的容量界线
HashMap 不保证遍历顺序

扩容、删除和实现变化都可能改变遍历结果。业务需要插入顺序时使用 LinkedHashMap，需要排序时考虑 TreeMap。

02

LOOKUP FLOW

一次 put 与 get 发生了什么

计算哈希

调用键的 hashCode()，并通过实现内部的扰动方式让高位信息参与桶定位。

key → hash
定位桶

根据当前容量计算桶下标，直接找到可能保存目标键的位置。

hash → bucket
比较键

先比较哈希，再使用 equals() 判断是否是同一个逻辑键。

hash + equals
更新或插入

键已存在就替换值；不存在则添加节点，并在需要时扩容或调整桶结构。

replace / append
键值行为示例JAVA
Map<Long, String> users = new HashMap<>();
users.put(1001L, "Alice");
users.put(1001L, "Bob");

// 相同逻辑键会替换旧值，最终得到 Bob
String name = users.get(1001L);
03

HASH COLLISION

哈希冲突与树化

不同键可能映射到同一个桶，这就是哈希冲突。现代 OpenJDK 的 HashMap 会在桶内使用链式节点；当冲突足够严重且表容量满足条件时，可能将桶转换为树结构，以减轻极端碰撞下的线性查找成本。

树化阈值属于实现细节

具体阈值不是 Map 接口契约，可能随 JDK 实现变化。工程代码不应依赖内部常量，而应保证键具有合理且稳定的哈希分布。

如果大量不同键返回相同 hashCode()，再先进的数据结构也只能缓解问题，无法替代正确的键设计。

04

CAPACITY

负载因子与扩容

默认负载因子 0.75 是时间与空间的通用折中。映射数量超过阈值后，HashMap 会扩大桶数组并重新分布节点。扩容会产生额外计算和内存峰值，因此已知数据规模时应考虑合理初始容量。

为预期数据量准备容量JAVA 19+
// 为预计 10_000 条映射创建合适容量，减少扩容次数
Map<Long, String> users = HashMap.newHashMap(10_000);

如果目标版本没有 newHashMap 工厂方法，可以根据预期条目数和负载因子计算初始容量，但不要为了避免一次扩容而创建远超实际需要的巨大 Map。

05

KEY CONTRACT

hashCode 与 equals 契约

  • 01equals 相等的对象必须具有相同 hashCode
  • 02hashCode 相同不代表对象一定相等
  • 03键放入 Map 后，不要修改参与比较的字段
  • 04实体键应明确身份语义，避免随数据库状态变化
不可变键示例JAVA
/**
 * 租户内用户键，使用不可变字段保证哈希值稳定。
 */
public record UserKey(long tenantId, long userId) {
}

如果键对象入 Map 后被修改，新的哈希位置可能与插入时不同，导致逻辑上“键还在”，却无法通过新状态正常找到。

06

THREAD SAFETY

HashMap 的并发边界

HashMap 不是线程安全容器。多个线程并发修改时，不能依赖外部“碰巧没有冲突”。只读共享也必须保证 Map 在发布前构建完成并被安全发布，之后不再修改。

单线程或局部变量

HashMap 是自然选择，结构简单，额外同步成本低。

并发读写

优先使用 ConcurrentHashMap 的原子方法，而不是自行组合 contains 与 put。

需要快照

构建后通过不可变副本发布，避免读线程观察到后续修改。

Map.copyOf
需要顺序

明确是插入顺序、访问顺序还是键排序，再选择对应容器。

07

ENGINEERING NOTES

实践建议

  • ✦根据语义选择 Map，而不是默认全部 HashMap
  • ✦使用稳定、不可变、分布合理的键
  • ✦已知规模时设置合适初始容量
  • ✦不要依赖遍历顺序和内部树化阈值
  • ✦并发场景选择专用容器和原子 API
  • ✦性能问题先用实际数据和分析工具验证
