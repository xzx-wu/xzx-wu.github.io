01

EVENT MODEL

JFR 记录事件，JMC 负责分析

JFR

集成于 JDK 的事件记录框架，采集 JVM 与应用运行数据。

recording
JMC

独立安装的可视化分析工具，读取 .jfr 文件并提供规则和视图。

analysis
jcmd

控制运行中 JVM 的记录启动、检查、转储与停止。

control

Oracle 文档说明，默认 JFR 配置适合持续记录并强调低开销；profile 配置会采集更多事件，适合较短的针对性分析。

02

ON-DEMAND

对运行中 JVM 启动短时记录

五分钟性能记录SHELL
jcmd 2468 JFR.start \
  name=incident \
  settings=profile \
  duration=5m \
  filename=/tmp/incident-%p-%t.jfr

jcmd 2468 JFR.check name=incident

短时 profile 记录适合复现窗口明确的延迟、分配或锁问题。开始前确认磁盘、JDK 版本、业务流量和采集时长，不把高信息量配置长期留在高负载实例上。

03

CONTINUOUS

保留最近一段时间的滚动事件

持续记录与故障转储SHELL
jcmd 2468 JFR.start \
  name=continuous \
  settings=default \
  disk=true maxage=2h maxsize=512m

jcmd 2468 JFR.dump \
  name=continuous \
  filename=/tmp/incident-%p-%t.jfr

滚动记录保留最近的事件，达到时间或大小限制后丢弃更老数据。告警发生时立即 dump，可以保存故障前的上下文，而不需要等待问题再次复现。

路径到 GC Root 要谨慎

JFR 的相关选项可能增加耗时或停顿，只在确实怀疑内存泄漏、已评估影响时开启。

04

MISSION CONTROL

在 JMC 中按时间窗分析

  1. 01先看自动分析结果和异常时间段，不直接浏览全部事件
  2. 02对齐业务告警时间，缩小到故障前后数分钟
  3. 03查看方法采样、线程、锁、分配、GC 和 I/O 页面
  4. 04从热点事件跳到调用栈，确认是否为业务代码
  5. 05用同一负载下的正常记录作对照，避免把常态误判为异常

JMC 的线程、GC、内存与方法分析应互相验证。例如 CPU 热点同时伴随高分配和频繁年轻代 GC，才形成更完整证据。

05

KEY EVENTS

不同现象关注不同事件

CPU

方法采样、线程 CPU、JIT 编译与执行状态。

hot methods
锁与延迟

Java Monitor 阻塞、线程 Park、Socket/File I/O。

latency
内存

对象分配样本、类统计、GC 原因与暂停。

allocation
系统

CPU 负载、上下文切换、进程与环境信息。

host context

JFR 是采样和事件视角，某个方法没有出现在热点列表中不代表它从未运行；结论应与日志、指标和业务请求关联。

06

PRODUCTION SAFETY

生产采集检查清单

  • ✦长期记录优先 default，profile 只在短时调查使用
  • ✦设置 maxage、maxsize 和有容量余量的输出目录
  • ✦记录文件可能包含类名、路径、参数或环境信息，应受控传输
  • ✦故障后保存原始文件，分析副本避免误操作
  • ✦不同 JDK 和 JMC 版本先做兼容验证
结束记录SHELL
jcmd 2468 JFR.stop \
  name=incident \
  filename=/tmp/incident-final.jfr
