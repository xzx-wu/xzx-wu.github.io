性能档案 · Sampling Profiler
01

SAMPLING

低开销采样而非逐次记录

async-profiler 周期性采集线程栈，并能结合 HotSpot、perf events 与 Native 栈。官方项目强调它避免传统安全点偏差，并可观察 GC、JIT 等非 Java 线程。

  • ✦选择和目标机器架构、操作系统匹配的发行包
  • ✦以目标 JVM 用户执行，并核对容器 PID 命名空间
  • ✦先用 30～60 秒短采样建立基线
  • ✦线上设置明确输出路径、持续时间和事件类型
  • ✦采样结果必须对应已知流量和问题窗口
02

CPU PROFILE

定位真正消耗处理器的路径

30 秒 CPU 火焰图SHELL
./asprof -d 30 -e cpu \
  -f /tmp/cpu-2468.html 2468

CPU 模式适合算法循环、序列化、正则、压缩、加密、日志格式化和 Native 调用等热点。若应用 CPU 很低但响应慢，CPU 火焰图可能不是正确工具，应改看 wall、锁或 I/O 事件。

03

ALLOCATION

找到对象分配最密集的调用栈

分配采样SHELL
./asprof -d 30 -e alloc \
  -f /tmp/alloc-2468.html 2468
  • 01关注宽度最大的分配路径，而不是只看某个类名
  • 02区分启动预热、正常请求与异常流量下的分配
  • 03结合 GC 日志判断高分配是否真的造成回收压力
  • 04减少临时集合、重复序列化或大对象前先做基准

Allocation 火焰图反映分配热点，不直接等于内存泄漏。泄漏需要证明对象持续存活和被引用，应结合堆转储或 JFR 分析。

04

LOCK & WALL

观察竞争与非 CPU 等待

锁竞争与 Wall-clockSHELL
./asprof -d 30 -e lock \
  -f /tmp/lock-2468.html 2468

./asprof -d 30 -e wall \
  -f /tmp/wall-2468.html 2468
Lock

查看竞争监视器及其调用路径，适合锁粒度与热点临界区问题。

contention
Wall

按墙上时间采样运行与等待线程，适合 I/O、sleep、park 等延迟。

waiting time

Wall 模式可能产生大量线程样本，应通过线程、包含/排除规则和持续时间控制范围。

05

INTERPRETATION

火焰图应该怎么读

  • ✦横向宽度代表样本占比，不代表调用发生的时间顺序
  • ✦纵向表示调用栈深度，下方是调用者，上方是被调用者
  • ✦宽而平的顶部方法常是直接消耗位置，但仍要看完整调用路径
  • ✦颜色通常用于区分或提升可读性，不能默认解释成“越红越慢”
  • ✦搜索类名后比较总宽度，注意同一路径可能分散在不同位置
样本比例不是绝对耗时

采样存在统计误差。热点越接近噪声范围，越需要更长或多次同条件采样验证。

06

VALIDATION

让优化前后可以比较

  1. 01固定 JDK、实例资源、数据规模、预热和请求模型
  2. 02记录业务吞吐、P95/P99、CPU、GC 和错误率
  3. 03采集优化前火焰图并保存命令与时间窗口
  4. 04一次只修改一个主要变量
  5. 05以相同负载重新采样，比较热点宽度和业务指标
  6. 06若 CPU 热点下降但延迟未改善，继续检查新的瓶颈

容器中 perf 权限、内核参数或安全策略可能限制 Native 采样。应按平台安全规范配置，不为方便直接赋予不必要的特权。
