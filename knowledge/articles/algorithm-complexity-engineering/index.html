01

COST MODEL

复杂度必须绑定成本模型

同一段代码可以按比较次数、数组访问、哈希计算、网络往返或磁盘 I/O 建模。输入规模也不总是单一 n：图算法常同时使用顶点数 V 和边数 E，批处理可能同时受记录数、字段宽度与分区数影响。

  • ✦明确输入规模及其上限、分布和是否有序
  • ✦选择真正稀缺的基本操作，而不是机械数代码行
  • ✦分别记录时间复杂度和额外空间复杂度
  • ✦把网络、磁盘、缓存命中与对象分配纳入工程成本
02

ASYMPTOTIC BOUNDS

O、Ω 与 Θ 描述什么

O

给出增长率的渐进上界，不自动等同于精确耗时。

upper bound
Ω

给出渐进下界，说明至少需要怎样的增长。

lower bound
Θ

上下界同阶时，描述更紧的增长范围。

tight bound

“最坏情况、平均情况、最好情况”是输入情形，“O、Ω、Θ”是渐进边界，两组概念不能混用。线上容量规划通常更关注可信的最坏边界和高分位输入。

03

ORDER OF GROWTH

常见增长阶如何判断

  • 01O(1)：操作次数不随规模增长，但常数仍可能很大
  • 02O(log n)：每一步按固定比例缩小问题，如二分查找
  • 03O(n)：完整扫描一次输入
  • 04O(n log n)：分层处理且每层覆盖全部元素
  • 05O(n²)：对每个元素再处理近似完整输入
循环嵌套不必然相乘

双指针中两个下标可能各自只前进 n 次；几何级数 n + n/2 + n/4 最终仍是 O(n)。要统计总操作次数，而不是只看语法层数。

04

AMORTIZED ANALYSIS

一次很慢与一串操作很慢不是一回事

动态数组扩容时单次复制可能是 O(n)，但把扩容成本分摊到一连串追加操作后，追加可以具有摊还常数成本。Oracle 的 ArrayList 规范只保证追加的摊还常数时间，并未承诺具体扩容倍率。

  1. 01先定义要分析的操作序列
  2. 02累计所有扩容、迁移或重建成本
  3. 03除以操作次数得到摊还成本
  4. 04不要把摊还 O(1) 误写成每次都 O(1)
05

MEASUREMENT

复杂度结论仍要经过工程测量

相同增长阶可能因缓存局部性、对象布局、分支预测和 GC 表现产生巨大差异。Java 微基准应使用 JMH，设置预热、分叉、参数规模和结果消费，避免 JIT 消除无用代码。

可重复实验记录CHECKLIST
JDK / JVM 参数 / CPU 与内存限制
数据规模与分布 / 预热次数 / 测量次数
吞吐或延迟单位 / 分配率 / GC 情况
基线版本 / 变更版本 / 原始结果

微基准证明局部实现差异，不能代替端到端压测。数据库、锁竞争与网络等待可能让局部更快的实现对整体延迟毫无改善。

ENGINEERING CHECKLIST

算法分析交付清单

  • ✦写清 n、V、E 等规模变量的业务含义
  • ✦区分最坏、平均、期望和摊还边界
  • ✦说明额外内存、对象数量与数据局部性
  • ✦用真实分布覆盖空集、小集、极端倾斜与上限规模
  • ✦保留可复现实验配置和优化前后数据
