数据库档案 · Pool Tuning
01

CHOICE

Druid 与 HikariCP 如何选择

HikariCP 以精简实现和低开销著称，常见于 Spring Boot 默认数据源；Druid 提供较丰富的内置统计、过滤器和监控能力。选择应基于团队可观测体系、配置复杂度和既有运维经验，而不是把基准测试中的单一数字当作绝对结论。

先明确职责

连接池负责管理连接与并发许可；慢 SQL 分析、数据库容量和事务设计仍需要应用监控与数据库端证据共同完成。

02

PARAMETER MAP

核心参数对照

最大连接数

HikariCP maximumPoolSize；Druid maxActive。必须乘以最大实例数核算数据库总预算。

capacity
获取等待时间

HikariCP connectionTimeout；Druid maxWait。它限制的是从池中获取连接的等待。

borrow timeout
最小空闲数

HikariCP minimumIdle；Druid minIdle。避免为了“看起来稳定”盲目预建过多连接。

idle floor
连接寿命

HikariCP maxLifetime；Druid 通过连接生存与空闲淘汰相关参数管理。应短于基础设施强制断连时间。

lifetime
连接校验

HikariCP 优先使用 JDBC4 isValid；Druid 常配置 validationQuery 与 testWhileIdle。

validation
泄漏诊断

HikariCP leakDetectionThreshold；Druid removeAbandoned、logAbandoned。诊断参数不是业务修复方案。

diagnostics
03

HIKARICP

Spring Boot 中的 HikariCP 示例

application.ymlYAML
spring:
  datasource:
    url: jdbc:mysql://db.example:3306/archive
    username: archive_app
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 3000
      validation-timeout: 1000
      idle-timeout: 600000
      max-lifetime: 1700000
      leak-detection-threshold: 60000

示例值不能直接复制到生产。尤其是 maximum-pool-size 必须结合实例数和数据库上限计算；leak-detection-threshold 适合在诊断期间识别异常持有，不应设得短于正常长事务。

04

DRUID

Spring Boot 中的 Druid 示例

application.ymlYAML
spring:
  datasource:
    type: com.alibaba.druid.pool.DruidDataSource
    druid:
      initial-size: 5
      min-idle: 5
      max-active: 20
      max-wait: 3000
      validation-query: SELECT 1
      test-while-idle: true
      test-on-borrow: false
      time-between-eviction-runs-millis: 60000
      min-evictable-idle-time-millis: 300000

Druid Starter 的属性前缀和支持项可能随版本变化，应以所用版本文档为准。对每次借用都执行校验会增加数据库压力，通常先采用空闲检测，并结合驱动与网络环境验证。

05

LIFETIME

让连接寿命短于外部断连时间

数据库、代理、负载均衡或防火墙可能按空闲时间或绝对寿命关闭连接。若池仍把连接视为可用，下一次借用才会暴露错误。连接池的最长寿命应留出抖动余量，主动在外部设施之前更新连接。

  • 01确认数据库和网络层的 idle timeout
  • 02设置稍短的池内 maxLifetime 或淘汰策略
  • 03避免所有连接同时到期造成重建风暴
  • 04监控连接创建率、关闭率和校验失败率
06

LEAK INVESTIGATION

连接泄漏排查顺序

确认池是否真的饱和

同时查看 active、idle、pending、timeout，而不是只看最大连接数。

metrics
查找长时间持有者

使用泄漏日志或 APM 事务跨度定位借用堆栈和持有时间。

borrow stack
检查事务边界

关注远程调用、循环、锁等待和异常分支是否发生在事务中。

transaction scope
核对数据库会话

确认长事务、锁等待和慢 SQL，区分未归还与执行过慢。

database evidence
不要用强制回收掩盖问题

自动关闭“疑似泄漏”连接可能中断仍在执行的事务。先将诊断证据关联到代码路径，再修复资源关闭或缩短事务。

07

OBSERVABILITY

从指标反推配置

  • ✦获取连接等待时长及其 P95、P99
  • ✦活跃、空闲、等待线程和获取超时数量
  • ✦连接创建、关闭、校验失败与使用时长
  • ✦SQL 延迟、事务时长、数据库 CPU 与锁等待
  • ✦按应用实例数核算总连接数和扩容后的上限

如果等待增加但数据库仍有余量，可以评估小幅扩池；如果数据库 CPU、I/O 或锁竞争已经饱和，扩大池只会提高并发压力。
