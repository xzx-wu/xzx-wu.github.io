TRANSACTION BOUNDARY

先缩小跨服务不变量

单库本地事务可以用原子提交保护一组数据变化；跨数据库、消息或外部服务后，协调范围、故障模式和恢复成本迅速上升。设计前先判断是否能调整聚合边界、合并权威写入或把同步不变量改为异步流程。

02

TWO-PHASE COMMIT

2PC 用准备与决议协调原子提交

第一阶段协调者要求参与者准备事务，参与者持久化状态并保留资源；全部准备成功后，第二阶段发送提交，否则发送回滚。它能提供较强原子性，但参与者在等待决议期间可能长期持锁，协调者和日志恢复也成为关键运行责任。

两阶段PROTOCOL
Phase 1  PREPARE
  → 所有参与者确认可提交并持久化准备状态

Phase 2  COMMIT / ROLLBACK
  → 协调者持久化决议并通知全部参与者
预备事务不能长期遗留

数据库文档通常建议只在外部事务管理器中使用 PREPARE TRANSACTION。遗留事务会继续持锁并阻碍清理，必须监控和恢复。

03

TRY · CONFIRM · CANCEL

TCC 把资源预留与补偿交给业务实现

  • ✦Try：校验并预留库存、额度或其他业务资源
  • ✦Confirm：确认预留并完成最终业务状态
  • ✦Cancel：释放预留，使业务回到可接受状态

三个动作都要幂等，并处理空回滚、悬挂和重复确认。Try 成功但协调者未收到响应时，Cancel 可能先到；服务端必须用事务记录和状态机判断真实阶段，而不是无条件释放或确认。

TCC 侵入业务模型，适合能够自然表达预留且价值足以承担复杂度的核心流程。不是所有 CRUD 都值得拆成三个接口。

04

SAGA

Saga 用本地事务与语义补偿组成长事务

Saga 把流程拆成多个本地事务，每步提交后触发下一步；后续失败时按业务规则执行补偿。编排式由协调器维护状态和顺序，协同式由事件推动参与者，但事件依赖更难全局理解。

  • ✦补偿不是数据库回滚，已发送通知或已使用权益可能无法完全撤销
  • ✦每一步和补偿都要幂等、可重试、可观测
  • ✦流程状态持久化，进程重启后从明确节点恢复
  • ✦定义不可补偿点，并在之前完成高风险校验
  • ✦并发 Saga 通过版本与业务规则处理隔离异常
05

TRANSACTIONAL OUTBOX

业务数据与待发布事件在同一本地事务落库

Outbox 在更新业务表的同一数据库事务中插入事件记录，再由独立发布器或 CDC 读取并发送到消息系统，从而避免“数据库提交成功但消息未发”或相反的双写窗口。

Outbox 数据流EVENT
本地事务：业务表更新 + outbox INSERT
  → CDC / Publisher 读取 outbox
  → 发布消息并携带稳定 eventId 与 aggregateId
  → 消费者按 eventId 幂等处理

Outbox 仍可能重复发布，因此消费者必须幂等。事件 Schema、顺序键、积压、清理、重放和敏感字段都需要治理；它解决原子双写，不自动解决整个跨服务业务事务。

06

SELECTION & OPERATIONS

按一致性目标与失败成本选型

  1. 01单一数据库内优先本地事务，不主动扩大协调范围
  2. 02强原子且资源支持协议时评估 2PC 与锁定成本
  3. 03可预留高价值资源时评估 TCC 的业务侵入成本
  4. 04长流程与可补偿步骤采用 Saga 并持久化状态机
  5. 05数据库变更后可靠发事件采用 Outbox 与幂等消费
  6. 06统一监控处理中事务、补偿失败、事件积压和人工工单

无论选择哪种方案，都要演练协调者崩溃、参与者超时、重复消息、乱序和补偿失败，并通过对账证明最终不变量成立。
