01

RESOURCE SEMANTICS

用资源和 HTTP 语义表达操作结果

URI 表达资源身份，方法表达意图：GET 读取，POST 创建或触发非幂等处理，PUT 替换，PATCH 部分修改，DELETE 删除。是否安全、是否幂等应遵循 HTTP 语义，并由服务端实际行为保证。

  • ✦集合与单项 URI 命名保持一致
  • ✦创建成功返回资源位置或稳定标识
  • ✦异步任务返回可查询的任务资源
  • ✦条件请求可用 ETag 防止丢失更新
  • ✦幂等写入使用业务键或 Idempotency-Key
  • ✦状态码与响应体表达同一结果
02

DATA CONTRACT

把缺省、空值、范围和分页写进契约

请求 DTO 明确必填项、长度、枚举、格式和跨字段约束；响应 DTO 明确字段类型、是否可空、单位、时区和精度。不要把持久化实体直接暴露给外部，也不要依靠客户端猜测缺字段和 null 的差别。

时间

约定 ISO 8601、时区与精度，避免本地时间歧义。

2026-07-17T08:00:00+08:00
金额

明确币种和最小单位或十进制定点精度。

amount + currency
分页

说明页码/游标、排序稳定性、总数成本与上限。

cursor + limit
枚举

说明未知值策略，新增值可能影响旧客户端。

status
03

PROBLEM DETAILS

用统一错误载体区分机器判断与人类说明

RFC 9457 定义 application/problem+json，常用成员包括 type、status、title、detail 和 instance。业务可增加稳定的错误码、字段错误列表和 traceId，但扩展字段也属于契约。

Problem Details 示例RFC 9457
{
  "type": "https://example.com/problems/order-state",
  "title": "订单状态不允许此操作",
  "status": 409,
  "code": "ORDER_STATE_CONFLICT",
  "traceId": "8f2d..."
}

detail 面向当前实例，不能包含堆栈、SQL、凭据或内部拓扑。type 应稳定可识别，不能把临时错误文案当机器判断依据。

04

SECURITY BOUNDARY

认证之后仍要校验授权与数据范围

  1. 01校验 Token 的签名、发行方、受众、有效期与用途
  2. 02按操作和资源所有权执行授权，不信任客户端传入身份
  3. 03租户标识来自受信上下文，并进入数据库查询条件
  4. 04限制请求体、分页、批量大小和上传内容
  5. 05日志记录 traceId、主体和结果，但脱敏凭据与个人数据
  6. 06对管理接口、导出和重放操作单独审计

401 表示缺少或无效认证，403 表示身份已知但不允许访问。实际错误暴露仍需考虑避免泄露资源是否存在。

05

EVOLUTION

优先兼容演进，破坏性变化才进入新版本

新增可选字段通常比删除、改名、改类型更容易兼容。版本可放在路径、媒体类型或其他明确机制中，关键不是形式，而是全站一致、可发现、可度量、有废弃窗口。

  • ✦记录消费者和实际版本使用量
  • ✦公告迁移指南、截止时间与替代接口
  • ✦旧版本只修复必要问题，避免双向漂移
  • ✦删除前验证关键消费者已经迁移
  • ✦数据库变化采用兼容的 expand/contract
  • ✦不擅自添加静默兜底掩盖非法请求
06

CONTRACT VERIFICATION

让 OpenAPI 成为可执行的变更证据

  1. 01在 OpenAPI 中维护 Schema、示例、错误和安全要求
  2. 02CI 检查规范有效性和潜在破坏性差异
  3. 03为状态码、头、空值、分页和权限编写集成测试
  4. 04使用契约测试验证提供方与真实消费者预期
  5. 05观测端点、版本、延迟、失败码与限流结果
  6. 06变更文档、实现、测试和发布说明同步提交
