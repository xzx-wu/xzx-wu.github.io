运维档案 · Resource Diagnostics
01

BASELINE

先记录主机与故障窗口

最小现场信息SHELL
date
uptime
uname -a
nproc
free -h
df -h
ps -eo pid,ppid,comm,%cpu,%mem --sort=-%cpu | head
  • ✦记录告警时间、主机、容器、版本、流量与最近发布
  • ✦确认 CPU 核数、内存、磁盘和 cgroup 限制
  • ✦区分主机资源紧张和单个进程异常
  • ✦多次采样而不是依赖命令输出的第一行

vmstat 等工具的第一行常表示自开机以来平均值，后续行才对应指定采样间隔，应区别解读。

02

CPU

从整体利用率下钻到线程

CPU 采样组合SHELL
top
top -H -p 2468
pidstat -u -p 2468 1 10
pidstat -t -u -p 2468 1 10
vmstat 1 10
us

用户态 CPU，高时下钻业务线程与方法热点。

application
sy

内核态 CPU，关注系统调用、网络、上下文切换。

kernel
wa

CPU 等待 I/O 的时间比例，需要结合设备延迟确认。

iowait
st

虚拟化环境中被宿主机拿走的 CPU 时间。

steal

Load Average 包含可运行任务和不可中断睡眠任务，并不等同 CPU 使用率。高 Load 既可能是 CPU 排队，也可能是 I/O 阻塞。

03

MEMORY

重点看 Available、回收与交换

内存与缺页采样SHELL
free -h
vmstat 1 10
pidstat -r -p 2468 1 10
cat /proc/2468/status
cat /proc/2468/smaps_rollup
  • 01MemAvailable 比单看 free 更接近系统还能提供的内存
  • 02si/so 持续出现说明正在发生 Swap 读写
  • 03RSS 包含进程驻留页，不应直接等同 Java 堆
  • 04主缺页、内存回收和 OOM 日志用于判断压力来源
  • 05容器需同时查看 cgroup 限制与使用量
Linux 使用缓存不是泄漏

Page Cache 会利用空闲内存提升 I/O。只有结合 Available 下降、持续回收、Swap、OOM 或进程 RSS 增长，才能判断是否存在内存压力。

04

BLOCK I/O

查看吞吐、延迟、队列与请求大小

设备与进程 I/OSHELL
iostat -xz 1 10
pidstat -d -p 2468 1 10
df -h
df -i
  • ✦r/s、w/s：每秒读写请求数量
  • ✦rkB/s、wkB/s：读写吞吐
  • ✦await：请求完成的平均等待时间
  • ✦aqu-sz：平均队列长度
  • ✦%util：设备忙碌程度，需要结合设备类型解释

现代 SSD、RAID、云盘和并行设备上，不能仅凭 %util=100 宣判饱和。还要结合延迟、队列、吞吐上限和云平台指标。

05

HISTORY

用 sar 回看故障发生时刻

历史与实时数据SHELL
sar -u 1 10
sar -r 1 10
sar -q 1 10
sar -d 1 10

# 读取已配置采集的某天历史文件
sar -u -f /var/log/sa/sa17

sysstat 套件包含 sar、iostat、pidstat 和 mpstat。历史能力依赖系统已提前配置采集任务；故障发生后才安装工具无法恢复过去未采集的数据。

06

DECISION FLOW

资源问题排查顺序

  1. 01对齐业务告警、系统指标和发布变更时间
  2. 02用 uptime、vmstat 判断 CPU 排队还是 I/O 阻塞
  3. 03用 pidstat 找出贡献最大的进程和线程
  4. 04CPU 问题转入线程栈、JFR 或火焰图
  5. 05内存问题对比 RSS、堆、Native 与 cgroup
  6. 06I/O 问题关联具体进程、文件、设备和外部存储
  7. 07保存采样参数和原始输出，修复后按同条件复测
