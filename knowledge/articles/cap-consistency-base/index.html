01

CAP THEOREM

C、A、P 必须放回严格定义

Consistency

CAP 中通常指原子或线性一致的读写语义，而非任意“数据正确”。

safety
Availability

每个到达非故障节点的请求最终都获得响应，不保证响应内容最新。

liveness
Partition

节点间消息可能长时间延迟或丢失，使系统被分成无法通信的部分。

network

正式结论是在可能发生通信故障的网络中，原子读写服务无法同时保证每个请求都得到响应。它不是“任意时刻三选二”，也不意味着正常网络下不能同时提供一致结果和高可用。

02

PARTITION DECISION

分区发生时必须决定错误还是旧值

若节点无法确认另一侧是否已经写入，为保持强一致，可以拒绝或等待部分请求；若坚持响应，则可能返回旧值或接受产生冲突的写入。工程选择通常细化到具体操作，而不是整个系统只有一种模式。

分区期间的操作策略DECISION
余额扣减      → 无法确认权威状态时拒绝写入
商品介绍读取  → 允许短暂旧值并标记缓存来源
购物车编辑    → 接受本地写入，恢复后按规则合并
权限变更      → 优先保护撤权生效与数据安全
超时无法证明发生了网络分区

慢节点、GC、队列拥塞、丢包和真正分区在客户端看来都可能是超时。设计应基于无法区分这些状态的事实，而非依赖完美故障检测。

03

CONSISTENCY MODELS

一致性不是强与弱的单一开关

  • ✦线性一致性：操作仿佛在调用与返回之间某一瞬间按全局顺序生效
  • ✦顺序一致性：所有节点观察同一操作顺序，但不强制尊重真实时间先后
  • ✦因果一致性：存在因果关系的操作保持顺序，并发操作可有不同观察顺序
  • ✦读己之写：同一会话能看到自己已确认的写入
  • ✦单调读：同一会话不会先看到新版本又退回更旧版本
  • ✦最终一致性：停止更新后，各副本最终收敛，但未说明收敛前读到什么

系统必须把具体保证写入 API 与业务文档。“最终一致”如果没有最大延迟、冲突规则和用户体验说明，不能指导实现和测试。

04

BASE

可用与软状态仍需要可验证约束

BASE 常用于描述 Basically Available、Soft State、Eventually Consistent 的设计取向：允许中间状态和异步收敛，以换取故障期间的响应能力。它不是 ACID 的简单反义词，也不代表可以忽略事务。

写入状态

明确已接收、处理中、已完成、失败与待补偿。

state
收敛机制

重试、事件、对账和修复任务必须可重复执行。

converge
冲突规则

版本、业务优先级或合并函数需要确定性。

resolve
时间目标

定义正常与故障情况下的可接受收敛窗口。

05

BUSINESS INVARIANTS

先保护业务不变量，再选择一致性成本

库存不能为负、同一支付不能重复入账、撤销权限不能继续访问，这些是不变量。应先识别违反不变量的代价，再决定哪些操作必须协调、哪些读可接受旧值、哪些冲突可以合并。

  • ✦按聚合或资源划分单一权威写入边界
  • ✦强一致只用于真正需要的字段与操作
  • ✦旧读必须标识版本、来源或更新时间
  • ✦冲突解决不能仅依赖机器时间的最后写入获胜
  • ✦恢复后通过对账证明数据重新满足不变量
06

DESIGN CHECKLIST

一致性设计检查清单

  1. 01为每个关键读写声明所需一致性和延迟目标
  2. 02定义超时、重复、乱序、丢失和网络分区下的行为
  3. 03明确权威数据源、版本号、冲突检测与合并方式
  4. 04设计幂等重试、补偿、对账和人工处理入口
  5. 05监控副本延迟、冲突、积压、收敛时间和修复失败
  6. 06通过故障注入验证不变量，而不是只做正常链路测试
