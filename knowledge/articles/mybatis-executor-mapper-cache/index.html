02

PARAMETER BINDING

优先使用参数占位，谨慎拼接 SQL 结构

#{value} 会创建 JDBC 参数占位并进行类型处理，适合绝大多数业务值；${value} 是原样字符串替换，只应在无法参数化的受控 SQL 结构上使用，例如经过白名单映射后的排序列。

03

EXECUTOR TYPES

SIMPLE、REUSE 与 BATCH 解决不同问题

  • ✦SIMPLE：每次执行创建新的 Statement，行为直接
  • ✦REUSE：在会话内复用相同 SQL 的 PreparedStatement
  • ✦BATCH：把更新语句批量提交给驱动，需显式关注 flush 结果

BATCH 不是把任意业务循环自动变快。批次大小、驱动能力、生成键、错误定位、内存占用和事务时长都需要验证；在读取或提交前可能触发刷新，失败时还要确定哪些批次已被数据库接受。

提交、回滚和关闭会话是事务语义的一部分。批量写入应设置上限并按可恢复边界分批，避免单个超大事务长时间持锁。

04

CACHE BOUNDARY

一级缓存属于会话，二级缓存属于命名空间

MyBatis 本地缓存默认以 SqlSession 为边界，用于避免同一会话中的重复查询，也参与处理循环引用。更新、提交、回滚或显式清理会影响缓存；跨会话不能把它当作业务一致性缓存。

二级缓存可以在 Mapper 命名空间范围共享，但对象可序列化、跨事务陈旧、分布式实例一致性和外部写入失效都要单独处理。涉及资金、权限或强一致读取时，不应仅凭“开启缓存”默认接受旧数据。

缓存命中不代表事务可见性正确

多数据源、外部 SQL、异步更新和不同 Mapper 命名空间都可能绕过预期失效路径，应先定义一致性目标再决定是否缓存。

05

DYNAMIC SQL

让条件组合稳定生成合法 SQL

if、choose、where、set 和 foreach 可以消除手写字符串拼接。where 会在存在条件时添加 WHERE 并处理开头的 AND/OR，foreach 适合构造受控集合条件。

  • ✦空条件是否意外变成全表查询或全表更新
  • ✦空集合生成何种 SQL，是否需要提前返回
  • ✦批量 IN 的元素数量是否超过数据库和网络预算
  • ✦可选更新字段是否误把 null 与“不更新”混为一谈
  • ✦动态分支是否都有单元测试和真实 SQL 日志证据
06

PERFORMANCE

查询性能与排查清单

  1. 01记录最终 SQL、参数类型、耗时和返回行数，敏感值脱敏
  2. 02使用 EXPLAIN 验证索引与扫描，而不是只修改 Mapper
  3. 03识别循环内查询和嵌套映射引起的 N+1
  4. 04大结果集使用分页、游标或流式处理，并明确关闭资源
  5. 05复杂结果用 resultMap 显式表达列、对象和集合关系
  6. 06批处理记录批次大小、flush 点、事务时长与失败位置
