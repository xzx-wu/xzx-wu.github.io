LOCK RECIPE

临时顺序节点建立公平队列

  1. 01在 /locks/order 下创建临时顺序节点 lock-
  2. 02读取并排序所有候选子节点
  3. 03若自己序号最小，则获得锁
  4. 04否则只 Watch 紧邻自己的前驱节点
  5. 05前驱删除后重新读取、排序并判断
  6. 06完成工作后显式删除；会话过期也会自动删除
02

PREDECESSOR WATCH

为什么不监听所有子节点

如果每个等待者都 Watch 锁根节点，任一持有者释放都会唤醒全部客户端，形成羊群效应。每个候选只监听前驱，通常只唤醒下一位。

Watch 与 exists 必须形成可靠步骤

前驱可能在读取列表与注册 Watch 之间被删除。客户端应通过带 Watch 的 exists 检查结果决定立即重试，避免永久等待。

03

FAILURE MODES

连接丢失会带来结果不确定

  • ✦创建请求超时后，节点可能已创建，也可能没有创建
  • ✦短时断连时会话可能仍有效，临时节点不会立即消失
  • ✦会话过期后旧客户端必须放弃领导权和锁持有状态
  • ✦长 GC 停顿可能让客户端在恢复时仍误以为自己是持有者

成熟客户端会为创建操作加入唯一标识并在重连后查找结果。Java 项目通常优先使用 Apache Curator recipe，而不是从裸 API 手写重试状态机。

04

FENCING TOKEN

让外部资源拒绝旧持有者

把顺序节点的递增序号作为 fencing token，写数据库、对象存储或下游服务时携带该 token。资源端只接受大于已记录 token 的请求。

外部资源规则TEXT
当前已接受 token = 102

客户端 A（旧持有者）提交 token = 101 → 拒绝
客户端 B（新持有者）提交 token = 103 → 接受并更新

如果外部资源不校验 token，旧进程在暂停恢复后仍可能并发写入。单靠“锁节点已经删除”无法撤销已经运行的客户端代码。

05

LEADER ELECTION

选举与锁共享同一骨架

  • 01候选者创建临时顺序节点
  • 02最小序号成为 Leader
  • 03其他候选监听前驱，而非都监听 Leader
  • 04Leader 会话过期后节点删除，下一候选接任
  • 05新 Leader 使用 epoch/token 防止旧 Leader 继续写
06

PRACTICE

生产实践清单

  • ✦锁等待必须有超时、中断和取消语义
  • ✦临界区保持短小，不在锁内执行未知长任务
  • ✦会话 Expired 立即停止受保护操作并重建状态
  • ✦锁路径、ACL、唯一标识与清理行为写入协议文档
  • ✦对断连、过期、重复回调和进程暂停做故障测试
