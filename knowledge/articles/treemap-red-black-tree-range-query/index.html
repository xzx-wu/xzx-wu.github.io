结构档案 · Balanced Tree
01

BINARY SEARCH TREE

顺序不变量让查找可以折半下行

二叉搜索树按比较关系组织键：左子树较小，右子树较大。查找、插入和删除成本取决于树高；若输入顺序让树退化成链，高度可增长到 n。

  • ✦查找沿一条根到叶路径前进
  • ✦中序遍历得到按键排序的序列
  • ✦前驱、后继和区间查询利用全序关系
  • ✦树高是核心成本，不是节点总数本身
02

RED-BLACK TREE

红黑约束控制最坏树高

TreeMap 是基于红黑树的 NavigableMap。插入和删除后通过重新着色与旋转恢复约束，使 containsKey、get、put 和 remove 获得规范承诺的 O(log n) 成本。

平衡不是“左右节点完全相等”

红黑树维护的是一组颜色与路径约束，允许局部不对称，以较少调整换取受控高度。

03

ORDERING CONTRACT

比较结果定义键的身份

TreeMap 使用键的自然顺序或构造时提供的 Comparator。若 compare(a, b) 返回 0，映射会把两个键视为同一位置；因此比较关系最好与 equals 一致，否则 Map 的一般契约可能出现令人意外的行为。

  • 01比较器满足反对称、传递和稳定
  • 02比较字段在键进入 Map 后不要变化
  • 03多字段排序明确 null、大小写和区域规则
  • 04避免用减法比较整数导致溢出
04

NAVIGABLE MAP

前驱、后继与区间是 TreeMap 的优势

按时间键查询邻近与区间JAVA API
Map.Entry<Instant, Event> before = timeline.floorEntry(at);
Map.Entry<Instant, Event> after = timeline.ceilingEntry(at);
NavigableMap<Instant, Event> window =
    timeline.subMap(start, true, end, false);

floor、ceiling、lower、higher 分别表达不同边界语义。区间接口应明确端点是否包含，避免时间窗口重复或遗漏。

05

BACKED VIEWS

范围结果通常是联动视图

subMap、headMap、tailMap 和 descendingMap 返回由原映射支持的视图，双方修改会互相反映。向范围视图插入越界键会抛出异常。

需要快照就显式复制

若结果要跨线程传递、长期缓存或与后续修改隔离，应创建独立副本，并清楚评估复制成本。

06

DECISION

什么时候选择有序映射

  • ✦需要范围、最近邻、最小值或最大值查询
  • ✦需要稳定的按键遍历顺序
  • ✦能提供正确且长期稳定的比较器
  • ✦只做精确键查询时，比较 HashMap 的成本与内存
  • ✦并发有序访问时评估 ConcurrentSkipListMap，而非共享修改 TreeMap
