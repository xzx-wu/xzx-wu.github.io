01

INVERTED INDEX

词项如何指向文档

文本进入索引后被拆成词项，倒排表记录每个词项出现在哪些文档中，并可保留词频、位置等信息用于相关性评分。查询同样经过适当分析，再与词项表匹配。

_source 与索引结构不同

_source 保存原始 JSON 便于返回和重建；倒排索引、doc values 等结构服务于搜索、聚合和排序。

FIELD TYPES

text 与 keyword 不可互换

text

经过分析，适合全文检索；通常不直接用于聚合和精确排序。

full text
keyword

作为整体词项，适合状态、编号、标签、聚合与排序。

exact value
数值与日期

使用专用类型支持范围查询、排序和压缩，不要全部存成字符串。

typed data
object 与 nested

对象数组需要保持元素内部关联时使用 nested，并接受额外索引成本。

relation
04

EXPLICIT MAPPING

核心字段显式定义

文章索引HTTP
PUT /articles-v1
{
  "mappings": {
    "dynamic": "strict",
    "properties": {
      "title": {
        "type": "text",
        "fields": {"raw": {"type": "keyword"}}
      },
      "status": {"type": "keyword"},
      "createdAt": {"type": "date"},
      "authorId": {"type": "long"}
    }
  }
}

动态映射方便探索，但生产索引若持续接收不可控字段，可能产生字段爆炸和错误类型推断。核心业务字段应通过模板或显式 Mapping 管理。

05

SCHEMA CHANGE

字段类型变更通常需要重建索引

  • ✦创建带新 Mapping 的版本化索引
  • ✦使用 reindex 或业务重放迁移文档
  • ✦校验数量、查询结果与聚合结果
  • ✦通过 alias 原子切换读写入口
  • ✦保留旧索引一段确认期后再删除

不能把已索引的 text 原地改成 keyword 并期待旧数据自动重写。索引版本和 alias 是可控演进的关键。

06

CHECKLIST

Mapping 设计检查

  • 01明确字段是全文检索、精确过滤、聚合还是排序
  • 02用 _analyze 验证中英文分词与同义词预期
  • 03限制动态字段和总字段数量
  • 04只为确实需要的能力建立 multi-fields
  • 05预先设计索引版本与 alias 切换方案
