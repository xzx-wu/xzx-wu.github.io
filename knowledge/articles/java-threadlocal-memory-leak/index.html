ThreadLocal 隔离的是线程之间的变量副本，不会自动管理请求结束，也不会自动跨线程传播。

并发档案 · ThreadLocal
01

THREAD COPY

每个线程看到自己的值

请求上下文JAVA
/** 当前线程关联的请求编号。 */
private static final ThreadLocal<String> REQUEST_ID =
    new ThreadLocal<>();

REQUEST_ID.set("request-1001");
String current = REQUEST_ID.get();

ThreadLocal 实例像一个访问键，实际值与当前线程关联。不同线程对同一个 ThreadLocal 调用 set，不会覆盖彼此副本。

02

THREAD LOCAL MAP

值为什么跟随线程存活

OpenJDK 实现中，每个 Thread 维护 ThreadLocalMap。Entry 的 key 对 ThreadLocal 使用弱引用，而 value 仍是强引用。key 被回收后，陈旧 Entry 会在后续 get、set、remove 等操作中被机会性清理。

弱 key 不保证 value 立即释放

若线程长期存活且对应槽位一直没有触发清理，value 仍可能被线程持有。显式 remove 才是可靠生命周期边界。

03

RETENTION

内存滞留如何形成

  1. 01任务向 ThreadLocal 写入较大对象或业务上下文
  2. 02任务结束时没有调用 remove
  3. 03承载任务的工作线程被线程池长期复用
  4. 04线程持续持有 ThreadLocalMap 与其中的 value
  5. 05重复任务不断留下不同上下文，形成内存和语义风险

即使 ThreadLocal 本身是 static 且 key 不会消失，未清理的值仍会跨任务保留；此时问题不仅是内存，还可能把前一个用户的上下文泄露给后一个请求。

04

THREAD POOL

线程复用导致上下文串扰

错误示例JAVA
executor.execute(() -> {
    REQUEST_ID.set(requestId);
    handleRequest();
    // 缺少 remove：下一个任务可能读取到旧 requestId
});

线程池复用的是线程，不是业务请求。任何“当前用户、租户、Trace、数据库路由”等 ThreadLocal 上下文都必须按任务建立和销毁。

05

CLEANUP

使用 try-finally 建立边界

可靠清理JAVA
/** 在当前线程内执行带请求上下文的任务。 */
public void runWithRequestId(String requestId, Runnable task) {
    REQUEST_ID.set(requestId);
    try {
        task.run();
    } finally {
        REQUEST_ID.remove();
    }
}
  • ✦set 与 remove 放在同一层基础设施代码中
  • ✦finally 保证异常路径也清理
  • ✦嵌套上下文需要保存旧值并在退出时恢复
  • ✦不要把可变大对象长期放进 ThreadLocal
06

CONTEXT PROPAGATION

异步线程不会自动继承

向线程池提交任务后，执行线程与提交线程不同，普通 ThreadLocal 不会自动传播。InheritableThreadLocal 只在线程创建时复制父线程值，与线程池的任务复用模型并不匹配。

  • 01显式捕获需要的最小上下文并包装任务
  • 02在执行线程设置，在 finally 中恢复或清理
  • 03优先使用日志、Tracing 或框架提供的上下文传播机制
  • 04权限信息仍需在服务边界校验，不能只信任传播值
