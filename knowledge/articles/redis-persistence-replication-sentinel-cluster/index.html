缓存档案 · Durability & HA
01

PERSISTENCE

RDB 与 AOF 记录不同形态

RDB

生成某个时间点的数据快照，文件紧凑、恢复快；两次快照之间的变更可能丢失。

snapshot
AOF

顺序记录写命令，按 always、everysec 或 no 控制刷盘取舍，并通过 rewrite 压缩历史。

append-only log
组合使用

同时开启可以兼顾恢复速度和数据窗口，但会增加磁盘、fork 与运维复杂度。

RPO / RTO

后台快照或 AOF 重写通常涉及 fork 与写时复制。大内存实例要监控 fork 耗时、内存峰值、磁盘吞吐与重写积压。

02

REPLICATION

副本异步追随主节点

  • ✦主节点把写命令流复制到副本，正常情况下复制是异步的
  • ✦复制 ID 与 offset 用于识别历史并尝试部分重同步
  • ✦复制积压缓冲区不足或历史不匹配时，需要全量同步
  • ✦副本延迟和故障转移窗口内仍可能丢失已确认写入
  • ✦读副本会引入陈旧读，应由业务决定是否允许
WAIT 不是强一致承诺

等待副本确认可以降低故障时丢失概率，但 Redis 复制模型仍不是强一致存储，不能据此宣称绝不丢写。

03

SENTINEL

为非分片 Redis 提供自动故障转移

  1. 01多个 Sentinel 独立监控主从节点并判断主观下线
  2. 02达到法定判断后确认客观下线并选举故障转移负责人
  3. 03提升合适副本为新主，并重配置其他副本
  4. 04客户端通过 Sentinel 发现当前主节点并处理连接切换

Sentinel 自身也要多实例、跨故障域部署。应用连接池必须验证拓扑变化后的 DNS、连接重建、重试和只执行一次的业务语义。

04

CLUSTER

Cluster 用哈希槽切分键空间

  • ✦16384 个哈希槽分配给不同主节点，每个主节点可配置副本
  • ✦客户端根据 MOVED 或 ASK 重定向更新槽位路由
  • ✦多 key 操作通常要求 key 位于同一槽，可用 hash tag 有意识地共置
  • ✦扩缩容本质是迁移槽位，期间要观察延迟、热点和失败重试
  • ✦网络分区与异步复制下仍存在已确认写入丢失窗口
05

DECISION

按容量与可用性选择

单节点

本地开发、可丢失缓存，接受单点故障。

simple
主从 + Sentinel

容量可放在单主内，需要自动故障转移。

high availability
Redis Cluster

容量或吞吐需要跨主节点分片，并能接受槽位约束。

scale out

如果数据不能丢、必须线性一致或需要复杂事务，应重新评估 Redis 是否是权威数据源，而不是继续叠加拓扑。

06

RECOVERY

恢复能力需要定期证明

  • 01定义可接受数据丢失窗口 RPO 和恢复时间 RTO
  • 02将备份复制到独立故障域，并校验文件完整性
  • 03在隔离环境执行 RDB/AOF 恢复，记录耗时与校验结果
  • 04演练主节点宕机、网络分区、磁盘满和错误删除
  • 05监控复制延迟、积压缓冲区、持久化状态和最近成功时间
