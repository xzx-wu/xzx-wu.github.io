数据库档案 · MySQL 8.x / InnoDB
01

WHY INDEXES

索引要解决什么问题

没有合适索引时，数据库通常只能检查大量数据行，再逐条判断是否满足条件。数据量增加后，扫描行数、页读取次数和排序成本都会放大。

索引会额外维护一份有序的数据结构，使数据库能够快速定位某个值、一个范围，或者直接按照索引顺序返回结果。它常用于：

  • ✦加速 WHERE 精确查询和范围查询
  • ✦减少关联查询需要检查的数据量
  • ✦辅助 ORDER BY 与分组操作
  • ✦通过唯一索引约束数据唯一性
索引也有成本

每增加一个索引，都需要额外空间，并在插入、更新和删除时同步维护。优化目标不是“索引越多越好”，而是用尽量少的索引覆盖真实的高价值查询。

02

DATA STRUCTURE

为什么常用 B+Tree

本文讨论的是 InnoDB 常见的 B+Tree 索引。索引以页为基本单位组织成多层树结构：非叶子节点主要负责导航，叶子节点保存索引记录，并按照键值顺序排列。

适合磁盘与范围查询

树高通常较低

一个页可以保存大量键和子节点指针，较大的数据集也能通过较少层级完成定位。

root → branch → leaf
叶子节点有序

找到范围起点后，可以沿叶子页继续读取，适合范围条件和顺序扫描。

BETWEEN / > / < / ORDER BY
查询复杂度稳定

从根节点逐层缩小范围，不必像无索引查询那样检查大量无关记录。

按键值逐层定位
兼顾多种条件

既可以查单个键，也可以查前缀、范围以及满足顺序的联合条件。

= / IN / LIKE 'abc%'

Hash 结构擅长等值定位，但天然不保留键的顺序，因此不适合直接支持范围扫描与排序。选择索引结构，取决于存储引擎和访问模式，而不是单纯比较某一次查找速度。

03

INNODB STORAGE

聚簇索引与二级索引

每张 InnoDB 表都有一个聚簇索引。通常，表的主键就是聚簇索引，它的叶子节点保存完整行数据。

普通索引、唯一索引等其他索引属于二级索引。二级索引的叶子记录除了索引列，还保存对应行的主键值。需要取得索引之外的列时，InnoDB 会先通过二级索引找到主键，再到聚簇索引查找完整行，这个过程常被称为“回表”。

示例表结构MYSQL
CREATE TABLE order_record (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '订单主键',
  user_id BIGINT UNSIGNED NOT NULL COMMENT '用户编号',
  status TINYINT UNSIGNED NOT NULL COMMENT '订单状态',
  amount DECIMAL(12, 2) NOT NULL COMMENT '订单金额',
  created_at DATETIME NOT NULL COMMENT '创建时间',
  PRIMARY KEY (id),
  KEY idx_user_status_time (user_id, status, created_at)
) ENGINE = InnoDB
  DEFAULT CHARSET = utf8mb4
  COMMENT = '订单记录';
主键为什么不宜过长

二级索引记录会携带主键列。主键越宽，每个二级索引占用的空间通常也越大，因此 InnoDB 表通常更适合使用短小、稳定的主键。

04

COMPOSITE INDEX

联合索引与最左前缀

联合索引可以理解为按照多个列依次排序的索引。对于 (user_id, status, created_at)，先按 user_id 排序；相同用户内部再按 status 排序；前两列相同后，才按 created_at 排序。

因此，这个索引具备以下可用于查找的最左前缀：

  • ✓(user_id)
  • ✓(user_id, status)
  • ✓(user_id, status, created_at)
  • ×不能把 (status, created_at) 当作它的最左前缀
能够充分利用索引顺序的查询SQL
SELECT id, created_at
FROM order_record
WHERE user_id = 10001
  AND status = 2
  AND created_at >= '2026-07-01 00:00:00'
ORDER BY created_at DESC
LIMIT 20;

索引列在 SQL 文本中的书写先后不是判断标准，优化器可以调整等值条件。关键在于联合索引内部的排序顺序，以及查询条件能否形成连续的左侧前缀。

遇到范围条件后怎么办

范围列之后的索引列，通常不能继续用于缩小连续查找区间，但仍可能参与索引条件下推或覆盖查询。不要只背“范围后全部失效”，应结合执行计划确认每一列的实际作用。

05

COVERING INDEX

覆盖索引与回表

如果查询需要的列都能从当前索引记录中取得，就不需要再次访问聚簇索引，这称为覆盖索引。

示例索引包含 user_id、status、created_at，二级索引记录还携带主键 id。因此前面的查询只读取 id 和 created_at 时，可以形成覆盖查询。

加入非索引列后可能需要回表SQL
SELECT id, created_at, amount
FROM order_record
WHERE user_id = 10001
  AND status = 2
ORDER BY created_at DESC
LIMIT 20;

amount 不在该二级索引中，数据库通常需要根据索引记录里的主键回到聚簇索引读取金额。对于返回少量数据的查询，这未必是问题；当扫描和回表行数很大时，成本才会明显上升。

不要盲目把所有返回列塞进索引

覆盖索引可以减少回表，但也会增加索引宽度和写入成本。是否扩展索引，要结合查询频率、返回行数、字段大小和写入压力判断。

06

COMMON PITFALLS

常见索引误区

跳过联合索引首列

只查询 status 时，不能把它当作该联合索引的常规最左前缀使用。

WHERE status = 2
对索引列做计算

函数或运算可能使普通索引无法直接完成定位，可考虑改写条件或使用合适的函数索引。

WHERE DATE(created_at) = ...
发生隐式类型转换

字段类型与参数类型不一致时，转换发生在哪一侧会影响索引定位能力。

字符列不要传数字参数
前置通配符

普通 B+Tree 索引无法从字符串开头确定查找范围。

LIKE '%keyword'
选择性太低

即使存在索引，优化器也可能判断全表扫描成本更低。

需结合数据分布判断
只看是否用了索引

命中索引不代表执行计划一定优秀，还要关注扫描行数、回表、排序和实际耗时。

EXPLAIN + 真实数据验证
两条容易误导的“口诀”

IS NULL、!=、OR 并非绝对不能使用索引。能否使用、是否值得使用，取决于索引结构、数据分布、查询组合和优化器成本估算。

07

VERIFY THE PLAN

使用 EXPLAIN 验证

索引设计完成后，要用执行计划验证，而不是凭感觉判断。先执行普通 EXPLAIN：

查看执行计划SQL
EXPLAIN
SELECT id, created_at
FROM order_record
WHERE user_id = 10001
  AND status = 2
  AND created_at >= '2026-07-01 00:00:00'
ORDER BY created_at DESC
LIMIT 20;

重点关注这些字段

  • ✦type：访问方式，如 const、ref、range、index、ALL
  • ✦key：优化器最终选择的索引
  • ✦key_len：实际使用的索引键长度线索
  • ✦rows / filtered：预计检查行数与过滤比例
  • ✦Extra：覆盖索引、额外排序和条件处理信息
  • ✦possible_keys：候选索引，不等于最终采用

在支持的 MySQL 8.x 环境中，还可以使用 EXPLAIN ANALYZE 获取实际执行过程和耗时。它会真正执行查询，在线上对高成本语句使用前必须评估影响。

正确理解 Extra

Using index 通常表示覆盖索引；Using filesort 表示需要额外排序步骤，但不等于一定写入磁盘。执行计划字段要结合查询规模与实际耗时一起判断。

08

DESIGN CHECKLIST

索引设计检查清单

  • 01从真实的高频、慢查询出发设计索引
  • 02联合索引优先考虑高频等值条件和查询组合
  • 03结合范围条件与排序确定列顺序
  • 04评估是否需要覆盖索引，避免索引过宽
  • 05删除被联合索引完整覆盖的冗余索引前先验证
  • 06用 EXPLAIN 和接近生产的数据分布验证
  • 07观察查询收益，也观察写入与存储成本
  • 08上线后持续关注慢 SQL 与执行计划变化
一条实用原则

先写出要优化的 SQL，再根据过滤、关联、排序和返回列设计索引。不要脱离查询场景，孤立地讨论“应该给哪个字段加索引”。

09

SUMMARY & REFERENCES

总结与资料来源
