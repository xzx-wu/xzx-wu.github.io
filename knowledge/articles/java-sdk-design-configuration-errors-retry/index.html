01

PUBLIC SURFACE

用少量、稳定且符合 Java 习惯的入口

客户端通常是 SDK 的主要入口。通过 Builder 收集 endpoint、credential、timeout、retryPolicy 与 transport，在 build 时一次性校验必填项，生成不可变、可复用并明确线程安全性的 Client。

Client

校验配置并构建有效客户端。

create
Options

承载复杂调用的可选参数。

request
Model

表达协议无关的请求与响应。

contract

常用场景保持短路径，低频高级能力放入 Options 或显式扩展点。避免大量布尔参数、含义相近的重载和泄漏内部 HTTP 类型，让调用代码在升级后仍然易读。

02

CONFIGURATION

配置可校验，凭据可替换且不可泄漏

  • ✦endpoint、区域、服务版本等配置有类型、有默认值、有边界校验
  • ✦凭据通过 Credential Provider 提供，支持轮换与短期令牌
  • ✦日志、异常、toString 和调试输出默认脱敏
  • ✦允许注入 HTTP Client、时钟和执行器，便于测试与统一治理
  • ✦构建成功后的 Client 配置不可被并发修改
不要把访问密钥当作普通字符串配置

密钥生命周期、刷新失败和权限范围应由专门抽象管理；SDK 不能把它复制到日志、指标标签或异常消息。

03

ERROR MODEL

让调用者能够分类、恢复和追踪

SDK 异常需要保留稳定分类：客户端配置错误、连接或超时、服务端业务错误、限流、认证授权和反序列化失败。服务异常可携带状态码、服务错误码、requestId 与可安全公开的错误摘要，但不能把响应中的秘密原样拼进消息。

错误处理决策CONTRACT
配置错误        → 立即失败，提示修正参数
认证授权失败    → 刷新凭据或修正权限，不盲目重试
限流            → 尊重服务端等待提示并限制次数
暂时性网络失败  → 仅在幂等且预算允许时重试
业务拒绝        → 暴露稳定错误码，由调用者决策

不要要求使用者解析异常文本。错误类型与字段应成为公共契约，并在兼容性规则下维护。

04

TIMEOUT & RETRY

每次调用都受总预算约束

连接超时、单次请求超时与整个操作截止时间解决不同问题。重试必须计入总预算，采用有限次数、指数退避和随机抖动，并优先尊重服务端明确返回的等待信息。

  • ✦只重试已确认的暂时性失败
  • ✦读取通常可重试，写入需要幂等键或明确幂等语义
  • ✦请求体不可重复发送时禁用自动重试
  • ✦重试次数、单次耗时与最终结果都进入指标
  • ✦调用方已有重试时避免 SDK 再叠加形成指数放大
超时不代表服务端没有执行

客户端未收到结果时，写操作可能已经成功。查询状态、幂等键与请求标识比立即重复提交更安全。

05

OPERATIONS

分页、限流和可观测性也属于 API

列表操作应暴露迭代语义或明确的 page token，而不是把所有页面一次性装入内存。调用者需要能够限制页数、记录续传位置并在部分失败后恢复。

  • ✦响应暴露 requestId、状态和必要的速率限制元数据
  • ✦指标区分调用、尝试、重试、限流与最终失败
  • ✦Tracing 传播上下文，但不把高基数 ID 放进指标标签
  • ✦HTTP 日志默认关闭正文，敏感头始终脱敏
  • ✦同步与异步 API 的资源、取消和错误语义写入文档
06

COMPATIBILITY

把升级风险纳入发布流程

公共类型、方法签名、异常层级、默认行为和序列化格式都可能影响使用者。采用语义化版本时，破坏性变更进入主版本；新增能力保持向后兼容；修复不应偷偷改变业务语义。

  1. 01为公共 API 建立二进制与源码兼容检查
  2. 02废弃项先标注替代方案并保留明确迁移窗口
  3. 03发布说明列出行为、依赖和最低运行时变化
  4. 04通过 BOM 或 dependencyManagement 对齐模块版本
  5. 05维护最小示例、契约测试与上一稳定版升级测试
