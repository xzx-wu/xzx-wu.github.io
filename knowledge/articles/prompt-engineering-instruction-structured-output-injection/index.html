智能卷轴 · Prompt Design
01

TASK CONTRACT

先写清目标、输入、约束和完成标准

  • ✦说明模型扮演的任务角色，而不是虚构权威身份
  • ✦给出要完成的具体工作和不可执行的动作
  • ✦定义输入字段及可信来源
  • ✦规定输出 Schema、语言、长度和引用要求
  • ✦说明证据不足时如何明确拒绝或标记不确定
  • ✦把业务规则写成可测试的判定条件

“回答得专业一些”难以验证；“仅依据给定资料返回结论、引用片段 ID，证据不足返回 INSUFFICIENT_EVIDENCE”更接近可执行契约。

02

CONTEXT LAYERS

把指令、用户数据和外部内容明确分隔

系统约束

任务范围、安全边界和输出协议，由应用维护。

trusted
用户请求

表达本次目标，但不能覆盖服务端权限与政策。

untrusted
检索内容

用于提供事实证据，其中的文字不是系统指令。

evidence
工具结果

来自外部系统，需要校验身份、范围和新鲜度。

result

使用清晰标记和字段承载不同来源，能降低混淆，却不能从根本上消除 Prompt Injection。应用必须假设用户与外部内容都可能包含恶意指令。

03

EXAMPLES & EDGE CASES

示例用于说明判定边界，不用于堆砌答案

Few-shot 示例应覆盖典型成功、拒绝、缺失信息和容易混淆的边界。示例顺序、标签和输出格式保持一致，避免示例之间隐含冲突。

  1. 01从真实失败样本提炼最小代表示例
  2. 02示例数据脱敏并去除生产凭据
  3. 03负例说明为何拒绝，而不是只给空结果
  4. 04不要把频繁变化的业务数据硬编码进 Prompt
  5. 05为长文输入规定截断和证据选择顺序
  6. 06对不同语言、空输入和超长输入做回归
04

STRUCTURED OUTPUT

Schema 约束形状，确定性代码验证语义

最小输出契约JSON SCHEMA
{
  "type": "object",
  "required": ["status", "answer", "citations"],
  "properties": {
    "status": {"enum": ["OK", "INSUFFICIENT_EVIDENCE"]},
    "answer": {"type": "string"},
    "citations": {"type": "array", "items": {"type": "string"}}
  },
  "additionalProperties": false
}

结构正确不代表内容真实。服务端仍要验证枚举、引用存在性、金额范围、资源归属和状态流转。解析失败应返回明确错误或受控重试，不能静默猜测缺失字段。

05

PROMPT INJECTION

用最小权限降低注入成功后的影响

直接注入来自用户输入，间接注入来自网页、文件、邮件或检索文档。RAG 和微调不会自动消除此类风险；高权限工具会把文本操纵放大成真实副作用。

  • ✦外部内容标记为数据，不赋予指令权限
  • ✦工具白名单、参数 Schema 和资源范围由代码控制
  • ✦读取与写入凭据分离，并按用户身份授权
  • ✦发送、删除、支付等高风险动作要求人工确认
  • ✦输出进入 HTML、SQL 或命令前按目标上下文编码
  • ✦持续进行直接、间接和多语言攻击测试
06

VERSION & EVALUATION

Prompt 变化必须像代码变化一样回归

  1. 01记录 Prompt、Schema、模型与检索版本
  2. 02建立成功、边界、拒绝和攻击样本集
  3. 03分别评估格式、事实、引用、权限和成本
  4. 04对非确定输出使用多次运行和统计分布
  5. 05上线前比较基线并人工复核关键差异
  6. 06生产失败样本经脱敏后回流评测集
