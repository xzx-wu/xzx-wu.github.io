02

UUID

去中心化生成换来更大键与弱局部性

UUID 提供 128 位标识空间，可在无需中心协调的节点上生成。Java UUID 支持解析、比较以及不同版本和变体信息；常见随机 UUID 适合跨系统唯一标识。

  • ✦无需分配 workerId，也不依赖中心数据库
  • ✦文本表示较长，存储时可选择原生 UUID 或 16 字节形式
  • ✦随机键在 B+Tree 聚簇索引上可能造成较差写入局部性
  • ✦不同 UUID 版本的时间、随机和隐私特征不同
  • ✦UUID 不是访问令牌，不能因为难猜就跳过权限校验
03

SEQUENCE & SEGMENT

数据库序列简单可靠，号段降低中心压力

数据库自增或 Sequence 能提供中心化唯一值和较好索引顺序，但生成能力依赖数据库可用性和写入路径。多库多实例下需要明确唯一命名空间，不能让每个库从相同起点独立增长。

号段方案由中心服务一次分配一段区间，应用在内存中递增生成；它减少中心请求并允许短时离线，但会因为实例重启或切换产生未使用空洞。空洞通常不是唯一性问题，不应为了连续而回收旧号段。

业务 ID 不应要求绝对连续

事务回滚、缓存号段和并发都会产生缺口。若法规或财务编号要求连续，应使用独立、受审计的业务编号流程。

04

SNOWFLAKE

时间、节点与序列共同组成 64 位 ID

经典 Snowflake 把时间戳、worker 标识和毫秒内序列组合为 64 位整数，提供高吞吐和大致时间有序。具体位数、纪元和容量属于实现契约，不应盲目照搬。

组成示意BIT LAYOUT
sign | timestamp | worker | sequence

时间位决定可用年限
worker 位决定同时生成节点数
sequence 位决定单时间片吞吐
  • ✦workerId 分配必须唯一、持久并处理容器快速重建
  • ✦时钟回拨时等待、拒绝或切换逻辑必须明确
  • ✦同毫秒序列耗尽时等待下一时间片并监控
  • ✦只保证生成器约定内的趋势有序，不等于业务事件全局顺序
05

STORAGE & SECURITY

ID 类型会影响索引、分片与外部暴露

数据库列、Java 类型和 JSON 表示必须对齐。大整数发送给部分 JavaScript 客户端可能超过安全整数范围，可采用字符串协议；数据库索引需根据插入顺序、页分裂和分区策略实测。

  • ✦内部主键与外部公开 ID 可以分离
  • ✦不要从连续 ID 推断权限，详情接口始终校验资源归属
  • ✦避免把含时间或节点信息的 ID 当作匿名化数据
  • ✦分库分表键要服务数据分布，不能只追求全局唯一
  • ✦日志和消息统一记录字符串形式，避免精度丢失
06

SELECTION CHECKLIST

分布式 ID 选型清单

  1. 01估算峰值吞吐、节点数、使用年限和命名空间
  2. 02确认需要全局顺序、趋势顺序还是仅唯一
  3. 03演练中心故障、worker 冲突、时钟回拨和序列耗尽
  4. 04测试数据库写入、索引大小和分页行为
  5. 05定义序列化格式、跨语言精度和外部暴露策略
  6. 06监控重复、时钟偏差、号段余量与生成延迟
