01

PROXY MECHANISMS

JDK 代理与类代理

JDK 动态代理

代理接口，调用者通过接口方法进入拦截链。

interface proxy
类代理

通过生成目标类的子类覆盖可代理方法；final 类或 final 方法无法被覆盖。

class proxy

代理对象与目标对象不是“同一个调用入口”。外部调用先进入代理，代理执行事务、日志或权限切面，再调用目标方法。

02

TRANSACTION FLOW

一次声明式事务如何执行

  1. 01调用方持有 Spring 容器返回的代理 Bean
  2. 02代理匹配到事务属性并取得事务管理器
  3. 03按传播行为加入或创建事务
  4. 04执行目标业务方法
  5. 05根据正常返回或异常规则提交、回滚

readOnly 是事务语义和优化提示，不是数据库层面的绝对写保护；隔离级别、超时和传播行为也需要底层资源与事务管理器支持。

03

SELF INVOCATION

同类自调用为什么常常失效

绕过代理JAVA
@Service
public class OrderService {

    /** 批量处理订单。 */
    public void importOrders(List<Order> orders) {
        for (Order order : orders) {
            saveOne(order); // this.saveOne，未重新经过外部代理
        }
    }

    /** 保存单个订单。 */
    @Transactional
    public void saveOne(Order order) {
        // 数据库写入
    }
}

把事务方法提取到另一个职责清晰的 Spring Bean，由外部 Bean 调用，通常比通过当前代理或自注入绕路更清晰。也可以将事务定义在真正的批处理边界上，但必须明确失败策略。

04

ROLLBACK RULES

抛出异常不一定回滚

  • ✦默认规则通常对 RuntimeException 和 Error 回滚
  • ✦受检异常是否回滚需按业务显式配置 rollbackFor
  • ✦方法内部 catch 后正常返回，代理看不到失败
  • ✦异常转换时不要丢失原因与业务语义
  • ✦嵌套传播产生 UnexpectedRollbackException 时要检查内层回滚标记
显式回滚规则JAVA
/** 导入归档文件，任何导入异常都回滚数据库操作。 */
@Transactional(rollbackFor = ArchiveImportException.class)
public void importArchive(Path file) throws ArchiveImportException {
    // 导入逻辑
}

BOUNDARIES

常见的其他失效边界

对象不是 Spring Bean

手工 new 出来的对象没有容器代理。

ownership
方法不可代理

private、final 或不符合所用代理机制的方法无法被正常拦截。

visibility
线程发生切换

事务上下文通常绑定当前线程，@Async 或手工线程不会自动继承。

thread context
事务管理器选错

多数据源场景必须确认方法使用的管理器与实际资源一致。

resource
数据库不支持预期语义

存储引擎、DDL 行为和底层驱动能力会影响结果。

database
06

CHECKLIST

事务失效排查顺序

  1. 01确认对象由 Spring 容器创建且调用方拿到的是代理
  2. 02确认调用从 Bean 外部穿过代理边界
  3. 03检查方法可见性、final 与代理类型
  4. 04检查异常是否被捕获、转换和匹配回滚规则
  5. 05检查传播行为、事务管理器和数据源
  6. 06通过集成测试验证提交和回滚，而不是只看日志
事务测试必须检查数据库结果

仅看到方法抛异常不能证明已经回滚。测试要在事务外重新查询，确认数据状态符合预期。
