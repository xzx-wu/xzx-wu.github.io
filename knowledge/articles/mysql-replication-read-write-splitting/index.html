01

REPLICATION FLOW

数据如何到达副本

Source 提交事务

变更事件写入二进制日志，成为复制的数据来源。

binary log
Receiver 获取事件

副本 I/O receiver 线程从源端接收事件并写入 relay log。

relay log
Applier 重放事件

副本 applier 线程执行或应用事件，使副本逐步追上源端。

apply

MySQL 复制默认是异步的：源端提交成功不等于副本已经完成应用。网络波动、大事务、锁等待或副本资源不足都可能扩大延迟。

02

FORMAT & GTID

复制格式与事务身份

  • ✦Statement Based 记录 SQL，结果依赖语句的确定性与环境一致性
  • ✦Row Based 记录发生变化的行，通常更直接地表达数据变更
  • ✦Mixed 根据语句选择格式，但仍需理解两类行为
  • ✦GTID 为已提交事务提供全局身份，简化拓扑切换与位置管理
LIMIT 必须有确定顺序

在 statement-based 场景中，不带确定 ORDER BY 的 LIMIT 写操作可能不安全。业务 SQL 本身也不应依赖未定义的行顺序。

03

READ ROUTING

哪些查询可以走副本

适合副本

统计、报表、历史列表、允许秒级延迟的查询。

eventual read
应走源端

写后立即读、余额与库存确认、事务内查询、强一致决策。

authoritative read
不可只看 SQL 前缀

SELECT ... FOR UPDATE、临时表和会话状态都可能依赖主连接或事务上下文。

transaction aware

路由器必须与事务管理协作：事务一旦在源端开始，同一事务的后续读取不应随意切到副本。

04

READ AFTER WRITE

处理写后读不一致

  • 01关键操作完成后的一段时间内固定读源端
  • 02按请求或用户会话记录“需要主读”的上下文
  • 03等待副本执行到指定 GTID，再允许从副本读取
  • 04超出延迟阈值时摘除副本，而不是返回陈旧关键数据

每种方案都在延迟、吞吐和复杂度之间取舍。不要通过固定 sleep 猜测复制完成时间；它既不能证明一致性，也会无谓增加请求延迟。

05

FAILOVER

故障切换不只是修改地址

  • ✦选择数据最完整且状态健康的候选副本
  • ✦防止旧源端恢复后继续接受写入形成双主
  • ✦重新指向其余副本并校验 GTID/数据位置
  • ✦刷新应用连接池与路由配置，验证写入和关键读取
  • ✦记录可能丢失或未复制完成的事务边界
复制不等于备份

误删除和错误更新也会被复制。仍需独立备份、恢复演练和必要的延迟副本策略。
