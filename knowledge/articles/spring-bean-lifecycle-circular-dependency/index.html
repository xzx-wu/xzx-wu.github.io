01

LIFECYCLE

从 BeanDefinition 到可用 Bean

  1. 01读取 BeanDefinition，确定类型、作用域和依赖
  2. 02实例化对象，执行构造器或工厂方法
  3. 03填充属性与依赖注入
  4. 04调用 Aware 系列回调
  5. 05执行初始化前 BeanPostProcessor
  6. 06执行 @PostConstruct、afterPropertiesSet 和自定义 init
  7. 07执行初始化后 BeanPostProcessor，可能返回代理
  8. 08容器关闭时执行适用的销毁回调

这是理解模型，不同扩展点与作用域会让细节发生变化。业务代码通常只需要选择稳定的初始化与销毁回调，不应依赖内部创建时序。

02

CALLBACKS

初始化回调如何选择

生命周期回调JAVA
/** 管理归档索引的 Spring 组件。 */
@Component
public class ArchiveIndex {

    /** 校验配置并建立内存索引。 */
    @PostConstruct
    public void initialize() {
        // 这里只做快速且确定的本地初始化
    }

    /** 释放组件持有的资源。 */
    @PreDestroy
    public void close() {
        // 关闭由本组件拥有的资源
    }
}

现代 Spring 应用通常优先使用 @PostConstruct 与 @PreDestroy，减少对 Spring 接口的耦合。耗时的远程初始化不适合放在单例创建锁内，应使用容器完成单例创建后的扩展点或事件。

03

BEAN POST PROCESSOR

代理和注解能力从哪里进入

BeanPostProcessor 可以在初始化前后检查或包装 Bean，AOP 自动代理、部分注解处理都借助类似扩展机制。初始化后的返回对象可能已经不是原始实例，而是代理。

不要在 BeanPostProcessor 中执行业务

它是容器扩展点，应保持快速、可预测，并避免在容器创建期间触发新的复杂依赖链。

04

DESTRUCTION

容器能销毁哪些对象

  • ✦单例 Bean 由容器完整管理，关闭上下文时执行销毁回调
  • ✦prototype Bean 创建后交给调用方，容器通常不负责后续销毁
  • ✦请求和会话作用域由对应作用域管理生命周期
  • ✦业务代码只关闭自己拥有的资源，避免重复关闭共享对象
05

CIRCULAR DEPENDENCY

A 依赖 B，B 又依赖 A

构造器循环JAVA
@Service
class OrderService {
    OrderService(PaymentService paymentService) { }
}

@Service
class PaymentService {
    PaymentService(OrderService orderService) { }
}

双方都要求构造完成后才能创建另一方，容器无法得到第一个完整对象。某些单例属性注入循环可以通过提前引用被解析，但代理、异步和初始化时序会让行为更复杂；这不应成为保留环形设计的理由。

06

REFACTOR

优先消除环，而不是打开开关

  • 01提取双方共同依赖的第三个服务
  • 02区分命令与查询职责，减少双向调用
  • 03用领域事件表达完成后的通知，而非同步回调
  • 04重新审视事务边界，避免两个服务互相控制流程
  • 05@Lazy 只能延迟解析，使用前仍需确认语义正确
构造器注入暴露了真实结构

它让必需依赖和循环在启动期直接失败。把依赖改成字段注入只为“启动成功”，会把结构问题推迟到更难诊断的运行期。
