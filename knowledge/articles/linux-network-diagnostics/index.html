02

DNS & HTTP

先复现并拆分请求阶段

解析与请求计时SHELL
dig +short api.example.com
dig api.example.com

curl -sv \
  --connect-timeout 3 \
  --max-time 10 \
  -o /dev/null \
  -w 'dns=%{time_namelookup} connect=%{time_connect} tls=%{time_appconnect} start=%{time_starttransfer} total=%{time_total}\n' \
  https://api.example.com/health
  • ✦DNS 时间高：核对解析服务器、缓存、搜索域和返回地址
  • ✦connect 高或超时：检查路由、防火墙、监听和握手
  • ✦TLS 时间高：检查证书链、SNI、协议和代理
  • ✦starttransfer 高：连接已建立，重点转向服务端处理

-k 跳过证书验证会掩盖 TLS 问题，不应作为生产修复方案。

03

SS

检查监听、连接状态与队列

Socket 状态SHELL
ss -lntp
ss -s
ss -tan state established
ss -tan state time-wait
ss -tin '( sport = :8080 or dport = :8080 )'
  • 01服务未监听：确认进程、绑定地址、端口和网络命名空间
  • 02Recv-Q 持续堆积：应用读取速度可能不足
  • 03Send-Q 持续堆积：对端、网络或拥塞可能限制发送
  • 04SYN-SENT 多：目标不可达、端口未开放或响应丢失
  • 05TIME-WAIT 多：先确认是否短连接和端口压力，再决定是否优化

连接状态是现象，需要结合速率和持续时间。仅凭 TIME-WAIT 数量大不能判断故障。

04

LSOF

把端口、文件描述符与进程对应起来

端口与连接归属SHELL
lsof -nP -iTCP:8080 -sTCP:LISTEN
lsof -nP -a -p 2468 -iTCP
lsof -p 2468 | wc -l
ls /proc/2468/fd | wc -l

当应用报告“地址已被占用”“连接太多”或“Too many open files”时，先确定监听者和描述符增长类型，再检查进程限制、连接泄漏和关闭路径。

05

TCPDUMP

抓包用于证明线上的实际报文

有范围、有上限地采集SHELL
tcpdump -i any -nn -s 0 \
  -c 2000 \
  -w /tmp/api-443.pcap \
  'host 10.0.0.25 and port 443'

tcpdump -nn -r /tmp/api-443.pcap
  • ✦-nn 避免名称和服务解析干扰
  • ✦-s 0 捕获完整包，文件体积会相应增加
  • ✦-c、主机和端口过滤器控制采集上限
  • ✦-w 保存 pcap，后续在受控环境离线分析
抓包涉及敏感数据

报文可能包含账号、Cookie、Token、业务载荷和内部地址。必须按授权范围采集、加密传输、限制访问并及时销毁。

在容器环境中，抓包位置决定能看到哪一侧流量。应明确宿主机、Pod 网络命名空间、Sidecar 和负载均衡器之间的路径。

06

INCIDENT FLOW

网络问题定位顺序

  1. 01记录源、目标、端口、协议、时间、错误和超时阶段
  2. 02从故障节点执行 dig 与 curl，避免只在个人电脑复现
  3. 03用 ss 确认监听、连接状态、队列和 TCP 信息
  4. 04用 lsof 将端口与进程、描述符对应
  5. 05检查路由、防火墙、安全组、代理和容器网络
  6. 06仍无法证明时，在最小过滤和时长下抓包
  7. 07把客户端、服务端和中间设备时间线对齐后再定根因
