深分页的核心问题不是返回 20 行，而是为了找到它们已经扫描并丢弃了多少行。

数据库档案 · Pagination
01

OFFSET COST

为什么页码越大越慢

传统分页SQL
SELECT id, title, created_at
FROM article
WHERE status = 'PUBLISHED'
ORDER BY created_at DESC, id DESC
LIMIT 100000, 20;

数据库需要定位排序结果中的前 100020 条记录，再丢弃前 100000 条。即使索引避免了全表排序，深 OFFSET 仍会带来大量索引扫描；若查询列不在索引中，还可能产生大量回表。

02

STABLE ORDER

分页必须有确定顺序

只按可能重复的 created_at 排序，数据插入或同值记录会导致跨页重复、遗漏。将唯一主键作为最后一个排序键，并让索引顺序与过滤和排序一致。

复合索引SQL
CREATE INDEX idx_article_status_created_id
ON article(status, created_at DESC, id DESC);
排序方向要统一

游标条件、ORDER BY 与索引列顺序必须表达同一条时间线，否则边界判断会产生错误或失去索引优势。

03

KEYSET PAGINATION

从上一页边界继续查询

下一页查询SQL
SELECT id, title, created_at
FROM article
WHERE status = 'PUBLISHED'
  AND (created_at, id) < (?, ?)
ORDER BY created_at DESC, id DESC
LIMIT 20;

两个占位符来自上一页最后一条记录的 created_at 和 id。这种 Seek/Keyset 方式通常只扫描边界之后需要的少量索引记录，适合连续翻页、信息流和无限滚动。

  • ✦第一页不带游标，只执行排序和 LIMIT
  • ✦下一页游标包含完整排序键
  • ✦上一页需要反转条件与排序后再恢复展示顺序
  • ✦数据删除不会让 OFFSET 整体漂移
04

LATE MATERIALIZATION

必须跳页时使用延迟关联

先取主键再回表SQL
SELECT a.id, a.title, a.created_at
FROM article AS a
JOIN (
  SELECT id
  FROM article
  WHERE status = 'PUBLISHED'
  ORDER BY created_at DESC, id DESC
  LIMIT 100000, 20
) AS page_ids ON page_ids.id = a.id
ORDER BY a.created_at DESC, a.id DESC;

子查询尽量只扫描覆盖索引，减少深 OFFSET 阶段的回表数据量。它仍然没有消除扫描前 N 行的成本，因此只能作为保留随机跳页时的折中方案。

05

API CURSOR

游标接口如何设计

  • 01游标由服务端编码排序字段，不让客户端拼 SQL 条件
  • 02响应返回 nextCursor 和 hasMore，而不是虚假的精确总页数
  • 03游标绑定筛选条件或查询版本，避免跨条件误用
  • 04给游标增加签名或完整性校验，避免参数被篡改

COUNT(*) 也可能是昂贵操作。产品若只需要“是否还有下一页”，多取一条记录即可判断，不必每次计算精确总数。

06

VERIFY

用执行证据确认收益

对照执行SQL
EXPLAIN ANALYZE
SELECT id, title, created_at
FROM article
WHERE status = 'PUBLISHED'
  AND (created_at, id) < ('2026-07-17 10:00:00', 900001)
ORDER BY created_at DESC, id DESC
LIMIT 20;

对比实际扫描行数、执行时长、是否回表与排序，并使用接近生产的数据分布测试。小表上的“毫秒级”不能证明深分页在真实数据量下仍然稳定。
