02

WATCH

在明确观察点读取上下文

只观察慢调用的结果ARTHAS
watch com.example.OrderService query \
  '{params, returnObj, throwExp, #cost}' \
  '#cost > 100' -x 2 -n 5 -f
  • ✦-b 在方法进入前观察，返回值此时尚不存在
  • ✦-e 只观察异常退出，适合筛选 throwExp
  • ✦-s 成功返回，-f 在方法结束时观察
  • ✦-x 控制对象展开深度，过深可能产生大量输出
  • ✦-n 限制匹配次数，条件表达式进一步降低影响

参数和返回对象可能包含账号、令牌或个人信息，诊断记录要脱敏并控制访问。

03

TRACE

找到慢方法里的慢子调用

只输出超过 200 ms 的调用ARTHAS
trace com.example.OrderService submit '#cost > 200' -n 5

Trace 展示当前方法内部第一层调用的耗时分布，适合判断时间消耗在数据库客户端、远程调用、序列化还是业务循环。一次结果中的耗时相加可能受并发、异步与统计边界影响，不能机械等同于总时间。

逐层下钻

先在入口方法找到最慢子调用，再对该子调用继续 trace。不要一开始匹配整个包或通配大量方法。

04

STACK

回答“谁调用了这个方法”

定位异常调用路径ARTHAS
stack com.example.InventoryService deduct \
  'params[1] <= 0' -n 5

Stack 适合处理“这个方法为什么被调用”“定时任务和接口是否走到同一逻辑”“废弃入口是否仍有流量”等问题。结合条件表达式只保留异常参数或慢请求的调用栈。

05

TIME TUNNEL

记录调用，再按索引回看

记录与清理ARTHAS
tt -t com.example.PaymentService pay -n 10
tt -l
tt -i 1000
tt --delete-all
  • 01-t 记录匹配方法的调用片段
  • 02-l 列出记录，-i 按 INDEX 查看详情
  • 03历史对象是运行中的对象引用，内容可能随后变化
  • 04记录会占用内存，官方文档明确要求使用后主动清理
不要把重放当成无副作用调试

重新执行方法可能再次写数据库、发消息或调用外部系统。除非已证明幂等且环境受控，否则只查看记录，不执行重放。

06

PRODUCTION SAFETY

把影响控制在最小范围

  1. 01从精确类名和方法名开始，不使用宽泛通配
  2. 02设置条件、次数与最小对象展开深度
  3. 03避免在高频方法上长时间运行 watch、trace 或 tt
  4. 04观察 CPU、延迟与输出量，出现异常立即中止
  5. 05完成后执行 reset 清除增强，再执行 stop

这些命令通过字节码增强实现，官方文档也提示生产环境应精确限制并及时清理。
