缓存档案 · Failure Patterns
01

COMPARE

名字相近，故障形态不同

穿透

请求的数据本来就不存在，缓存和数据库都无法命中。

不存在的 key
击穿

单个高热 key 失效，大量并发同时回源重建。

hot key
雪崩

大量 key 集中过期，或 Redis 整体不可用，回源流量超过数据库承载。

many keys / outage
02

PENETRATION

阻止不存在的数据反复回源

  • ✦在入口完成参数格式、权限和资源范围校验，先挡住明显无效请求
  • ✦对确认不存在的结果缓存短 TTL 空值，并区分“空值”与真实业务 null
  • ✦稳定键空间可用布隆过滤器做前置判断，但要接受误判并规划重建
  • ✦对恶意枚举增加限流、风控与审计，不能只依赖缓存层
空值缓存不是永久墓碑

数据稍后可能被创建。空值 TTL 应短于正常缓存，并在创建成功后主动失效对应 key。

03

BREAKDOWN

让一个执行者重建热 key

  1. 01命中缓存直接返回；未命中后尝试获取有过期时间的互斥锁
  2. 02拿到锁后再次检查缓存，避免前一个执行者已经完成重建
  3. 03读取数据库并写入带抖动的 TTL，最后安全释放锁
  4. 04未拿到锁的请求采用有界等待、旧值降级或快速失败
带唯一令牌的锁REDIS
SET lock:product:1001 request-7f3a NX PX 3000

# 释放时使用 Lua：仅当 value 等于 request-7f3a 才删除

逻辑过期配合后台刷新可以降低请求延迟，但会短暂返回旧值。业务必须先确认允许多旧，再选择该方案。

04

AVALANCHE

避免同一时刻失效，也要能承受 Redis 故障

  • ✦在基础 TTL 上增加随机抖动，错开同批 key 的失效时间
  • ✦预热任务分批执行，发布和缓存重建不同时冲击数据库
  • ✦应用设置超时、隔离、限流、熔断与可解释的降级结果
  • ✦部署高可用 Redis，并演练故障切换期间的连接和重试行为
  • ✦为数据库预留回源预算，缓存不能成为无限放大请求的入口
05

CACHE ASIDE

写数据库后删除缓存

典型写入顺序FLOW
1. 开启数据库事务并更新数据
2. 提交事务
3. 删除对应缓存 key
4. 删除失败进入重试或事件补偿

这个顺序降低“缓存已删但数据库回滚”带来的旧值问题，但仍不能自动消除所有并发窗口。强一致场景需要版本号、变更事件、幂等消费或读路径校验，不能把“延迟双删”当成通用答案。

06

OBSERVABILITY

上线前检查清单

  • 01缓存命中率按业务、状态码和 key 模式拆分
  • 02监控热 key、回源 QPS、重建耗时与锁等待
  • 03统计 Redis 超时、连接池等待、错误率和淘汰量
  • 04为数据库设置独立限流阈值和熔断告警
  • 05压测批量过期、节点切换和缓存全空三类场景
