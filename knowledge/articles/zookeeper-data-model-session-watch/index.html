01

DATA TREE

Znode 构成层次命名空间

每个 znode 都有路径、数据、ACL 和统计元信息；节点可以同时保存数据并拥有子节点。常见布局会把配置、服务成员、选举和锁分到不同根路径。

命名示例TEXT
/archive
├── config
│   └── search
├── members
│   ├── worker-0000000012
│   └── worker-0000000013
└── locks
    └── reindex
02

NODE MODES

持久、临时与顺序

Persistent

创建后持续存在，直到客户端显式删除。

persistent
Ephemeral

与创建者会话绑定，会话过期后由服务端删除，不能拥有子节点。

ephemeral
Sequential

服务端在路径后追加单调递增序号，适合建立全局顺序。

sequential

临时顺序节点组合是成员注册、选举和公平锁等 recipe 的基础。

03

SESSION

断开连接不等于会话立即过期

  • ✦客户端与 ensemble 建立带超时的会话
  • ✦短暂网络断开时客户端可以连接其他服务器并恢复同一会话
  • ✦超过协商超时后服务端判定 session expired
  • ✦会话过期后临时节点被删除，旧客户端不能继续复用该会话
服务实例要区分断连与过期

Disconnected 期间不能确认自己仍持有协调权；Expired 后必须重建客户端、临时节点和选举状态。

04

WATCH

通知变化，不携带完整状态

经典 Watch 通常是一次性触发：客户端先读取数据并设置 Watch，收到事件后重新读取状态并再次注册。事件告诉你“发生了变化”，业务仍需读取当前真值。

  • 01Watch 回调保持快速，不在事件线程执行长任务
  • 02重新读取后再做业务判断，处理多个变化合并成一次通知
  • 03连接断开期间可能无法及时收到事件，恢复后重新同步
  • 04新版本支持 persistent Watch，仍需处理重连和幂等
05

VERSION & MULTI

用版本实现条件更新

Stat 中的数据版本随更新变化。写入时携带期望版本，可以实现 compare-and-set；版本不一致说明状态已经被其他客户端修改，应重新读取而不是覆盖。

原子组合概念TEXT
读取 /config/search，得到 version = 7
按 version = 7 提交 setData
若返回 BadVersion：重新读取并重新计算

多个 create / setData / delete 可通过 multi 原子提交
06

BOUNDARIES

ZooKeeper 不适合保存什么

  • ✦不保存大型业务文档、日志和高频时序数据
  • ✦不把 Watch 当可靠消息队列或事件历史
  • ✦不依赖客户端本地状态代替重新读取服务端真值
  • ✦ACL 应限制路径操作权限，不能使用开放权限省事
  • ✦路径和数据结构作为协调协议，需要版本化管理
