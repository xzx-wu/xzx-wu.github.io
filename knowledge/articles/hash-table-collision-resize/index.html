结构档案 · Hash Table
01

LOOKUP MODEL

从键到桶，再从桶到条目

  1. 01对键计算稳定的 hashCode
  2. 02把哈希值映射到当前桶数组下标
  3. 03在桶内检查哈希和 equals
  4. 04找到条目、插入新条目或确认不存在

理想分布下查找、插入和删除具有期望常数成本；当大量键落在同一桶中，成本会被桶内结构主导。

02

KEY CONTRACT

equals 与 hashCode 决定正确性

  • ✦equals 相等的对象必须返回相同 hashCode
  • ✦对象参与比较的字段不变时，哈希值应保持稳定
  • ✦不要在放入 Map 后修改影响 equals/hashCode 的键字段
  • ✦业务实体键应明确身份语义，避免把可变展示字段当身份
可变键可能“丢失”

条目仍在原桶中，但修改后的键会计算出新桶位置，后续 get 和 remove 可能无法找到它。

03

COLLISION

冲突不可避免，分布必须可控

链地址法

每个桶保存多个条目，扩容前允许一定冲突。

bucket chain
开放寻址

在数组中继续探测空位，对装载率更敏感。

probing
树化

特定实现可在严重冲突时改变桶内结构。

tree bin

自定义键应让有效字段充分参与散列。外部输入若能制造大量冲突，还应考虑拒绝服务风险和数据规模限制。

04

CAPACITY

负载因子在空间与冲突之间权衡

HashMap 通过容量与负载因子决定扩容阈值。容量过小会频繁扩容和增加冲突，过大则浪费内存并可能增加遍历成本。已知元素数量时，应按预期规模选择合理初始容量。

  • 01估算峰值条目数，而不是只看平均数
  • 02一次性批量装载前预留容量
  • 03把扩容峰值、临时数组和 GC 压力纳入容量测试
  • 04不要依赖未写入 API 规范的具体扩容倍率
05

IMPLEMENTATION DETAIL

区分 API 保证与 OpenJDK 实现

Java API 规定 HashMap 的行为和一般性能特征；桶结构、树化阈值、哈希扰动和扩容迁移属于具体 JDK 实现细节。阅读 OpenJDK 源码可以解释当前版本，但不应把内部常量写成跨版本业务契约。

HashMap 不保证顺序和线程安全

需要插入或访问顺序时选择 LinkedHashMap，需要排序时选择 TreeMap；并发共享修改应使用 ConcurrentHashMap 或明确同步。

06

ENGINEERING CHECKLIST

哈希容器检查清单

  • ✦键是否不可变并正确实现 equals/hashCode
  • ✦是否需要顺序、范围查询或并发原子操作
  • ✦容量是否覆盖峰值并留出扩容空间
  • ✦是否存在低质量散列或攻击者可控的键
  • ✦是否对空键、空值和缺失值有明确语义
  • ✦是否用真实键分布测试冲突、内存与延迟
