01

SYNCHRONIZER

一个状态与一条等待队列

  • ✦整型 state 表示同步状态，含义由具体同步器定义
  • ✦获取失败的线程进入 FIFO 等待队列并被阻塞
  • ✦释放状态后唤醒合适的后继节点继续竞争
  • ✦CAS 用于协调状态更新，但不意味着整个同步器无锁

子类实现 tryAcquire、tryRelease 或共享模式方法，AQS 的 final 模板方法负责排队、阻塞、中断和超时等机械流程。

02

EXCLUSIVE & SHARED

独占模式与共享模式

独占模式

同一时刻只有一个线程满足获取条件，例如 ReentrantLock。

exclusive
共享模式

多个线程可以同时通过，例如 Semaphore 的许可和 CountDownLatch 的开放状态。

shared

队列大致按 FIFO 管理等待者，但同步器是否公平由具体策略决定。非公平锁可能允许新到线程抢先成功，以换取更高吞吐。

03

REENTRANT LOCK

为什么叫可重入

持锁线程再次调用 lock() 可以重复获得同一把锁，内部持有计数递增；每次成功获取都必须有对应的释放，否则锁不会真正空闲。

标准结构JAVA
/** 使用显式锁保护余额更新。 */
public void updateBalance(BigDecimal amount) {
    lock.lock();
    try {
        balance = balance.add(amount);
    } finally {
        lock.unlock();
    }
}

tryLock、可中断获取和多个 Condition 是显式 Lock 相比 synchronized 的主要扩展能力；不需要这些能力时，结构化 synchronized 往往更简单。

04

CONDITION

等待某个业务条件

有界队列等待JAVA
/** 放入元素，队列已满时等待。 */
public void put(Task task) throws InterruptedException {
    lock.lockInterruptibly();
    try {
        while (queue.size() == capacity) {
            notFull.await();
        }
        queue.addLast(task);
        notEmpty.signal();
    } finally {
        lock.unlock();
    }
}
  • 01调用 await、signal 前必须持有关联 Lock
  • 02使用 while 重新检查条件，处理虚假唤醒与竞争
  • 03await 会释放锁，唤醒后重新竞争并恢复持锁状态
  • 04signal 只把等待者转为可竞争，不会立即交出锁
05

PATTERNS

锁的工程边界

  • ✦锁范围只覆盖共享状态检查与修改
  • ✦不要持锁执行远程调用、磁盘 I/O 或未知回调
  • ✦多个锁建立固定获取顺序，降低死锁风险
  • ✦处理中断时恢复中断标记或向上传递，不要静默吞掉
  • ✦优先使用 BlockingQueue、Semaphore 等成熟同步器
自定义 AQS 是基础设施工作

需要严格定义状态不变量、可重入、公平、中断、超时和序列化语义。业务代码通常不应为了“学习后复用”自行实现锁。

06

DIAGNOSIS

锁竞争如何观察

  • 01采集多份线程转储，查看 WAITING、BLOCKED 与持锁栈
  • 02使用 JFR 分析 Java Monitor Blocked、锁和线程停顿事件
  • 03记录锁等待时间，而不只统计业务方法总耗时
  • 04确认热点锁保护的数据是否可以分片或改为无共享设计
