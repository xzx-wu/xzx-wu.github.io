01

CREATE STAGES

创建有结果与无结果任务

异步任务入口JAVA
CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(
        () -> userClient.queryUser(1001L),
        remoteCallExecutor);

CompletableFuture<Void> auditFuture = CompletableFuture.runAsync(
        () -> auditService.record("query-user"),
        auditExecutor);

supplyAsync 用于产生结果，runAsync 用于只关心完成状态的任务。没有显式传入 Executor 的 async 方法通常使用公共 ForkJoinPool，业务服务应确认这是否符合隔离和容量要求。

02

DEPENDENCY

thenApply 与 thenCompose

thenApply

同步地把一个结果转换为另一个普通结果，类似集合中的 map。

T → U
thenCompose

当前结果决定下一次异步调用，并把嵌套 Future 展平。

T → Future<U>
thenAccept

消费结果但不再返回业务值，后续阶段类型为 Void。

T → void
thenRun

只关心前一阶段完成，不需要它的结果。

completion → action
展开依赖异步调用JAVA
CompletableFuture<Address> addressFuture = userFuture.thenCompose(
        user -> CompletableFuture.supplyAsync(
                () -> addressClient.queryDefault(user.id()),
                remoteCallExecutor));
避免 Future 套 Future

回调本身返回 CompletableFuture 时使用 thenCompose；如果使用 thenApply，会得到 CompletableFuture<CompletableFuture<T>>。

03

PARALLEL COMPOSITION

thenCombine 与 allOf

两个互不依赖的调用可以同时启动，再使用 thenCombine 合并结果。动态数量任务可使用 allOf 等待全部完成，但 allOf 本身不收集结果，需要保留原 Future 列表。

并行查询并合并JAVA
CompletableFuture<User> user = queryUser(userId);
CompletableFuture<List<Order>> orders = queryOrders(userId);

CompletableFuture<UserDashboard> dashboard = user.thenCombine(
        orders,
        UserDashboard::new);

并行只适用于没有前置数据依赖的操作。为了“看起来异步”而并发访问同一个受限下游，可能更快耗尽连接池和限流额度。

04

ERROR PATH

异常处理不是统一吞掉

exceptionally

只在异常时把失败转换为替代结果，适合明确允许降级的步骤。

error → fallback
handle

无论成功失败都会执行，可同时查看结果和异常并产生新结果。

(value, error) → result
whenComplete

用于记录指标、日志或清理，不改变原阶段的结果语义。

observe completion
join

在边界处等待结果，异常会包装为 CompletionException，需要解包记录根因。

boundary wait
降级必须有业务依据

不要把任何异常都替换为空集合或 null。只有业务明确允许缺失某部分数据时才降级，并记录指标区分“正常为空”和“调用失败”。

05

TIME BUDGET

为整个链路分配超时预算

超时失败与默认结果JAVA
CompletableFuture<User> strict = queryUser(userId)
        .orTimeout(800, TimeUnit.MILLISECONDS);

CompletableFuture<List<Notice>> degradable = queryNotices(userId)
        .completeOnTimeout(List.of(), 300, TimeUnit.MILLISECONDS);

CompletableFuture 的超时只改变阶段完成状态，不应想当然地认为底层 HTTP、数据库或阻塞调用已经被中止。真正的资源释放还需要下游客户端自身配置连接、读取和请求超时。

06

EXECUTOR ISOLATION

线程池隔离与上下文

  • ✦阻塞 I/O 与 CPU 密集计算使用不同容量策略
  • ✦不同重要等级业务避免共享同一个无界池
  • ✦在线程切换前显式传递 trace、租户和安全上下文
  • ✦监控活跃线程、队列、拒绝、阶段耗时与失败率
  • ✦不要在异步回调中再次无边界阻塞等待
  • ✦应用关闭时同步关闭自建 Executor
07

END-TO-END

聚合页面示例

并行、超时与异常边界JAVA
/**
 * 并行查询用户和订单，生成页面聚合结果。
 */
public UserDashboard loadDashboard(long userId) {
    CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(
            () -> userClient.queryUser(userId),
            remoteCallExecutor);

    CompletableFuture<List<Order>> orderFuture = CompletableFuture.supplyAsync(
                    () -> orderClient.queryOrders(userId),
                    remoteCallExecutor)
            // 订单区块允许超时降级，但必须记录降级指标
            .completeOnTimeout(List.of(), 500, TimeUnit.MILLISECONDS)
            .whenComplete((value, error) -> metrics.record("order-query", error));

    try {
        return userFuture
                .thenCombine(orderFuture, UserDashboard::new)
                .orTimeout(900, TimeUnit.MILLISECONDS)
                .join();
    } catch (CompletionException exception) {
        throw new DashboardLoadException("加载用户聚合页面失败", exception.getCause());
    }
}
