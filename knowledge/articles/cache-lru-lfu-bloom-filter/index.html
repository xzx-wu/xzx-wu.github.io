02

RECENCY

LRU 假设最近访问预示近期复用

经典 LRU 由哈希表和双向链表组合：哈希表负责 O(1) 定位，链表维护访问顺序，命中时把节点移动到最近端，淘汰最久未访问节点。

LinkedHashMap 支持 access-order，可作为单线程或受控场景的 LRU 基础。但访问顺序模式下 get 也会改变结构；它不是线程安全缓存，也不包含并发加载、过期、统计和防止缓存击穿等能力。

顺序扫描会污染 LRU

一次性扫描大量冷数据可能逐出真正热点，因此只看最近性不能适配所有访问分布。

03

FREQUENCY

LFU 假设高频对象未来仍然高频

LFU 按访问频率淘汰，通常还需要同频内的时间顺序。精确 LFU 可用键到节点映射、频次桶和桶内链表组合，但状态维护和并发协调比 LRU 更复杂。

  • ✦旧热点可能因历史计数过高长期占据缓存
  • ✦需要衰减或时间窗口适应流量变化
  • ✦计数器要处理上限、采样和内存成本
  • ✦低频大对象可能需要按权重单独限制
04

ADMISSION POLICY

TinyLFU 把频率用于准入竞争

TinyLFU 使用紧凑频率估计判断新候选是否比淘汰候选更值得保留，并通过周期性衰减适应访问变化。Caffeine 的 Window TinyLFU 将近期窗口与频率准入结合，目标是在多种负载下兼顾近期突发和长期热点。

淘汰策略不能凭名称选

应使用真实访问轨迹或可代表生产的回放比较命中率、回源量、对象权重和维护开销。

05

PROBABILISTIC MEMBERSHIP

Bloom Filter 用允许误判换取紧凑空间

Bloom Filter 使用位数组和多个哈希位置表示集合。查询若发现任一位为 0，则元素一定未加入；所有位均为 1 时只能判断“可能存在”。标准结构会产生假阳性，但在没有删除和实现错误时不会产生假阴性。

容量设计关系MODEL
n = 预计元素数量
p = 可接受假阳性率
m = 位数组大小
k = 哈希次数

容量超出设计值后，假阳性率会快速上升

它适合在访问数据库前过滤明确不存在的键，但不能替代最终数据源确认。普通 Bloom Filter 不支持安全删除；需要删除时应选择计数型变体并承担额外空间与计数边界。

06

ENGINEERING CHECKLIST

缓存结构检查清单

  • ✦以条目数还是对象权重限制容量
  • ✦访问分布偏近期、偏频率还是存在扫描污染
  • ✦命中、未命中、淘汰、加载耗时和回源并发是否可观测
  • ✦加载失败、空值、超时和数据更新时如何处理
  • ✦Bloom Filter 的 n、p、m、k 是否记录并持续校验
  • ✦多实例本地缓存的一致性边界是否被业务接受
